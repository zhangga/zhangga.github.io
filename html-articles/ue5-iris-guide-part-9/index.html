🎯 第九部分导读：状态之外，还有一次性“附件”

前八部分已经把一条普通复制状态走通了：Health 和 Ammo 经过脏标记、过滤、优先级、量化、Writer 与 DataStream，最终在接收端 Apply。状态复制的目标是收敛到最新值；如果中间值丢失，后来的新状态往往可以直接覆盖它。

但网络游戏还有另一类信息：一次命中确认、一条可靠操作结果、一份绑定到某个对象的自定义附件，或者一个大到无法塞进当前包预算的对象载荷。这些信息不能简单地解释为“某个属性的新值”。Iris 用 NetBlob 表示这类有类型、可序列化、可路由的数据单元；当它面向特定复制对象时，则使用 FNetObjectAttachment 这一分支。

普通复制状态

Health: 100 → 73.5。目标是最终收敛，Lost 后可以发送更新后的最新状态。

可靠 Attachment

命中结算、购买结果等必须按顺序到达的事件。等待 ACK，Lost 后保留并重新调度。

不可靠 Ordered

需要和有序事件保持相对次序，但载荷只尝试发送一次；丢失不会升级成可靠事件。

NetBlob 不是第三条底层网络通道。它最终仍进入逐连接 Attachment 队列、ReplicationWriter 和第八部分的 ReplicationDataStream。
本文边界

RPC 是 NetObjectAttachment 的重要实例，但 RPC 参数反序列化、目标函数查找和执行上下文留到第十四部分；对象引用解析留到第十二部分；Delta/Baseline 留到第十部分。本章只在这些系统与 NetBlob 的交接点停留。

🗺️ NetBlob 的端到端总览

先把组件放回完整路径。游戏层或引擎桥接层创建 Blob，Manager 负责对象校验、连接选择与预处理；真正进入包预算后，AttachmentReplication 和 Writer 负责排队、序列化与投递记录；接收端依据 Type 创建实例，等待有序条件和分片完整，再交给原 Handler。

CREATE类型与目标

FNetBlobCreationInfo 给出 Type/Flags；FNetObjectAttachment 绑定队列 owner 与实际 target。

ROUTE & QUEUEManager / Handler

Manager 校验连接和对象、选择单播/多播；HandlerManager 维护 Type 到处理器的运行时映射。

WRITE & RECEIVE可靠队列 / Writer

Ordered 数据进入序列窗口，必要时先分片；Writer 写包，投递反馈驱动 ACK、重发和释放。

三层“管理器”不要混为一谈

FNetBlobManager

面向 ReplicationSystem：创建默认 Handler、校验对象目标、处理连接和单播/多播附件，并把数据送到对应 Writer。

FNetBlobHandlerManager

面向类型：根据 Handler Definitions 的索引分配 Type，负责创建接收 Blob 与最终 Handler 路由。

FNetObjectAttachmentSendQueue

面向某连接/对象：按可靠、有序或普通不可靠语义排队，生成提交记录并接收包投递结果。

UDataStreamManager

属于第八部分的外层传输协调：它管理 Replication 等 DataStream，不理解每种 NetBlob 的业务类型。

🧱 9.1 NetBlob 概述

CreationInfo：Type 决定“谁来读”，Flags 决定“怎么送”

FNetBlobCreationInfo 只有两个字段：FNetBlobType Type 与 ENetBlobFlags Flags。Type 是 Handler Definitions 数组索引；Flags 则由类型和发送语义共同决定。InvalidNetBlobType 使用全 1 的 uint32，表示尚未注册或无效。

FlagUE 5.7.4 语义工程提醒
Reliable可靠送达，并相对于其他可靠 Blob 保序；语义上隐含 Ordered。进入可靠/有序队列，等待 ACK；不要用于可以自然过期的高频效果。
RawDataNetBlob标识 RawData 派生路径，分片和组装可以直接处理位缓冲。是内部优化提示，不等于“任意大文件传输”。
HasExportsBlob 含可能需要导出的对象引用或 NetToken。发送队列会把 exports 加入当前 ExportContext；解析细节属于对象引用章节。
Ordered相对于其他 Ordered Blob（含 Reliable）保持顺序；不可靠 Ordered 本体只发送一次。“有序”不是“可靠”的同义词。

线上 CreationInfo 没有把所有 Flags 原样复制

UE 5.7.4 的 FNetBlob::SerializeCreationInfo 对 Type 做零值优化：先写 1 bit 表示 Type 是否非零，非零时再写 7 bits；随后只保留并写入 Reliable 这一位。因此当前 Handler 数量必须小于 128。

Type != 0
1 bit Type
7 bits（可选） Reliable
1 bit 具体 Blob payload

RawDataNetBlob、HasExports 和 Ordered 并不是由这个基类函数统一写入。接收端 Handler 创建具体类型时，可以依据类型本身恢复静态 Flags；例如当前 NetRPC 接收构造会根据函数特征恢复 Ordered。自定义类型若需要额外 CreationInfo，必须自己设计对称的序列化，而不能假设 Flags 整数被整块传输。

准确表述

Type 的 7-bit 编码和 128 个 Handler 上限是 UE 5.7.4 当前协议实现，不应写成跨版本永久保证。Handler Type 还是运行时分配值，同一类型在不同配置顺序下可能得到不同索引。

FNetBlob 类型树

FNetBlob ├─ FNetObjectAttachment // 面向一个复制对象的附件基类 ├─ FPartialNetBlob // 已拆分载荷的一片 ├─ FRawDataNetBlob // 原始位缓冲 ├─ FShrinkWrapNetBlob // 发送端预序列化包装 └─ FShrinkWrapNetObjectAttachment 支撑组件（不是 FNetBlob 子类） ├─ FNetBlobManager ├─ FNetBlobHandlerManager ├─ FNetBlobAssembler └─ FReliableNetBlobQueue

FNetBlob 自身支持两种实现方式：一种是给它设置 FReplicationStateDescriptor 与量化状态缓冲，让基类按 Descriptor 序列化；另一种是派生类覆写 Serialize/Deserialize。若对象 Handle 已由外层写过，可以走 SerializeWithObject/DeserializeWithObject 避免重复编码目标。

Blob 使用侵入式引用计数；构造时引用计数为 0，由 TRefCountPtr 等持有者管理生命周期。析构时如果量化状态含动态数据，基类会按 Descriptor 释放，不能把量化缓冲当普通 POD 内存随意复制。

FNetObjectAttachment：队列 owner 与实际 target

Attachment 保存两个引用：NetObjectReference 表示负责承载/排队的复制对象，TargetObjectReference 表示真正应用附件的对象。当目标是未直接复制的嵌套对象时，Manager 可以寻找已复制 outer 作为承载者，同时保留真实 target。外层已写入 owner Handle 时，Attachment 只需额外编码 subobject/target 信息。

🧭 9.2 NetBlobManager

Manager 的真实职责

FNetBlobManager 属于 ReplicationSystem 内部协调层。初始化时取得连接集合、FNetRefHandleManager、对象引用缓存和服务端/客户端角色；随后注册默认 Handler，并初始化一份跨连接的待处理 Attachment 队列。

对象校验

解析 root/subobject 索引；目标本身未复制时尝试已复制 outer。无效目标或无法解析的 outer 会拒绝排队。

连接选择

支持指定 ConnectionId 的单播，也维护多播入队当时仍开放的连接集合，避免后来关闭连接收到旧事件。

预处理

按连接特定序列化需求，预序列化附件、复用 ShrinkWrap 缓冲或拆成多个 PartialNetBlob。

默认 Handler：当前源码快照的三项

Handler用途本章深度
UNetRPCHandler创建和接收 NetRPC Attachment。只说明它是默认注册实例；完整 RPC 留给第十四部分。
UPartialNetObjectAttachmentHandler预序列化并拆分过大的 NetObjectAttachment。本章重点。
UNetObjectBlobHandler支撑过大复制对象批次等内部 Blob 路径。用于说明 Huge Object 交接点，不展开 Writer 内部格式。

默认定义还出现在 Engine 配置中，但本地 BaseEngine.ini 在任务开始前已有用户修改。本文把 RegisterDefaultHandlers 的代码意图与配置定义交叉核对，不把某个项目覆盖项写成跨项目永久默认。

QueueNetObjectAttachment 的校验路径

  1. 空 Attachment 直接失败。
  2. 连接 ID 无效时当前实现记录警告并丢弃，随后返回 true；这里的 true 表示调用已被消费，不等于远端收到。
  3. 先把 Target 当作 owner 和 target，尝试解析 root/subobject。
  4. Target 是有效引用但自身没有复制 Handle 时，向外寻找已复制 outer 作为承载 owner。
  5. 调用 SetNetObjectReference 保存 owner/target，然后进入单播或多播待处理队列。
边界

Manager 只把附件送到逐连接 Writer 队列。真正的包预算、提交记录、ACK、Lost 和重发发生在 AttachmentReplication 与 ReplicationWriter 中。

Scope 阶段与发送策略

普通待处理队列会先建立上下文，再分别处理对象即将离开 Scope 和仍在 Scope 的条目。OOB 队列有独立准备路径，用于尽快尝试发送不可靠附件。策略 Flags 是调度提示，而不是新的可靠性协议。

策略当前语义限制
None走普通 Attachment 调度路径。是否与对象同写还受 ReplicationSystem 配置影响。
ScheduleAsOOB进入 Out-of-Band 路径，尽早尝试发送。源码注释明确：只对不可靠 Attachment 有效。
SendInPostTickDispatch提示连接在 PostTickDispatch 阶段写入相关 OOB 数据。不是保证立刻获得包容量。
SendImmediateScheduleAsOOB | SendInPostTickDispatch 的组合。“立即”仍受连接、队列与位预算约束。

多播为什么可能每个连接重新序列化

若 Attachment Descriptor 带 HasConnectionSpecificSerialization，同一多播事件不能盲目复用一份位缓冲：Manager 会按 ConnectionId 重新预序列化与分片。没有连接特定序列化时，ShrinkWrap 可以把原始 Blob 序列化一次后复用，节省 CPU；这不代表所有多播都必然走 ShrinkWrap，也不代表网络位数被压缩。

🧩 9.3 NetBlobHandler 体系

一个 Handler 负责一种 Type

INetBlobReceiver 定义两个入口：CreateNetBlob(CreationInfo) 与 OnNetBlobReceived(Context, Blob)。UNetBlobHandler 实现这一接口，并要求派生类负责创建自己能理解的 Blob，以及处理已经反序列化完成的实例。

FNetBlobHandlerManager 则是类型交换机：注册时按 Definitions 数组查找类名，把数组索引写入 Handler 的 NetBlobType；接收时用 Type 索引找到 Handler，先创建实例，再在完整数据到达后投递。

Definition[2]
ClassName→ RegisterHandler→ NetBlobType = 2→ wire Type = 2→ Create + Route

Type 是运行时索引，不是稳定资产 ID

Handler 的 API 注释明确指出：Type 在运行时决定，不同运行可能不同。当前 Manager 初始化还检查 Definitions 数量小于 128，因为非零 Type 只占 7 bits。由此得到三条工程规则：

  • 收发两端必须使用一致的 Definitions 顺序和 Handler 类集合。
  • 不要把 Type 数字写入存档、配置资产或业务协议，当成长期稳定 ID。
  • 自定义 Handler 只有类名出现在 Definitions 中，且实例确实注册到 ReplicationSystem，才获得有效 Type。

创建与投递的错误边界

阶段失败条件UE 5.7.4 行为
Register空 Handler、重复注册、类名未出现在 Definitions。返回 false；缺定义时输出 Warning。
CreateType 越界、该槽位没有 Handler、Handler 创建失败。返回空 Blob；上层反序列化路径通常设置 UnsupportedNetBlob。
DispatchBlob 无效、Type 越界或 Handler 缺失。越界设置错误；缺 Handler 记录 Warning 并停止投递。

✂️ 9.4 Blob 分片与组装

什么时候进入 Partial 路径

UPartialNetObjectAttachmentHandler 会先把原 Attachment 写入一个临时位缓冲。如果在阈值内成功，就生成发送端 ShrinkWrap 包装或直接复用；如果临时 Writer overflow，则退回 USequentialPartialNetBlobHandler，按最大 Part 位数重新序列化并拆分。

128 BMaxPartBitCount = 128 × 8
4096当前类默认最大 Part 数
512 KiB两者推导的理论默认总载荷
192 B可靠载荷默认 split threshold

这些都是 UE 5.7.4 类默认值，可以被 Engine 或项目配置覆盖。可靠阈值默认为 (128 + 64) × 8 bits；不可靠阈值还按方向区分：客户端默认 850 bytes，服务端默认 256 bytes。阈值高于单片载荷，是因为分片本身有序列、首片和 payload 长度开销。

512 KiB 不是推荐业务上限

它只是 128 B × 4096 的类默认理论值。FRawDataNetBlob 的 API 注释明确不鼓励发送需要分片/组装的巨大 Blob；真实限制还包括可靠预队列、连接包预算、exports、内存峰值和延迟。

一片 PartialNetBlob 写了什么

UE 5.7.4 没有单独的“末片标志”。每片写 packed SequenceNumber；随后写 1 bit 的 IsFirst。只有首片继续写 PartCount - 1 和原始 CreationInfo。每片最后写 packed PayloadBitCount 与 payload。末片由首片的 PartCount 和连续 sequence 推导。

SequenceNumber

1 bit PartCount - 1
Packed uint16
（仅首片） Original Type + Reliable

分片器从进程内全局递增序列号中一次预留整个序列。首片还负责携带原 Blob 的对象引用与 NetToken exports；后续片只携带自己的 payload。所有非末片期望和首片 payload 大小一致，末片可以更小。

解释性 3500-bit 案例

假设已经得到 3500-bit 的纯载荷，不含 Partial 协议开销 PartSize = 1024 bits
PartCount = ceil(3500 / 1024) = 4
Payloads = 1024 + 1024 + 1024 + 428 bits

真实业务 Blob 在拆分前还要执行自身序列化，因此属性描述、对象引用、函数参数或自定义头都会影响最终 PartCount。这个计算只用于说明按位切片，不代表 3500-bit Gameplay 结构一定恰好得到四片。

Assembler 的顺序与错误规则

  1. 第一片建立原始 CreationInfo、RefHandle、首片大小和预期末序列；新的首片会覆盖此前未完成的非可靠组装上下文。
  2. 后续片必须使用同一 RefHandle，sequence 必须连续。
  3. PartCount 不能为 0，也不能超过接收端当前配置。
  4. 首片 payload 不能超过 MaxPartBitCount；中间片必须与首片等长，只有末片可更短。
  5. 可靠序列断裂会把 Serialization Context 置为错误；不可靠断裂可以放弃当前序列并等待后续新首片。
  6. 所有片到齐后，Assembler 按原 Type 调用 HandlerManager 创建原始 Blob。
  7. RawData 类型直接接管组装缓冲；其他类型创建子 BitReader，执行原 Blob 的 Deserialize。
  8. 反序列化后消费位数必须与组装位数一致，否则设置 BitStreamError。

📦 9.5 特殊 Blob 类型与可靠队列

FRawDataNetBlob：已经序列化好的位缓冲

FRawDataNetBlob 适合无状态、已经写入位流的任意数据。它保存 TArray<uint32> 和有效 bit 数；无论外层是否提供 RefHandle，序列化都只是写 packed bit count 和原始位流。构造函数自动增加 RawDataNetBlob Flag。

分片时无需再次调用复杂对象 Serializer；组装时也可把聚合缓冲直接 Move 给新的 RawData Blob。这是 CPU 与内存拷贝路径优化，不是对“巨大 Blob 是好设计”的背书。

ShrinkWrap：减少重复序列化，不减少位数

FShrinkWrapNetBlob 和 FShrinkWrapNetObjectAttachment 持有原 Blob、已经序列化的位缓冲以及 NetToken exports。发送到多个目标时，外层可以序列化一次，然后反复走优化的 WriteBitStream。

Original Blob→ Serialize once→ Serialized buffer→ Connection A / B / C

接收端并不会创建 ShrinkWrap 类型；wire 上仍使用原 CreationInfo，接收 Handler 创建原 Blob 并直接消费这段位流。启用 NetTrace 时，为了得到正确跟踪信息，发送端当前实现会回退到序列化原 Blob，而不是使用缓存缓冲。

FReliableNetBlobQueue：为什么不是“只装可靠 Blob”

这个类的名字强调它提供可靠、有序交付能力，但 Attachment 发送队列会把带 Reliable | Ordered 任一 Flag 的 Blob 放入它。因此它既管理可靠 Blob，也管理不可靠 Ordered Blob 的序列位置。

🎮 命中事件附件的完整旅程

场景输入：状态和事件并行

服务器判定一次射击命中。角色 Health 从 100 变为 73.5、Ammo 从 30 变为 29——这两项继续走普通状态复制。与此同时，项目需要向一个观察客户端发送自定义“命中事件附件”，其中包含攻击者、受击部位、服务器判定时间和解释性回放元数据。

Health / Ammo

允许跳过中间态，只要客户端最终收敛到服务器最新值。

小型可靠命中事件

单个 Blob 可放入包，依靠 reliable queue 等待确认。

大型可靠附件

预序列化 overflow 后拆为 Partial；所有片完整才投递原 Handler。

小型可靠附件

T0 · Handler 创建

自定义 Handler 用自己的运行时 Type 创建 Attachment，并设置 Reliable。

T1 · Manager 校验

目标对象可解析，ConnectionId 有效；Manager 保存 owner/target 并进入普通队列。

T2 · PreSerialize

载荷低于可靠 split threshold，预序列化成功，不需要 Partial。

T3 · Writer Commit

可靠队列分配 sequence，Writer 把 CreationInfo 与 payload 写入包，提交本地 ReplicationRecord。

T4 · Receiver Create

接收端读取 Type 与 Reliable，HandlerManager 创建对应 Blob，反序列化 payload。

T5 · Dispatch + ACK

有序队头允许处理，调用自定义 Handler；发送端收到 Delivered 后释放记录并推进窗口。

大型可靠附件：缺一片就等，但不重发全部

假设附件分成 sequence 900–903，901 所在包丢失，900、902、903 已到。接收端可靠队列不能越过 901 把同一原 Blob 组装完成；发送端收到 Lost 后只清除该记录覆盖 sequence 的 Sent 位。已经 Delivered 的片保持 Acked，后续包重发缺口并可同时携带新 sequence。

900 Acked→ 901 Lost→ 902/903 retained→ 901 retransmit→ Assemble once

ReplicationSystemTestPlugin 的 TestSplitBlobIsReceivedDespitePacketLoss 会在第二个包附近制造连续丢包，确认原 Handler 在完整恢复前调用次数为 0，恢复后恰好收到一次。另一个测试验证超过 1024 窗口的 MaxUnackedBlobCount + 3 个分片也可借助 PreQueue 分批送完。

不可靠 Ordered：事件本体只试一次

如果相同附件只设置 Ordered 而不设置 Reliable，它仍进入有序窗口，从而不会越过更早的可靠/Ordered 数据。但包一旦 Commit，payload 引用立即释放。若包 Lost，队列只能补写“该序列无数据”的标记以推进远端序列，不能重新产生原事件内容。

这正是“Ordered 不是 Reliable”的实现证据：它保护相对顺序，不保护载荷最终到达。业务若必须精确执行一次，还需要可靠传输之外的事件 ID、去重、幂等与服务器权威校验。

与普通状态同时丢包

若同一网络包还带着 Health=73.5 和 Ammo=29，包 Lost 后两条链分别处理：对象状态重新合并 dirty mask，下一包可以直接发送更新后的最新状态；可靠命中 Attachment 则保持自己的 sequence 和 payload 等待重发。一个包可以承载多种语义，但不能用一种重试策略解释全部数据。

可靠附件长期不前进FirstSeq/LastSeq、Sent/Acked、窗口与 PreQueue。ACK 未返回、Lost 回调丢失、窗口已满、包容量始终不足。无限扩大可靠队列。
不可靠 Ordered 事件消失是否只有 Ordered、Commit 后 payload 是否释放。这是设计语义：只发送一次。认为“进了 ReliableNetBlobQueue 就一定可靠”。
Partial 无法组装首片、sequence、RefHandle、PartCount、payload size。首片缺失、可靠序列断裂、配置不一致或位数不匹配。跳过缺片直接拼接后续片。
ShrinkWrap 没省带宽序列化 CPU 与 wire bits 分开测量。ShrinkWrap 复用位缓冲，但不压缩 payload。把名称理解成压缩包装。
PostTickDispatch 仍没立刻发OOB 是否不可靠、连接是否有预算和待发送包。发送策略只是调度提示，仍受网络写入条件约束。把 SendImmediate 当同步发送函数。

安全与错误上下文

  • Type 越界、无法创建 Blob、bitstream overflow、Partial 断序或消费位数不一致后，立即检查 Context.HasErrorOrOverflow()。
  • 不要在错误 Context 上继续读取“下一个 Blob”；位边界已经不可再信任。
  • 对网络输入恢复出的 payload 做业务层长度、枚举、对象权限和速率校验，Handler Type 正确不等于内容可信。
  • 可靠到达不等于业务 exactly-once；重连、应用层重试和服务器回滚仍需要事件 ID 与幂等。

性能取舍

  • 少用可靠高频事件：可靠窗口和 PreQueue 都持有状态，缺口会延迟后续 Ordered 数据。
  • 优先缩小 Blob：512 KiB 理论分片上限不应成为内容设计目标。
  • 多播先看连接特定序列化：能安全复用时才用预序列化缓冲；不能复用时按连接计算。
  • 区分 CPU 和带宽：ShrinkWrap 优化重复序列化，量化/压缩才可能减少 bit 数。
  • 看 NetTrace 与队列指标：跟踪序列缺口、ReliableWindowFull、分片数量和每帧 OOB 丢弃，而不是只看平均包大小。

本章要点回顾

  • NetBlob 是有类型的可序列化数据单元；对象附件是它面向复制对象的分支。
  • Type 由 Definitions 顺序在运行时分配，收发两端必须一致。
  • 基类 CreationInfo 位流只统一编码 Type 和 Reliable；其他 Flags 由具体类型恢复或处理。
  • Manager 负责对象和连接层组织，HandlerManager 负责类型创建与路由，Attachment 队列负责顺序和投递记录。
  • PartialNetBlob 使用首片 PartCount 与连续 sequence，没有单独末片标志。
  • RawData 是位缓冲快路径，ShrinkWrap 是重复序列化优化，ReliableQueue 是支撑组件而非 Blob 子类。
  • Reliable 与 Ordered 不同：不可靠 Ordered 保护相对顺序，但不保证事件本体最终到达。
