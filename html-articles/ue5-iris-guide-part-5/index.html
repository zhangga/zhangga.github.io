🎯 第五部分：过滤系统

如果把 Iris 看成一家实时配送公司，序列化解决的是“包裹怎么打包”，过滤系统解决的则是更早、也更重要的问题：这个包裹今天到底要不要送到这位玩家手里？

在一局百人规模的战术竞技游戏里，服务器同时管理玩家、载具、掉落物、队伍标记、私人任务、背包数据和全局比赛状态。若把所有对象都交给每条连接，后面的优先级和带宽预算再聪明，也只能在一座已经堆满包裹的仓库里救火。

Iris 的过滤系统会为每条连接计算对象作用域（scope）：对象在作用域内，才有机会继续进入轮询、优先级、序列化和发送阶段；对象被移出作用域时，远端对应实例也可能被销毁。因此，过滤不只是“省一点流量”，而是在定义每位玩家眼中的网络世界。

本章阅读目标

读完后，你应该能判断一个对象适合 Owner、Connection、Group 还是 Dynamic Filter；能配置并调用 GridFilter；也能从 FReplicationFiltering::Filter() 的执行顺序解释“为什么它明明在组里，却仍然被复制了”。

本文以 Unreal Engine 5.5 的 Iris 官方文档和本机引擎源码为基准。Iris 仍位于 Experimental 模块，升级版本时请以目标版本头文件为准，不要照搬函数名和配置项。

🧭 5.1 过滤系统概述

先回答“谁能看见”，再回答“先发给谁”

过滤（Filtering）与优先级（Prioritization）最容易混淆。过滤做二元决策：某对象是否属于某连接的复制作用域；优先级则只在通过过滤的对象中决定带宽紧张时谁先发送。

服务器中的全部网络对象
        │
        ├─ Owner / Connection / Group 过滤
        │          ↓
        ├─ Dynamic Filter（例如 GridFilter）
        │          ↓
        ├─ 得到每条连接的 ObjectsInScope
        │          ↓
        └─ 轮询 → 优先级 → 序列化 → 发送

四类过滤手段

类别适合的问题典型对象成本与特点
Owner Filtering只复制给拥有者私人背包、个人任务状态规则简单，依赖正确的 Owning Connection
Connection Filtering明确指定允许或禁止的连接集合小规模观战数据、管理员对象连接集合较稳定、对象较少时最合适
Group Filtering一批对象共用同一套连接规则队伍标记、队伍任务、阶段性事件批量管理，适合大量对象
Dynamic Filtering规则需要逐连接、逐帧计算玩家、载具、掉落物最灵活，但 CPU 成本最高；一个对象同时只能挂一个动态过滤器

官方文档的建议很实用：如果允许列表比较固定、受影响对象又少，优先使用 Connection Filtering；如果很多对象共享规则，用 Group Filtering；只有规则确实依赖实时状态时，再进入 Dynamic Filter。

战术竞技项目的第一版映射

业务对象首选策略原因
玩家、载具、地面掉落物GridFilter是否相关主要由观察位置和剔除距离决定
队伍信标、队伍任务Inclusion Group队友即使相距很远也必须看见，可覆盖空间过滤结果
私人背包、个人任务Owner Filter天然只属于一个拥有者连接
仅若干裁判可见的数据Connection Filter允许连接集合明确且对象数量有限
比赛阶段、毒圈状态、公共计时不过滤或保持全局相关每条连接都需要，空间过滤没有意义

这张表不是绝对答案，而是一个“从便宜规则开始”的决策顺序。业务规则变化时，可以组合组过滤与动态过滤，但要先理解它们的执行先后。

🧱 5.2 UNetObjectFilter 基类

动态过滤器的生命周期

UNetObjectFilter 是自定义动态过滤器的基类。它不直接接触 Actor 指针来做全量扫描，而是围绕 Iris 的内部对象索引、位数组和批量回调工作。官方 API 与 UE 5.5 头文件暴露的关键阶段如下：

回调何时发生应该做什么
OnInit / OnDeinit过滤器创建和销毁读取配置、建立或释放缓存
AddObject / RemoveObject对象进入或离开该过滤器分配轻量内部信息，不要做昂贵全局查询
AddConnection / RemoveConnection网络连接变化扩缩每连接状态
UpdateObjects对象状态需要刷新时更新位置、标签或自定义缓存
PreFilter本帧所有连接过滤之前一次准备共享数据
Filter按连接执行把允许对象写进 OutAllowedObjects
PostFilter本帧所有连接完成后一次收尾或交换双缓冲

两个 Trait 决定系统如何配合你

UE 5.5 的 ENetFilterTraits 有两个关键标记：

  • Spatial：过滤器依赖世界位置。引擎可以据此把它识别为空间过滤器。
  • NeedsUpdate：过滤器需要接收 UpdateObjects，用于同步会变化的对象信息。

这不是装饰性元数据。如果自定义空间过滤器漏报 Trait，系统可能不会按你期望的路径向它提供更新。

最小自定义过滤器骨架

解释性骨架，非可直接编译的完整类：

class UMySquadRangeFilter : public UNetObjectFilter
{
protected:
    void OnInit(const FNetObjectFilterInitParams& Params) override
    {
        // 读取配置、设置 Traits、建立紧凑缓存
    }

    bool AddObject(uint32 ObjectIndex,
                   FNetObjectFilterAddObjectParams& Params) override
    {
        // 为对象登记过滤所需的最少信息
        return true;
    }

    void PreFilter(FNetObjectPreFilteringParams& Params) override
    {
        // 每帧一次：准备所有连接共享的数据
    }

    void Filter(FNetObjectFilteringParams& Params) override
    {
        Params.OutAllowedObjects.ClearAllBits();

        // 只遍历候选集合；满足当前连接规则的对象置位
        // Params.OutAllowedObjects.SetBit(ObjectIndex);
    }
};

性能原则只有一句话：把能在 PreFilter 或 UpdateObjects 做一次的事情，不要在每个连接的 Filter 里重复做。因为 100 条连接会把一次看似便宜的查询放大 100 倍。

动态过滤器并不是万能挂钩

SetFilter 为对象设置动态过滤器；再次调用 SetFilter 或 SetConnectionFilter 会替换之前的设置。也就是说，不能给同一个对象同时挂“空间过滤器”和“阵营动态过滤器”，然后期待二者自动 AND。需要复合规则时，应把逻辑合并进一个动态过滤器，或把稳定的阵营规则转为 Group Filtering。

🧰 5.3 内置过滤器

UNetObjectGridWorldLocFilter：按世界位置筛选

UE 5.5 的默认 Spatial 定义指向 UNetObjectGridWorldLocFilter，配置类为 UNetObjectGridFilterConfig。它把对象登记到二维网格，再按连接的 Replication View 找候选单元格；开启精确距离后，还会对候选对象做距离平方判断。

UFilterOutNetObjectFilter：全部拒绝

它的实现非常直接：Filter() 清空 OutAllowedObjects。BaseEngine.ini 里用名称 NotRouted 注册它，可把某类对象彻底挡在常规复制路径之外。注意：名字叫 NotRouted 不代表类名也是 NotRouted。

UE 5.5 源码节选（语义保持，省略签名细节）：

void UFilterOutNetObjectFilter::Filter(FNetObjectFilteringParams& Params)
{
    Params.OutAllowedObjects.ClearAllBits();
}

UNopNetObjectFilter：全部允许

Nop 的行为正好相反，它把允许位数组全部置位，效果等同于动态过滤阶段不剔除任何对象。它适合测试自定义过滤器接线、对比性能或作为最小参考实现；UE 5.5 默认 BaseEngine.ini 并未给它注册名称，若要通过 GetFilterHandle 获取，需要自行配置定义。

void UNopNetObjectFilter::Filter(FNetObjectFilteringParams& Params)
{
    Params.OutAllowedObjects.SetAllBits();
}

UNetObjectConnectionFilter 与 SetConnectionFilter 不是一回事

这是最值得单独强调的命名陷阱：

方案性质适用场景
UReplicationSystem::SetConnectionFilter内建的静态连接位集过滤对象较少、允许连接集合较稳定；通常优先使用
UNetObjectConnectionFilter动态 pre-poll 过滤器对象可能是 Dependent Object 等必须依赖动态过滤语义的情况

UE 5.5 头文件明确说明，动态连接过滤器并非为海量对象设计，默认配置的 MaxObjectCount 为 4096。若只是“这 20 个管理对象只给 3 个裁判连接”，轻量的 SetConnectionFilter 更清晰。

🗺️ 5.4 空间过滤器详解（GridFilter）

为什么先用网格，而不是每连接扫描全世界

假设场上有 50,000 个可复制对象和 100 条连接。朴素做法每帧要做 500 万次距离判断。GridFilter 先通过网格把候选集合缩到观察位置周围的若干单元格，再做精确判断，计算量从“全世界 × 所有连接”变成“局部候选 × 所有视图”。

世界对象
  └─ 按 CullDistance 覆盖范围登记到一个或多个二维 Cell

连接的 Replication View
  └─ 找到当前 Cell 和最近经过的 Cell
       └─ 收集候选对象
            ├─ bUseExactCullDistance = false → 单元格内直接允许
            └─ bUseExactCullDistance = true  → 再做 DistSquared 检查

UE 5.5 源码里的实际步骤

  1. OnInit 添加 Spatial Trait，读取 Grid 配置并建立每连接缓存。
  2. AddObject 读取对象世界位置与剔除距离，计算它覆盖的单元格；对象可能跨多个 Cell。
  3. PreFilter 推进帧计数。
  4. Filter 遍历连接的每个 Replication View，收集相关 Cell。
  5. 若启用精确距离，对候选对象执行距离平方判断，并更新短期保留计数。
  6. UNetObjectGridWorldLocFilter::PreFilter 更新位置已变脏对象的 Cell 信息。

源码还包含一个容易忽略的保护：对象的 Cull Distance 若超过配置的 MaxCullDistance，会从 GridFilter 移除并按 always relevant 处理。若误把 Max 设置得过小，结果可能不是“更积极地裁剪”，而是对象反而全局相关。

核心参数怎么理解

参数作用调优思路
CellSizeX / CellSizeY二维网格尺寸太小会让对象跨很多 Cell、维护成本上升；太大则候选对象过多
DefaultCullDistance没有对象级覆盖时的默认剔除距离按游戏信息密度和移动速度设基线
MaxCullDistance过滤器能处理的最大距离必须覆盖业务允许的最大值；0 在默认配置中表示不设上限
bUseExactCullDistance是否在 Cell 候选后做精确距离检查更准确但增加 CPU；密集场景通常值得开启
ViewPosRelevancyFrameCount非精确模式下保留近期视图 Cell减轻高速移动造成的边界闪烁
DefaultFrameCountBeforeCulling精确模式下对象越界后的短期保留帧数在稳定性与“多留几帧”的成本间折中
FilterProfiles按 Profile 覆盖行为玩家、载具、掉落物可用不同保留帧数

UE 5.5 源码构造时的默认值包括：Cell 约 20000×20000、DefaultCullDistance 15000、精确距离开启、默认保留 4 帧。它们是引擎默认，不是项目最佳值。厘米制世界里，20000 代表 200 米；你的地图密度、速度和服务器 TickRate 会改变最优点。

边界抖动与两层滞后

玩家在 Cell 边界或 Cull Distance 附近高速移动时，如果“进一帧、出一帧”，远端对象可能反复创建和销毁。GridFilter 自身通过近期 Cell / 对象帧计数缓和空间边界；过滤系统还提供 Object Scope Hysteresis，在动态过滤器判定出界后继续保留若干帧。二者都能抗抖，但叠加过大也会让对象在视野外滞留太久。

👥 5.5 组过滤（Group Filtering）

组不是标签，而是一条批量规则

Group Filtering 的价值在于：连接规则改变时，不必逐对象修改。你可以把所有蓝队信标、任务点、复活倒计时加入同一个组，然后只对组设置“蓝队连接允许”。新对象加入组后自动继承这条连接规则。

Exclusion 与 Inclusion 的方向相反

组类型默认状态执行位置典型用途
Exclusion Group默认拒绝组内对象动态过滤器之前隐藏尚未开放的活动对象、隔离测试对象
Inclusion Group默认不覆盖动态过滤动态过滤器之后让队伍对象无视距离，强制对队友可见

Inclusion Group 是解决“队伍标记在地图另一端也要看见”的关键：GridFilter 可能把它判为距离过远，但后执行的 Inclusion 状态可以把对象重新纳入指定连接的作用域。

UE 5.5 组过滤示例

API 组合示例，句柄获取与错误处理已简化：

using namespace UE::Net;

FNetObjectGroupHandle BlueTeamGroup =
    ReplicationSystem->CreateGroup(TEXT("BlueTeamMarkers"));

ReplicationSystem->AddInclusionFilterGroup(BlueTeamGroup);

for (FNetRefHandle MarkerHandle : BlueTeamMarkerHandles)
{
    ReplicationSystem->AddToGroup(BlueTeamGroup, MarkerHandle);
}

for (uint32 ConnectionId : BlueTeamConnectionIds)
{
    ReplicationSystem->SetGroupFilterStatus(
        BlueTeamGroup,
        ConnectionId,
        ENetFilterStatus::Allow);
}

组不再使用时要先取消过滤影响并销毁组；成员阵营变化时及时 RemoveFromGroup / AddToGroup。连接重连会获得新的 ConnectionId，不要把旧 ID 当成永久玩家身份。

子对象的特殊规则

UE 5.5 头文件明确说明：加入 Inclusion Group 的 SubObject 在处理时会被忽略，它的状态由 Root Object 决定。需要单独过滤子对象时，应使用 SubObject Filter API。把武器组件塞进 Inclusion Group，却没有处理所属角色 Root，很容易得到与预期相反的结果。

⚙️ 5.6 过滤器配置与完整实战

第一层：注册动态过滤器定义

名称是运行时 API 与实现类之间的桥。UE 5.5 的 BaseEngine.ini 默认注册 Spatial 与 NotRouted：

配置示例：

[/Script/IrisCore.NetObjectFilterDefinitions]
+NetObjectFilterDefinitions=(FilterName=Spatial, ClassName=/Script/IrisCore.NetObjectGridWorldLocFilter, ConfigClassName=/Script/IrisCore.NetObjectGridFilterConfig)
+NetObjectFilterDefinitions=(FilterName=NotRouted, ClassName=/Script/IrisCore.FilterOutNetObjectFilter, ConfigClassName=/Script/IrisCore.FilterOutNetObjectFilterConfig)

第二层：给类分配过滤器

UObjectReplicationBridgeConfig 可按类路径绑定 DynamicFilterName 和可选的 FilterProfile。更具体的类配置优先；实例自身的 Always Relevant / Only Relevant To Owner 语义也会影响是否采用类配置，除非设置 bForceEnableOnAllInstances。

[/Script/IrisCore.ObjectReplicationBridgeConfig]
DefaultSpatialFilterName=Spatial
!FilterConfigs=ClearArray
+FilterConfigs=(ClassName=/Script/MyGame.BRLootActor, DynamicFilterName=Spatial, FilterProfile=Loot)
+FilterConfigs=(ClassName=/Script/MyGame.BRVehicle, DynamicFilterName=Spatial, FilterProfile=Vehicle)
+FilterConfigs=(ClassName=/Script/MyGame.BRGlobalMatchState, DynamicFilterName=None)

第三层：配置 Grid 与滞后

项目起点示例，不是通用推荐值：

[/Script/IrisCore.NetObjectGridFilterConfig]
CellSizeX=20000
CellSizeY=20000
DefaultCullDistance=15000
MaxCullDistance=100000
bUseExactCullDistance=true
DefaultFrameCountBeforeCulling=4
ViewPosRelevancyFrameCount=2
+FilterProfiles=(FilterProfileName=Loot, FrameCountBeforeCulling=2)
+FilterProfiles=(FilterProfileName=Vehicle, FrameCountBeforeCulling=8)

[/Script/IrisCore.ReplicationFilteringConfig]
bEnableObjectScopeHysteresis=true
DefaultHysteresisFrameCount=6
HysteresisUpdateConnectionThrottling=1

若升级引擎后配置不生效，第一件事不是继续猜数值，而是打开目标版本的 Config UCLASS / USTRUCT，确认 Section、属性名以及数组写法。Experimental API 的变化风险要显式纳入升级检查表。

第四层：运行时设置

运行时 API 示例：

// 给玩家或载具设置已注册的 Spatial 动态过滤器
const FNetObjectFilterHandle SpatialHandle =
    ReplicationSystem->GetFilterHandle(TEXT("Spatial"));

const bool bApplied =
    ReplicationSystem->SetFilter(ObjectHandle, SpatialHandle, TEXT("Vehicle"));

// 私人背包：先设置拥有连接，再使用特殊的 ToOwnerFilterHandle
ReplicationSystem->SetOwningNetConnection(InventoryHandle, OwnerConnectionId);
ReplicationSystem->SetFilter(
    InventoryHandle,
    UE::Net::ToOwnerFilterHandle);

// 裁判对象：使用轻量连接位集过滤
TBitArray<> AllowedConnections;
AllowedConnections.Init(false, MaxConnectionCount);
AllowedConnections[ObserverConnectionId] = true;

ReplicationSystem->SetConnectionFilter(
    RefereeDataHandle,
    AllowedConnections,
    UE::Net::ENetFilterStatus::Allow);

务必检查 GetFilterHandle 和 SetFilter 结果。SetFilter 失败后，官方头文件说明对象过滤状态可能保持旧值，也可能没有过滤，不能假设失败等同于“安全地全部拒绝”。

完整战术竞技方案

对象实现状态变化时的操作
玩家 / 载具 / 掉落物类配置 Spatial；按类型选择 Profile位置变脏由 WorldLoc 过滤器刷新；改变剔除距离时同步对象过滤信息
队伍标记 / 队伍任务加入对应队伍的 Inclusion Group换队时迁移组；连接加入队伍时设置 Allow
背包 / 私人任务Owning Connection + ToOwnerFilterHandle所有权转移时更新 Owning Connection
裁判数据SetConnectionFilter 允许位集裁判加入/退出时重建位集
全局比赛状态DynamicFilterName=None保持全局相关，交给后续优先级与带宽系统

🔬 5.7 过滤系统内部实现

FReplicationFiltering::Filter() 的真实流水线

UE 5.5 的 ReplicationFiltering.cpp 中，顶层过滤流程按以下顺序组织：

ResetRemovedConnections
InitNewConnections
UpdateObjectsInScope
UpdateGroupExclusionFiltering
UpdateGroupInclusionFiltering
UpdateOwnerAndConnectionFiltering
UpdateSubObjectFilters
PreUpdateObjectScopeHysteresis
UpdateDynamicFilters
FilterNonRelevantObjects

不要把这个列表误读成“每一步都直接覆盖上一步”。系统会维护多组位数组：连接过滤、组排除、动态过滤前作用域、组包含、动态过滤结果和滞后调整结果。最终通过位运算组合，形成每条连接的 ObjectsInScope。

动态过滤内部又分四步

NotifyFiltersOfDirtyObjects
        ↓
PreUpdateDynamicFiltering
        ↓
UpdateDynamicFiltering（逐连接调用 Filter）
        ↓
PostUpdateDynamicFiltering

每个动态过滤器只负责分配给自己的对象。它把允许对象写入 OutAllowedObjects，过滤系统再将这些位合并到该连接的动态过滤结果中。没有分配给动态过滤器的对象不会凭空被它处理。

为什么 Inclusion Group 能“救回”远距离对象

源码组合动态过滤结果时，会用 Group Included Objects 去清除相应的 filtered-out 位。因此 Inclusion 语义是对动态过滤结果的覆盖，而不是在 GridFilter 之前给对象贴一个普通标签。队伍标记的业务需求正好利用了这个顺序。

// 解释性伪代码：表达位数组关系，不是源码逐字复制
PreDynamicScope =
    GlobalScope
    - OwnerOrConnectionFilteredOut
    - ExclusionGroupObjects
    - SubObjectFilteredOut;

DynamicFilteredOut =
    RunAssignedDynamicFilters(PreDynamicScope);

DynamicFilteredOut -= InclusionGroupAllowedObjects;

ObjectsInScope =
    ApplyHysteresis(PreDynamicScope - DynamicFilteredOut);

全局 Relevant List 的最后收口

FilterNonRelevantObjects() 会把 always relevant 对象与各连接作用域做汇总，形成全局 relevant 列表。源码构建 always relevant 列表时，会排除已经使用 Owner、Connection、Dynamic 或 Group Filter 的对象，避免把有明确过滤规则的对象错误地当作全局相关。

Shared Connection 的合并语义

父子连接（例如分屏连接）会共享过滤状态。UE 5.5 的 FSharedConnectionFilterStatus 默认不允许；只要父或任一子连接允许，合并状态就允许。调试时若发现“主连接被禁用但仍在复制”，要检查同组的 Child Connection 是否仍为 Allow。

排查顺序建议固定为：对象是否进入 Iris → 句柄是否有效 → 静态/组过滤后的 pre-dynamic scope → 动态过滤结果 → Inclusion 覆盖 → Hysteresis → 最终 ObjectsInScope。直接从“网络没包”倒推，通常会绕远路。

⚡ 性能建议

  1. 先用便宜规则缩小集合。Owner、Connection 和 Group 能表达的业务，不要强行写成逐连接动态过滤。
  2. 动态过滤器只缓存必要数据。把 UObject 查询转换为紧凑数组或位集，在 UpdateObjects 更新。
  3. 避免 Filter 内分配。逐连接热路径中不要创建临时 UObject、TMap 或大数组。
  4. 测候选数量，不只测最终数量。Grid 的最终过滤率很高，不代表 Cell 候选集合就小；Cell 过大仍会浪费距离判断。
  5. 按对象类型分 Profile。高速载具需要更长抗抖，静态掉落物可以更快剔除。
  6. 把 Hysteresis 当稳定性预算。保留帧越多，创建销毁越平滑，但作用域和带宽占用也越久。
  7. 用真实人数和地图密度压测。10 个机器人下的参数，不能代表 100 人、50,000 个对象的服务器。

最终优化目标不是“过滤率越高越好”，而是服务器 CPU、对象创建销毁频率、连接带宽和玩家感知之间的平衡。过滤过狠造成关键对象晚出现，同样是网络体验问题。

🎯 本章小结

  • 过滤决定对象是否属于某条连接的复制作用域；优先级只决定通过过滤后谁先发送。
  • Owner、Connection、Group、Dynamic 是四类主要手段，应优先选择成本更低、语义更稳定的方案。
  • UNetObjectFilter 用批量回调和位数组工作；PreFilter 每帧一次，Filter 按连接执行。
  • GridFilter 先按 Cell 缩小候选集合，再可选地做精确距离判断，并用帧计数与 Scope Hysteresis 抗抖。
  • Exclusion Group 在动态过滤前排除；Inclusion Group 在动态过滤后允许，可让队伍对象覆盖空间距离结果。
  • UNetObjectConnectionFilter 与 SetConnectionFilter 是不同方案；海量对象不要误用前者。
  • 调试必须尊重源码组合顺序，尤其关注 Inclusion、SubObject、Shared Connection 和 MaxCullDistance 的特殊语义。
