🎯 第十部分导读：只发脏成员，为什么还需要 Baseline？

前九部分已经把 Iris 的状态、调度、序列化、传输和 NetBlob 串了起来。回到普通状态复制时，一个很自然的疑问是：既然 ChangeMask 已经能筛出 Health、Ammo 这类发生变化的成员，为什么还要再引入 Delta Compression？

答案是：“要发哪些成员”与“这些成员怎样编码”是两个问题。ChangeMask 可以让未变化成员不进入本次成员序列化，但一个脏的 32-bit 值仍可能按完整值写入。Delta Compression 再进一步，让成员相对“这条连接已经确认收到的量化状态”编码；这份双方认可的历史状态，就是连接级 Baseline。

ChangeMask 做集合裁剪

决定本次遍历和写入哪些成员。它回答“谁脏了”，不回答“脏成员相对什么值编码”。

Baseline 做历史对齐

为每条连接选择一份已确认的量化状态。不同连接的 ACK 进度不同，Baseline 也会分叉。

Serializer Delta 做位编码

具体类型读取 Source 与 Prev，自行决定写差值、相等标记还是回退完整值。

Delta 不保证更小

索引、相等位、稀疏 ChangeMask 和 Baseline header 都有成本；值差很大时，成员级 Delta 甚至可能比 full 多 1–2 bits。

Baseline 不是“服务器上一帧状态”，而是针对某个对象、某条连接、且发送端确认接收端已经拥有的一份量化快照。

🗺️ 三层压缩地图：Mask、Baseline、Serializer

把“增量压缩”拆成三层，能避免绝大多数误解。第一层来自 Poll/Push Model 和脏数据追踪；第二层由 Writer 与 FDeltaCompressionBaselineManager 协作；第三层才进入具体 FNetSerializer。

LAYER 01 · SETChangeMask

从 Protocol 成员集合中选择本次需要编码的成员，并用稀疏格式进入位流。

LAYER 02 · HISTORYPer-connection Baseline

用 2-bit index 指向接收端已确认保存的量化快照，决定 Prev 是谁。

LAYER 03 · VALUESerializeDelta

逐个脏成员比较 Source / Prev，按类型选择差值表或回退路径。

解释性调用链UE 5.7.4
Poll / Push Model
  → dirty objects + ChangeMask
  → ReplicationWriter selects LastAckedBaselineIndex per connection
  → SerializeWithMaskDelta(Current, Prev, ChangeMask)
  → member FNetSerializer::SerializeDelta(Source, Prev)
  → bitstream

三个常见但错误的等号

错误理解实际边界
ChangeMask = DeltaChangeMask 只选成员；被选中的成员仍可能完整序列化。
Baseline = 当前服务器状态Baseline 是针对某连接已确认的量化快照；当前状态可以比它新很多。
SerializeDelta = 数学减法它是 Serializer 自定义契约。float 当前按位模式走整数 Delta packing，并非直接写浮点差值。

完整 Protocol 的 Delta 调用仍会写稀疏 ChangeMask。换句话说，“哪个成员出现”与“成员值怎样出现”同时影响位数，不能只拿成员 payload 做最终包体估算。

📉 10.1 增量压缩概述

原理：以已签收版本为底稿

可以把当前量化状态看成一份新文档，把 LastAckedBaselineIndex 指向的状态看成接收端已签收的旧文档。发送端先写 Baseline 索引，再写稀疏 ChangeMask；具体成员只需表达“相对旧文档怎样变化”。接收端用同一槽位中的快照作为 Prev，反向恢复当前量化值。

关键限定是“已签收”。Writer 可以把当前状态保存成 pending Baseline，但承载它的发送记录在 Delivered 之前，接收端是否拥有这份快照仍不确定。因此 pending 不能立即替换 LastAckedBaselineIndex。

2MaxBaselineCount：每对象、每连接两个槽位
2InvalidBaselineIndex：0/1 之外的无效值
2 bitsBaselineIndexBitCount：能表示 0、1、2
60 帧后续新 Baseline 创建的默认最小间隔

适用场景

更可能有收益原因更可能不划算原因
状态较大、更新频繁Baseline 和 header 成本可由多次发送摊薄。一次性或寿命很短的对象尚未建立可复用的已确认 Baseline。
数值变化幅度通常较小整数差值表容易落入短编码。小状态、随机大跳变Delta 索引可能让 payload 反而变大。
连接稳定、ACK 及时已确认历史持续可用，重建较少。高丢包或条件频繁变化pending 丢失、失效与重传增加额外工作。
成员多数局部变化稀疏 ChangeMask 与成员 Delta 同时受益。动态状态克隆昂贵数组、字符串或嵌套动态成员会增加快照内存和 clone/free CPU。

性能收益要算净值

工程上应比较“不开 Delta 的对象状态总 bits”与“开启后的对象状态总 bits”，再同时记录 CPU 和内存，而不是只看一个整数成员的漂亮样例。净收益至少要扣除：

  • Baseline index，以及 valid 分支中的 bIsNewBaseline；
  • 稀疏 ChangeMask 和成员 Delta 自己的索引/相等位；
  • 发送端量化快照、ChangeMask 历史和接收端 Stored Baseline；
  • 动态状态 clone/free、失效处理、丢包重发与多连接查找。
测量边界：本文后面的 8/16-bit 数字仅指成员 payload 模型，不包含对象句柄、batch、稀疏 ChangeMask、Baseline header、DataStream、Packet header 或加密成本。

🗃️ 10.2 DeltaCompressionBaselineManager

Manager 管的不是一个“全局旧状态”

FDeltaCompressionBaselineManager 的核心职责是把对象、连接、槽位和量化快照关联起来。它在 PreSendUpdate 中更新帧计数、同步 scope、累积 dirty ChangeMask、处理条件失效并建立本次发送的共享上下文；在 PostSendUpdate 中提交或释放该上下文。

AddConnection 当前没有额外工作；RemoveConnection 会释放这条连接持有的 Baseline。对象退出可用范围或复制条件改变时，FDeltaCompressionBaselineInvalidationTracker 会让相关槽位失效，Writer 随后回退到无有效 Baseline 的路径。

两槽位为什么需要 2-bit index

源码常量为 MaxBaselineCount = 2、InvalidBaselineIndex = 2、BaselineIndexBitCount = 2。槽位 0/1 用来在旧的已确认快照和新的 pending 快照间轮换；数值 2 表示“没有可用 Baseline”，所以索引必须能表达三个状态，需要 2 bits。

Writer 持有 LastAckedBaselineIndex 与 PendingBaselineIndex；Reader 则持有 LastStoredBaselineIndex、PrevStoredBaselineIndex 和两个接收快照。新槽位通常按 (LastAcked + 1) % 2 选择。它们不是两个全局版本，也不是让两条连接共用同一个 ACK 状态。

快照存储：量化内部状态，不是 UObject 内存拷贝

FDeltaCompressionBaselineStorage 通过 FReplicationStateStorage reserve / commit / cancel 快照。底层按 Protocol 的 InternalTotalSize 分配，克隆的是量化后的内部状态；若 Protocol 含 dynamic state，还会调用对应 Protocol operations 深拷贝并在释放前清理。

每份内部 Baseline 带引用计数。同一 SendUpdate 内，多条连接为同一对象创建当前状态 Baseline 时，FBaselineSharingContext 可以只保存一次量化快照，再由各连接 AddRef。共享的是当前状态的存储；连接各自的槽位、累计 ChangeMask、pending/acked 生命周期仍然独立。

三份 mask 与 Lost 的历史修复

Manager 为每对象分配的 ChangeMask 区域，按连接包含三份逻辑 mask：一份连接累计 mask，加上两个 Baseline 槽位各自捕获的 mask。UpdateDirtyStateMasks 把每帧 dirty bits OR 到所有有效连接的累计 mask；CreateBaseline 把当前连接累计 mask 复制到新槽位，并清空累计值。

DestroyBaseline 只释放快照并丢弃槽位 mask；LostBaseline 则释放 pending 快照，同时把它捕获的 mask 合并回 Manager 的连接累计 mask。后者维护的是未来 Baseline 的历史完整性。真正让丢失成员进入下一次状态重发的，是 Writer 处理 dropped record 时，把该包的 RecordInfo.ChangeMask（扣除已经重发的 bits）恢复到 Writer 当前连接的发送 mask。两条修复链协作，但职责不同。

创建节流与失效

  • net.Iris.MinimumNumberOfFramesBetweenBaselines 默认 60；对象完全没有 Baseline 时可以先建立第一份。
  • PrevBaselineCreationFrame 是 per-object 状态，不是每连接独立计时。同一对象的后续创建共同受节流。
  • CreateBaseline 返回的快照只保证当前 SendUpdate 可用；Manager 可以因条件或内部策略失效它。
  • 条件成员的可见性改变时，旧 Baseline 可能不再是合法参照，相关连接或所有连接的槽位会被释放。

🧬 10.3 增量序列化：SerializeDelta / DeserializeDelta

发送与接收调用链

发送端Writer → Serializer
FReplicationWriter::SerializeObjectStateDelta
  → FReplicationProtocolOperations::SerializeWithMaskDelta
  → FReplicationStateOperations::SerializeDeltaWithMask
  → FNetSerializer::SerializeDelta
       Args.Source = current quantized member
       Args.Prev   = acknowledged quantized member
接收端Reader → Serializer
FReplicationReader::DeserializeObjectStateDelta
  → FReplicationProtocolOperations::DeserializeWithMaskDelta
  → FReplicationStateOperations::DeserializeDeltaWithMask
  → FNetSerializer::DeserializeDelta
       Args.Prev = StoredBaselines[BaselineIndex]

SerializeWithMaskDelta 先为完整 Protocol 写稀疏 ChangeMask，再只对脏成员调用 Delta；接收端用同样的 mask 和 Baseline 反向解码。FNetSerializeDeltaArgs 与 FNetDeserializeDeltaArgs 中的 Prev 都指向量化的历史数据，而不是业务 UObject。

默认 Delta：一个相等位，不等就回退

NetSerializer builder 的默认 Delta 实现可以概括为：先写一位表示 Source 是否等于 Prev；相同就结束，不同则调用普通 Serialize 写完整值。这个默认实现提供统一语义，却不承诺对“不同值”压缩——不同值通常是 1 bit + full payload。

same = IsEqual(Source, Prev)
WriteBool(same)
if (!same)
    Serialize(Source)

整数 Delta：表驱动的差值范围

整数 Serializer 会先计算无符号位模式差异，再用预设位宽表选择足以编码差值的档位。UE 5.7.4 当前表包括：≤8-bit 使用 {0}；≤16-bit 使用 {0, 4, 10}；≤32-bit 使用 {0, 4, 14}；≤64-bit 使用 {0, 14, 32}。索引本身也占位，所以大差异会回退到完整位宽并增加索引成本。

成员类型Delta 表相同小差异示例最大档位
8-bit integer{0}，1-bit index1 bit没有中间档1 + 8 = 9 bits
16-bit integer{0,4,10}，2-bit index2 bits2 + 4 = 6 bits；或 2 + 10 = 12 bits2 + 16 = 18 bits

浮点、结构与动态状态

float/double 的 Delta 不是直接做浮点减法。当前实现对量化后的位模式调用整数 Delta packing：float 表为 {0,16,25}，double 表为 {0,42,54}。相近数值不一定具有相近的位模式成本，必须根据真实数据分布测量。

Array、Struct 等 Serializer 可以提供自定义 Delta 和动态状态处理。Baseline 存储需要完整 clone 动态数据，在销毁或失效时也必须 free；因此一个看起来很省带宽的动态数组 Delta，可能把成本移到内存分配和 CPU。

初始状态 Delta 是另一条优化路径

net.Iris.DeltaCompressInitialState 默认开启时，创建对象的 initial state 可以相对 Protocol default state 调用 Delta 序列化。它的 Prev 是默认量化状态，不是某条连接已经 ACK 的对象 Baseline。不要用“初始状态也走 Delta”推导出该连接已经建立 LastAckedBaselineIndex。

🎛️ 10.4 增量压缩配置：允许不等于保证

对象真正进入 Delta 路径要通过多道门。SetDeltaCompressionStatus 官方 API 明确使用“allow”语义：调用它只是允许对象使用，不保证本次发送一定使用 Delta。

类级配置

UObjectReplicationBridgeConfig 提供 DeltaCompressionConfigs。Bridge 加载配置后按类判断；子类继承配置，距离实际类最近的匹配项优先。下面是项目配置示意，类路径必须替换为项目真实类型：

DefaultEngine.ini · 项目示例按实际类路径调整
[/Script/IrisCore.ObjectReplicationBridgeConfig]
+DeltaCompressionConfigs=(ClassName=/Script/MyGame.MyReplicatedPawn,bEnableDeltaCompression=true)
+DeltaCompressionConfigs=(ClassName=/Script/MyGame.OneShotActor,bEnableDeltaCompression=false)

UE 5.7.4 的引擎基础配置默认为 Pawn、PlayerState 和 GameplayDebuggerCategoryReplicator 启用这一策略。项目应通过自己的配置层追加或覆盖，不应修改 Engine/BaseEngine.ini。

运行时 API 与 Protocol 支持

Bridge 在绑定对象时可以调用 UReplicationSystem::SetDeltaCompressionStatus(..., Allow)。Manager 仍会同时检查全局开关与 Protocol trait。Protocol Manager 只有在组合 Replication State traits 支持 Delta 且 InternalTotalSize > 0 时，才标记 EReplicationProtocolTraits::SupportsDeltaCompression。

普通带成员、非 init-only、非 function 的 Replication State 通常会被 descriptor builder 标记为支持；但最终 Protocol 是组合结果，项目自定义 Serializer 或 State trait 仍需核对。

容量与内存权衡

FReplicationSystemParams::MaxDeltaCompressedObjectCount 在 UE 5.7.4 默认是 2048。Manager 最终容量还会取 NetObject 上限与该值的较小者；NetDriver 配置为 0 时使用 ReplicationSystem 默认。达到容量并不意味着普通复制失败，而是额外的 Delta 对象管理不能无限增长。

推荐先对白名单类启用，并按对象数量、连接数、Protocol 内部状态大小与动态成员比例估算，再用 NetTrace/统计验证。不要仅因为全局 CVar 默认 true 就把所有类都当作已经获得收益。

🔀 双连接、同一对象：ACK / Lost 后的 Baseline 分叉

用一个对象和两条连接走完生命周期。状态依次为：

S0 = { Health = 100, Ammo = 30 }
S1 = { Health =  80, Ammo = 30 }
S2 = { Health =  80, Ammo = 25 }

A 与 B 都已确认 S0。服务器把 Health 改为 80，并在本次 SendUpdate 为两条连接发送 S1、尝试建立新 Baseline。FBaselineSharingContext 可让两条连接引用同一份 S1 量化快照，但它们拥有独立的发送记录和投递结果。

A：Delivered 推进 Baseline

A 的发送记录 Delivered 后，Writer 销毁旧的 last-acked 引用，把 PendingBaselineIndex 提升为 LastAckedBaselineIndex，再清空 pending。下一次 Ammo 变化时，A 可以相对 S1 编码。

B：Lost 同时触发两条修复链

B 的记录 Lost 后不能假设接收端存好了 S1。恢复分成两个相邻但不同的动作：

  1. Writer 的发送状态修复：HandleDroppedRecord<Created> 把丢包记录中的 RecordInfo.ChangeMask 恢复到 B 当前发送 mask；如果其中某些 bits 已在后续包重发，会先剔除，避免重复计入。
  2. Manager 的 Baseline 历史修复：LostBaseline 释放 B 的 pending snapshot，并把该 Baseline 捕获的 ChangeMask 合并回 Manager 的 per-connection accumulated mask，保证未来创建 Baseline 时不会遗漏历史变化。

因此，下一次 B 发送 S2 时，Writer 会包含仍未送达的 Health 以及新变化的 Ammo；Baseline 选择仍退回 S0 或无有效 Baseline 路径。不能把“Manager merge”单独解释成下一包重发的唯一来源。

同一状态，位流可以不同

到了 S2，A 的 Prev=S1，只需处理 Ammo；B 的 Prev=S0，必须让 Health 与 Ammo 都能正确收敛。服务端对象只有一份当前状态，但每条连接的 ACK 历史、mask、Baseline index 和位流都可以不同——这正是 Baseline 必须逐连接管理的原因。

📡 Writer → Wire → Reader：一次 Delta 的完整旅程

Writer 选择 full 还是 delta

FReplicationWriter::SerializeObjectStateDelta 总会先写 2-bit BaselineIndex。若 LastAckedBaselineIndex 有效，Writer 再写 1-bit bIsNewBaseline，并调用 SerializeWithMaskDelta；若无效，则写 2-bit CreatedBaselineIndex，随后用普通 SerializeWithMask 写完整成员值。

若当前没有 pending Baseline，Writer 会尝试创建新槽位：没有 last-acked 时从 0 开始，否则轮换到另一个槽位。新 Baseline 还可能用 in-flight changes 修补，确保接收端创建出的快照与随后相对它编码的数据一致。只有在本次 batch 成功提交后，Writer 才设置 PendingBaselineIndex。

Reader 选择 Prev 并保存新槽位

FReplicationReader::DeserializeObjectStateDelta 读取同样的 2-bit BaselineIndex。有效时，它读取 bIsNewBaseline，从 StoredBaselines[index] 取得 Prev，调用 DeserializeWithMaskDelta；需要新 Baseline 时使用轮换槽位保存解码后的当前 ReceiveState。

无效 index 分支会继续读取新 Baseline index，用普通 DeserializeWithMask 取得当前值，再将当前 ReceiveState 保存到声明的槽位。Reader 通过 LastStoredBaselineIndex 与 PrevStoredBaselineIndex 管理何时可以释放更旧快照。

投递反馈闭环

1 · Select

Writer 读取当前连接的 last-acked/pending 状态，验证 Manager Baseline 是否仍有效。

2 · Serialize

写 index、是否建新 Baseline、ChangeMask 和成员 full/delta payload。

3 · Commit

batch 成功进入发送记录后，才把新槽位标为 pending。

4 · Receive

Reader 选择 Stored Baseline，恢复当前状态，必要时保存新快照。

5A · Delivered

发送端把 pending 晋升为 last-acked，并释放旧槽位。

5B · Lost

发送 mask 与 Baseline 历史 mask 分别修复，pending 不得晋升。

注意 Baseline header 位数只是对象状态子格式的一部分。真实网络包还包含对象标识、batch、DataStream、packet、可靠性/加密等外层开销；本文没有把上图相加后宣称为“一个对象最终只需 N bits”。

成员级收益公式

仅用于分解思路 member saving = full member bits − delta member bits
object saving = full state bits − (baseline header + sparse mask + delta members)
system value = bandwidth saving − memory cost − CPU cost − loss/invalidation cost

默认 Delta 的相同值可能只花 1 bit，但正常情况下 ChangeMask 已经尽量排除了未变化成员，所以“相同值”是否大量进入 Delta 路径，要看 Serializer、条件状态和 mask 生成方式。不能把 1 bit 的最好情况乘以全部成员数。

压缩率优化顺序

  1. 先看对象选择：优先状态大、长寿命、更新模式稳定的类；不要全局盲开。
  2. 再看数值分布：统计差值落在哪个表档位，而不是只统计“成员发生了变化”。
  3. 分离 initial 与 steady state：创建峰值和稳定更新使用不同 Prev，收益不能混算。
  4. 按连接质量分组：高 RTT/高丢包连接会带来更多 pending 停留和重建。
  5. 同时采 CPU/内存：网络 bits 下降但动态 clone/free 或快照驻留飙升，仍可能是负优化。

⚖️ 内存、CPU 与失效成本

发送端内存不是简单的“对象大小 × 2”

Per-object metadata对象到 info 的映射、创建帧、Baseline 计数与状态标记
+
Per-connection masks × 3累计 mask + slot 0 mask + slot 1 mask

Quantized snapshots最多每连接两个逻辑槽位；同帧创建可引用共享
+
Dynamic state数组、字符串、嵌套结构的深拷贝与分配器开销

接收端也需要最多两个 Stored Baseline 和当前 ReceiveState。发送端初始化 BaselineStorage 的理论槽位容量按 MaxDeltaCompressedObjectCount × MaxConnectionCount × 2 规划，但同一帧多连接的新快照可以通过引用计数共享实际量化状态。共享能缓解快照重复，不会消除 per-connection mask 和生命周期数据。

CPU 成本落在哪里

  • 每帧把 dirty ChangeMask OR 到所有相关连接的累计 mask；
  • 创建 Baseline 时克隆量化状态，dynamic state 需要深拷贝；
  • 写/读稀疏 ChangeMask，并对脏成员执行 Serializer Delta；
  • 按连接查找 last-acked/pending，处理 in-flight changes；
  • Delivered/Lost 时更新引用、mask 和记录；
  • 条件变化、对象离开 scope 或连接移除时失效/释放快照。

失效为什么是正确性成本

一个 Baseline 只有在“双方对同一成员集合和条件可见性有一致理解”时才可作为 Prev。OwnerOnly、AutonomousOnly、SimulatedOnly、ReplicatePhysics 等条件切换，可能让旧快照对某连接包含了现在不可见或此前不可见的成员。继续相对它编码会产生错误状态，因此 Manager 宁可释放槽位、回退 full-with-mask，再重建干净历史。

这也是为什么“Baseline 命中率”不能无限追求：保留一份不再合法的历史比多发几个 bits 更危险。优化目标应是在保持条件正确性的前提下减少净成本。

推荐的启用策略

  • 从 1–3 个高价值长寿命类开始，而不是对所有复制对象开启。
  • 压测连接数要覆盖线上峰值，因为 mask 与槽位元数据按连接增长。
  • 对含动态数组/字符串的类单独记录 snapshot bytes 与分配次数。
  • 弱网测试必须包含连续丢包和条件切换，不只模拟固定 RTT。
  • 引擎升级后重新核对 CVar 默认值、Delta 表、Manager 布局和自动化测试。

💡 总结与下一步：从“相对谁”转向“谁脏了”

  • ChangeMask 选成员，per-connection Baseline 选历史，Serializer Delta 编码成员值；三层不可混为一谈。
  • 每对象、每连接有两个 Baseline 槽位；无效值为 2，所以索引固定为 2 bits。
  • pending 只有 Delivered 后才能晋升 last-acked；Lost 必须回退，不能假设接收端保存成功。
  • 丢包恢复有两条链：Writer 恢复状态重发 mask，Manager 合并 Baseline 历史 mask。
  • 同帧多连接可以共享量化快照引用，但连接各自保留槽位、mask 和 ACK/Lost 进度。
  • Delta 可能变大。8-bit 非零差值、16-bit 大差值和默认 Delta 的不同值都有额外索引成本。
  • 类级配置与 Allow API 只是门控之一；Protocol、容量、连接历史和全局 CVar 共同决定实际路径。
  • 正确优化需要同时测 bandwidth、CPU、memory、loss 与 invalidation，而不是只报一个理论压缩率。

第十部分回答了“当前成员相对哪份已确认历史编码”。下一部分进入这条链的更前端：Iris 怎样通过轮询、Push Model 与脏数据追踪判断当前状态中的谁需要复制，以及检测成本怎样影响整帧网络预算。
