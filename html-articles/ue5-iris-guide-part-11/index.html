🧭 第十一部分导读：状态不会自动进入网络

上一部分讨论了“当前脏成员相对哪份已确认 Baseline 编码”。这仍然跳过了更前面的问题：服务器上的 Health = 80 写入之后，复制系统怎样知道这个对象值得检查？如果世界里有十万个可复制对象，难道每帧都要把每个 UObject 的每个属性重新读取、量化并比较一遍吗？

Iris 的答案不是单选 Poll 或 Push，而是一套分层筛选：

Object对象级脏位回答“哪些对象要进入或绕过轮询门控”。
Member成员 Poll Mask 回答“对象内部哪些 Push 成员值得读取”。
Wire量化 ChangeMask 回答“哪些成员要进入后续传播与序列化”。
“标脏”是让对象或成员进入候选集，不是立即序列化，更不是保证本帧进入网络包。

这句话是全文的锚点。相关性过滤、优先级、带宽预算、DataStream 与 packet 仍在后面。即使调用 ForceNetUpdate，也只是更强地影响轮询候选，不会绕开所有发送阶段。

本文回答的六个问题

  1. Poll 与 Push 分别节省哪一种成本？
  2. FDirtyNetObjectTracker 为什么要同时保存三组位图？
  3. Push 属性已经标脏，为什么 FObjectPoller 仍然存在？
  4. 轮询频率限制器如何把同频对象错开，并允许 Dirty/Force 绕过？
  5. 对象脏位、Member Poll Mask 和 ChangeMask 为什么不能混为一谈？
  6. 如何在项目里正确声明 Push 属性、验证漏标并测量收益？

🗺️ 一帧全景图：从 Gameplay 写值到发送候选

先把 UE 5.7.4 中与本篇直接相关的顺序压缩成一条线。方法名来自 UReplicationSystem::NetUpdate、UObjectReplicationBridge::PreSendUpdate 和帧末清理路径；图中每个格子都不是“发包”的同义词。

01 · GAMEPLAY写值 / MarkDirty

Setter 修改属性；Push 宏可登记对象和 RepIndex。

02 · GATHERUpdateDirtyObjectList

Tracker 收集全局脏位，更新本帧与累计集合。

03 · SCHEDULEBuildPollList

相关对象、频率计数、Dirty/Force、Dormancy 合成轮询集合。

04 · CALLBACKPreUpdate

在复制读取之前让实例刷新派生状态；期间还可能新建子对象。

05 · LOCKFinalizeDirtyObjects

再次收集并锁住全局 dirty list，保证 Poll 看到稳定视图。

06 · COPYPollAndCopy

按 Poll/Push/Force 路径读取外部状态，产生待量化对象。

07 · QUANTIZEQuantizeDirtyStateData

量化并生成真正的成员 ChangeMask。

08 · SENDFilter / Prioritize

按连接筛选、排序、预算和序列化；可能延后发送。

09 · CLEANUPReconcilePolledList

只清除成功轮询对象对应的累计与 Force 位。

为什么收集 Dirty 要做两次？

帧开始的 UpdateDirtyObjectList 让调度阶段看到此前与本帧已有的脏状态；Bridge 在 PreUpdate 之后调用 FinalizeDirtyObjects，再执行 UpdateAndLockDirtyNetObjects，把回调期间新产生的标脏也纳入 Poll。锁定则防止多个 poller 在清理全局列表时互相踩踏。

随后 UpdateDirtyListPostPoll 把本帧 Dirty OR 进累计集合。帧末的 ReconcilePolledList(PolledObjects) 只清除确实被轮询的对象，没轮到的对象继续在累计集合中等待。

四个动作、四种含义

动作改变什么不保证什么
MARK_PROPERTY_DIRTY...登记 NetHandle 与 RepIndex 范围；Iris 可映射到成员 Poll Mask。不立即读取属性，不立即生成 ChangeMask。
MarkNetObjectStateDirty把对象级内部索引置入 Dirty 位图。不知道具体哪个成员发生变化。
ForceNetUpdate设置 Force 位并同时把对象标脏，可绕过频率门控。不保证相关性、优先级和带宽预算允许本帧发送。
ChangeMask bit = 1量化状态确认成员发生可复制变化。不保证每个连接都可见或本包有空间。

📮 11.1 脏数据检测机制概述：Poll、Push 与混合策略

什么是“脏数据”

可以把复制系统想成快递追踪。仓库里的一件货物被重新打包，并不意味着卡车已经出发：

  • 对象脏：仓库系统收到“这件货可能变了”的扫描记录；
  • 成员脏：复核后确认重量、目的地或内容中的哪些字段真的变化；
  • 连接可发送：这件货对某条线路相关、优先级足够并且车上有空间；
  • 投递确认：包已写出仍不等于对端收到，后面还有 ACK/Lost 与重发。

Iris 里的“Dirty”常常带有阶段限定。看到一个名为 Dirty 的 bit array，首先要问：它属于对象发现、外部状态 Poll、量化 ChangeMask，还是逐连接发送历史？

Poll：系统主动检查

传统 Poll 的思路是“到时间就读一次当前状态，与复制缓冲区比较”。它的优势是 Gameplay 写值代码无需承担通知契约；只要对象按期被轮询，变化最终会被发现。代价是大量对象没变化时，仍需执行 PreUpdate、读取、比较，甚至量化前的准备工作。

Push：Gameplay 主动通知

Push 把一部分责任交给写值方：属性变化时调用标脏宏，复制系统据此跳过完全没被标记的 fully-push 对象，或者只读取标记过的成员。优势是低变化率、大对象池中可以显著减少无效检查；代价是漏标会成为正确性缺陷，而且混合对象、对象引用与特殊 Fragment 仍可能需要 Poll。

Iris 为什么采用混合策略

POLL按期主动检查

适合变化来源分散、难以保证所有写入口都标脏的对象。

  • 契约简单
  • 空轮询成本高
  • 频率限制可控
PUSH写值时主动通知

适合写入口集中、对象多但大多数帧不变化的状态。

  • 可跳过 clean 对象
  • 可按 RepIndex 缩小成员范围
  • 漏标风险高
HYBRID同一对象混合 Fragment

Push 成员按标记读取，非 Push Fragment 继续 Poll，必要时刷新对象引用。

  • 渐进迁移
  • 边界更复杂
  • 不是零轮询

🧮 11.2 FDirtyNetObjectTracker：三组位图与跨帧状态

FDirtyNetObjectTracker 的职责不是逐成员比较，而是维护 ReplicationSystem 内部对象索引对应的粗粒度候选状态。UE 5.7.4 中它恰好保存 DirtyNetObjects、AccumulatedDirtyNetObjects 与 ForceNetUpdateObjects 三组 FNetBitArray：

DirtyNetObjects01001000本帧新标脏；net tick flush 结束时整体清空。
AccumulatedDirty...01011000跨帧保留，直到对应对象确实被轮询。
ForceNetUpdate...00010000显式强制请求；成功轮询后只清除对应位。

Mark 与 Force 的差别

MarkNetObjectStateDirty(ReplicationSystemId, InternalIndex) 最终走入 Tracker 的对象级 MarkNetObjectDirty。它只有内部索引，没有属性信息，因此只能表达“对象值得重新检查”。

ForceNetUpdate 会先设置 ForceNetUpdateObjects，再同时标记对象 Dirty。这样不仅频率调度能识别强制位，依赖 Dirty 的过滤或其他更新也不会遗漏该对象。

Force 的边界

它是“强制进入轮询候选”的语义，不是“强制绕过过滤、优先级和带宽预算”。如果对象对连接不相关、被过滤、优先级不够或本包预算耗尽，仍可能不在本帧发送。

更新、锁定、累计与清理

方法核心动作为什么需要
UpdateDirtyNetObjects从 Global Tracker 拉取；与本系统 scope 求交；OR 到本帧 Dirty，并维护 Accumulated。全局 Push 通知要投递到具体 ReplicationSystem。
UpdateAndLockDirtyNetObjects先更新，再锁住 global dirty list。Poll 阶段需要稳定的 per-property dirty 视图。
UpdateAccumulatedDirtyList把本帧 Dirty OR 进累计集合。本帧没轮询到的对象不能丢失变化。
ReconcilePolledList按 ObjectsPolled 清除 Accumulated 与 Force；整体清空当前 Dirty。只消费真实 Poll 成功的对象，保留延期对象。

FGlobalDirtyNetObjectTracker 与属性级 Push

Push 宏首先以 NetHandle 和 RepIndex 范围进入 FGlobalDirtyNetObjectTracker。如果 per-property tracking 可用，Protocol 中的 RepIndex → Fragment/Member 映射会把范围落到每个 Fragment 的 Member Poll Mask；否则至少保留对象级 dirty，后续做更宽的 Poll。

UE 5.7.4 的当前限制是：一个 Protocol 的 per-property Push 路径只支持单个 Push-based owner（源码有 OwnerIndex == 0 约束并关联 UE-278338）。多 owner、无效 RepIndex、禁用属性或某些 custom conditional 情况可能回退或忽略成员级细化。文章把它作为版本实现限制，而不是长期公共契约。

多个 ReplicationSystem / poller 怎样协作

每个 ReplicationSystem 向 Global Tracker 注册 poll handle。只有一个 poller 时可以在收集后立即重置全局 dirty；存在多个 poller 时，必须等各系统都有机会 gather，必要时还会把 global list 锁到 reset。FDirtyObjectsAccessor 把锁的生存期封装在作用域中，避免 Poll 期间列表被提前清空。

📣 11.3 Push Model：声明、标脏与 Legacy 适配层

Push 真正启用需要多道门

GATE 01编译支持

构建需要 WITH_PUSH_MODEL；没有该能力时宏不会形成有效通知链。

GATE 02运行期开关

net.IsPushModelEnabled 必须为真。源码默认 false，项目配置不能省略验证。

GATE 03Iris 模式

net.Iris.PushModelMode 在 5.7.4 源码默认 2，决定委托是否可运行时切换。

GATE 04属性契约

Lifetime 参数声明 Push；每个真实写入口调用正确的 Dirty 宏。

net.Iris.PushModelMode 的三个模式是：

  • 0：Iris Push 适配关闭，但允许运行时切换；
  • 1：启用，并采用不可运行时切换的直接委托；
  • 2：启用且允许运行时切换，委托调用时检查状态；这是 5.7.4 源码默认值。

但模式默认 2 不等于 Push 默认可用，因为全局 net.IsPushModelEnabled 的源码默认仍为 false。net.MakeBpPropertiesPushModel 默认 true，也只有在 Push Model 本身启用时才有意义。

一份完整而安全的 C++ 示例

ReplicatedPawn.cppUE 5.7.4 Push property
void AReplicatedPawn::GetLifetimeReplicatedProps(
    TArray<FLifetimeProperty>& OutLifetimeProps) const
{
    Super::GetLifetimeReplicatedProps(OutLifetimeProps);

    FDoRepLifetimeParams Params;
    Params.bIsPushBased = true;
    DOREPLIFETIME_WITH_PARAMS_FAST(AReplicatedPawn, Health, Params);
}

void AReplicatedPawn::SetHealth(int32 NewHealth)
{
    if (Health == NewHealth)
    {
        return;
    }

    Health = NewHealth;
    MARK_PROPERTY_DIRTY_FROM_NAME(AReplicatedPawn, Health, this);
}

这个 Setter 把“是否真的变化、写入新值、通知复制系统”放在一起，审查最容易。MARK_PROPERTY_DIRTY_FROM_NAME 是推荐的编译期名字形式；静态数组要使用带 STATIC_ARRAY_INDEX 或 STATIC_ARRAY 的专用宏。

宏族怎么选

宏用途建议
MARK_PROPERTY_DIRTY_FROM_NAME普通成员，通过类型与成员名在编译期取得 RepIndex。常规首选。
...FROM_NAME_STATIC_ARRAY_INDEX只标记静态数组的一个元素。索引明确时使用。
...FROM_NAME_STATIC_ARRAY标记整个静态数组。批量写入时使用。
COMPARE_ASSIGN_AND_MARK_PROPERTY_DIRTY比较、赋值、标脏合为一体。适合简单类型；昂贵结构体应评估比较和拷贝成本。
MARK_PROPERTY_DIRTY / UNSAFE使用 RepIndex 或缓存数据的底层形式。只有封装层确实需要时使用，避免手写索引漂移。

LegacyPushModel 做了什么

Iris 没有要求 Gameplay 换一套 Push 宏。LegacyPushModel 根据模式安装委托，把既有 Push Model 的 NetHandle 与 RepIndex 范围转发给 FGlobalDirtyNetObjectTracker::MarkNetObjectStateDirty。这样旧宏调用可接入 Iris 的对象位图和成员 Poll Mask。

三个高频陷阱

  1. 漏标：fully-push 属性发生变化但没有调用宏，Iris 可能把对象视为 clean 并跳过，客户端就收不到更新。
  2. 先标后写并假定快照已取：标脏只登记候选；真正读取在后面的 Poll/Copy 阶段。代码仍应让最终状态在轮询前稳定。
  3. 把 Push 当零轮询：hybrid 对象的非 Push Fragment 仍要 Poll；fully-push 对象也可能因为 WantsFullPoll、GC 引用刷新、多 owner 或特殊状态走更宽路径。
迁移建议

先选写入口集中、单元测试覆盖好、低变化率的大对象池；让所有写操作通过 Setter，再声明 Push。不要先把属性改成 Push，之后才去搜索遗漏写点。

🔎 11.4 FObjectPoller：把候选对象变成待量化状态

FObjectPoller 消费 Bridge 构建的 PollList 与 Tracker 的 Dirty 视图，读取对象外部状态并更新复制缓冲区。它的输出重点是 DirtyObjectsToQuantize 和 DirtyObjectsThisFrame，而不是网络包。

NORMALPollAndCopyObjects

遍历 PollList，按 InstanceProtocol 和 Fragment traits 读取、比较对象状态。

  • 可并行分片
  • 可过滤 clean fully-push 对象
  • 记录实际 PolledObjects
PUSH-AWAREPushModelPollObject

结合本帧/累计 Dirty、Member Poll Mask、WantsFullPoll 与 GC 状态选择窄或宽读取。

  • PollDirtyState
  • PollAllState
  • hybrid 继续 Poll 非 Push Fragment
FORCEDForcePollObject

单对象强制轮询和某些内部路径使用；根据 NeedsPoll 做全量读取或直接标记量化。

  • 清除 WantsFullPoll
  • 记录 wasted poll
  • 仍不代表直接发送

clean fully-push 对象怎样被过滤

当 net.Iris.Poll.FilterOutNonDirtyPushBasedObjects=true（5.7.4 源码默认）、Push 已启用，而且对象既没有本帧 Dirty、没有累计 Dirty、也没有 GC 影响时，Poller 可把 fully-push 对象从轮询循环中剔除。这是 Push 最直接的 CPU 收益来源之一。

注意它要求对象是 fully push-based。只要 Protocol 中仍有非 Push Fragment，就不能假定整个对象可跳过；Poller 可能只刷新 Push 部分的对象引用，同时继续读取非 Push 状态。

PushModelPollObject 的关键分支

  • 对象在累计或本帧 Dirty 集合中：先进入 DirtyObjectsToQuantize 与本帧 Dirty；
  • fully-push 且有 per-property tracking、又不要求 full poll：使用 PollDirtyState；
  • WantsFullPoll、多 owner 或 ChangeMask 不可信：回退 PollAllState；
  • 仅受 GC 影响：可以只 Poll object references；
  • hybrid：即使 Push 部分 clean，也要 Poll 非 full-push Fragment。

ForcePollObject 的两个结果

如果 InstanceProtocol 为空，直接返回。如果对象 NeedsPoll，它执行 PollAllState，比较发现变化后把对象加入量化集合；若没有变化，可计为 wasted poll。如果对象不需要外部 Poll，则直接标记 Dirty-to-quantize 和本帧 Dirty，表示内部状态仍需后续处理。

并行 Poll 的 5.7.4 实现细节

源码默认 net.Iris.NumPollingTasks=32，net.Iris.NumWordsToInterleave=16。一个 bit-array storage word 为 32 bits，因此 16 words 对应 512 个对象位，也就是 64 bytes 的块。任务交错领取块，避免稀疏对象集中在少数连续区域时负载不均。

只有允许并行、工作量足够且任务数大于 0 时才创建并行任务，否则同步执行。任务数与块大小都是 Private CVar 和当前默认，升级引擎或换平台后要重新核对，不应硬编码到项目逻辑。

⏱️ 11.5 FObjectPollFrequencyLimiter：把检查成本摊到不同帧

Push 解决“没通知就尽量不检查”，Frequency Limiter 解决“即使需要 Poll，也不必每帧检查”。它以对象内部索引为 key，维护四组核心数据：

  • FramesBetweenUpdates：每个对象的 uint8 周期；
  • FrameCounters：当前倒计时，同样是 uint8；
  • FrameIndexOffsets[256]：让相同周期对象从不同计数位置开始；
  • FrameIndex 与 MaxInternalHandle：推进帧和限定扫描范围。

PollFrequency 如何换算成帧周期

UE 5.7.4 UObjectReplicationBridge 换算 FramesBetweenUpdates = floor(MaxTickRate / max(0.001, PollFrequency))
if FramesBetweenUpdates > 0: --FramesBetweenUpdates
clamp to [0, 255]

PollFramePeriod=0 表示每帧轮询；255 表示每 256 帧命中一次。以 30Hz 为例，最大周期约 8.53 秒。频率是目标调度频率，不是网络投递 SLA：被 Poll 后仍要经过连接级过滤和发送预算。

相同频率为什么不会同帧尖峰

新对象设置周期时，不会全部从同一个 counter 开始。FrameIndexOffsets 会让同周期对象分布在周期窗口的不同槽位。测试 ObjectsToPollAreSpreadOutWhenCreatingObjectAndUpdatingOncePerFrame 与随机创建数量的测试都验证了这种错峰行为。

Update：underflow、重载与位图合成

每帧计数器减一，uint8 从 0 下溢到 255 表示到期，然后把计数器重载为配置周期。最终 Poll 集合可概括为：

简化集合表达式 ObjectsToPoll = (FrequencyDue ∪ DirtyOrForceOverride) ∩ RelevantObjects

x86 路径一次以两个 16-byte vector 处理 32 个对象；源码还提供 NEON 和标量 fallback。SIMD 优化的是大量计数器与位图的固定宽度更新，不会改变调度语义。

net.Iris.EnableForceNetUpdate 最容易被误读

Bridge 构建 PollList 时，5.7.4 的 net.Iris.EnableForceNetUpdate 源码默认 false，含义并非“完全禁用 ForceNetUpdate”：

  • false：累计 Dirty 就可以作为频率 override；普通 MarkDirty 也可能提前 Poll；
  • true：只有专用 ForceNetUpdate 位可绕过频率门控，普通 Dirty 等到频率到期。

这是兼容与调度策略开关。配置它之前应先决定项目要的是“任何 Dirty 都尽快 Poll”，还是“只有显式 Force 才打破频率预算”。

类配置、Actor 频率与子对象

UObjectReplicationBridgeConfig::PollConfigs 可按类覆盖 PollFrequency，并选择是否包含子类。出于性能原因不应直接对最宽泛的 Object/Actor 基类做覆盖；更具体的类路径更容易控制规模。没有类配置时，Actor 可从 NetUpdateFrequency 得到频率；net.Iris.EnableDynamicNetUpdateFrequency=true 时运行期变化也会更新 Iris。

SetPollWithObject 让根对象与子对象共享 counter/period，保证它们在同一帧 Poll。自动化测试同时覆盖从根对象和从子对象触发 ForceNetUpdate 的行为。

🎛️ 11.6 ChangeMask：从对象候选收窄到成员变化

ChangeMask 常被拿来泛指“脏位”，但在 Iris 里至少要分三层。下面三张卡片按产生和消费顺序排列：

LAYER 01 · OBJECTDirtyNetObjectTracker bits

以 InternalIndex 为 bit，选择需要进入或绕过轮询调度的对象。

LAYER 02 · POLLMember Poll Mask

Global Tracker 把 RepIndex 映射到 Fragment 内成员，限制 Push Poll 读取范围。

LAYER 03 · QUANTIZEDProtocol ChangeMask

读取、比较和量化确认后的真实成员变化，供传播与序列化使用。

成员怎样映射到 ChangeMask bits

FReplicationStateMemberChangeMaskDescriptor 为每个成员保存 BitOffset 与 BitCount。绝大多数成员占 1 bit，但动态数组等成员可分配多个 bit。一个 Protocol 中多个 ReplicationState 的 ChangeMask 按顺序扁平拼接，量化阶段写入 FChangeMaskCache。

小 ChangeMask 为什么不用单独分配

FChangeMaskStorageOrPointer 内部只有一个 64-bit 字段。BitCount <= 64 时，它把这个字段直接当作内联位存储；超过 64 bits 才让 allocator 分配外部 storage，并把字段解释为指针。这是一种典型的 small-buffer optimization，减少常见小对象的堆分配。

BitCount存储特点
1–64内联在 uint64 ChangeMaskOrPointer无需额外分配；访问时返回字段地址。
> 64外部分配，字段保存指针大小按 FNetBitArrayView 所需 storage words 计算。

教学模型中的 86.9% 从哪里来

假设一个状态有 10 个互相独立的 32-bit 成员，每次只变化 1 个。忽略一切外层协议和具体 Serializer：

仅用于说明“成员选择”的简化模型 full members = 10 × 32 = 320 bits
raw mask + one member = 10 + 32 = 42 bits
illustrative reduction = 1 − 42 / 320 = 86.9%

真实 Iris 网络侧会对 ChangeMask 使用 sparse bit array 编码，并叠加对象句柄、batch、Baseline/条件、DataStream、packet header 以及 NetSerializer 的量化位数。因此 86.9% 不是包体承诺；它只说明“先选成员”为什么可能降低后续工作量。

ChangeMask 在量化阶段才变得可信

QuantizeObjectStateData 只遍历 DirtyObjectsToQuantize，为对象在 Cache 中预留 ChangeMask 存储，再调用 Quantize 或 QuantizeIfDirty。只有比较和量化确认变化后，bHasDirtyChangeMask 才为真。也就是说，Push 的“可能变化”与 ChangeMask 的“量化后确实变化”之间还隔着 Poll/Copy 和 Serializer 的 equality 语义。

如果漏掉 Health 的标脏宏

在 fully-push Fragment 中，Health 可能不会进入 Member Poll Mask；对象若也没有其他 Dirty、Force、GC 或 WantsFullPoll 原因，甚至会被 clean fully-push 过滤直接跳过。值留在服务器对象里，复制缓冲区和客户端都看不到变化。

如果调用 ForceNetUpdate

Tracker 同时设置 Force 与 Dirty。频率限制器把它纳入 override，Poller 在本帧读取并产生待量化状态。之后对象仍需经过过滤和发送预算。若本帧 Poll 成功，ReconcilePolledList 清除其 Force 和累计 Dirty；若没被 Poll，对应累计状态保留到后续帧。

如果 PreUpdate 中又标脏另一个对象

5.7.4 的自动化测试专门覆盖 PreUpdate 中对自己或另一个对象 MarkDirty/Force 的情况。由于 FinalizeDirtyObjects 在 PreUpdate 后再次收集并锁定，标记不会凭空消失；但对象在当前 PollList 中的位置和本帧是否来得及被 Poll，仍受调度时序影响。测试展示某些“轮询较晚对象”可在同帧被捕获，而已错过轮询位置的对象可能在下一次更新处理。

什么时候 Push 可能不划算

  • 属性几乎每帧变化，通知成本并不能换来大量 clean skip；
  • 写入口散落在大量代码、蓝图和第三方模块，漏标风险高；
  • 对象多数 Fragment 仍是 Poll，fully-push 过滤收益有限；
  • Setter 为复杂结构体做昂贵比较，成本转移到了 Gameplay；
  • 对象数量很少，优化收益低于维护和测试成本。

推荐实验矩阵

  1. 固定服务器 TickRate、对象数和连接数，录制可重复 Gameplay 回放；
  2. 基线：关闭 Push，保留项目当前 PollFrequency；
  3. 只启用对象级 Push，记录 clean skip 与漏标验证；
  4. 启用 per-property tracking，观察 PollDirtyState 比例和成员数；
  5. 分别调整 PollFrequency 与 net.Iris.EnableForceNetUpdate 策略；
  6. 用 Networking Insights/NetTrace 把发现侧 CPU 与最终网络结果对齐。

✅ 接入清单与工程最佳实践

Push 属性接入清单

  • 确认目标构建包含 WITH_PUSH_MODEL，项目配置启用 net.IsPushModelEnabled。
  • 确认 net.Iris.PushModelMode 符合项目是否需要运行时切换。
  • 为属性设置 FDoRepLifetimeParams::bIsPushBased = true。
  • 把所有写入口收口到 Setter，并先比较、再赋值、最后标脏。
  • 普通成员优先用 MARK_PROPERTY_DIRTY_FROM_NAME。
  • 静态数组使用 index/whole-array 专用宏，不把一个元素误当普通成员。
  • 为漏标、重复赋相同值、Dormancy、子对象、Owner 条件和 GC 场景写自动化测试。
  • 先在少数高价值类启用并采样，再扩大范围。

轮询频率配置清单

  • 把 PollFrequency 理解为检查频率，不是发送频率或投递保证。
  • 优先按具体类配置，不对 Object/Actor 根类做宽泛覆盖。
  • 根对象与子对象需要一致快照时使用共享轮询帧。
  • 动态修改 Actor NetUpdateFrequency 前确认 Iris 动态频率开关。
  • 明确普通 Dirty 是否应绕过频率；再配置 net.Iris.EnableForceNetUpdate。
  • 对低频对象的关键事件显式调用 Force，但不要假设其必定本帧发出。
  • 查看错峰后的 P99，而不只看平均每帧 Poll 数。

代码审查的三个问题

  1. 通知是否完整？每个 Push 属性的所有写入口是否都会调用正确的宏？
  2. 调度是否符合玩法？普通 Dirty、Force 与 PollFrequency 的组合是否满足关键状态延迟预算？
  3. 收益是否真的到达下游？Poll CPU 降低后，ChangeMask、逐连接过滤和最终带宽是否也改善？

版本升级清单

升级 UE 后重新核对：Push mode 与 CVar 默认值、Global Tracker 的 owner 限制、Poller 过滤条件、任务分片默认值、频率计数位宽与 SIMD 路径、ChangeMask 内联阈值，以及本文列出的自动化测试。Private 实现是理解工具，不是版本承诺。

💡 11.7 总结与下一部分：从“谁脏了”到“它引用谁”

  • Poll 主动检查，Push 主动通知；Iris 允许 fully-push、pure-poll 和 hybrid 对象共存。
  • FDirtyNetObjectTracker 用本帧 Dirty、跨帧 Accumulated 与 Force 三组位图避免丢失和语义混淆。
  • 属性级 Push 通过 Legacy 适配层进入 Global Tracker，并可按 RepIndex 生成 Member Poll Mask。
  • Push 不是零轮询：Poller 仍负责读取、比较、GC 引用刷新和 hybrid Fragment。
  • ForceNetUpdate 可越过轮询频率门控，但不保证绕过连接级筛选和本帧预算。
  • Frequency Limiter 用 uint8 counter、错峰 offset 与 SIMD 把大量对象的 Poll 成本摊开。
  • 对象脏位、Member Poll Mask 与量化 ChangeMask 是三层不同数据；只有最后一层确认真实成员变化。
  • 小 ChangeMask 在 64 bits 内内联；网络侧使用 sparse mask，教学位数不能当作最终包大小。
  • Push 收益没有通用倍数，必须同时测候选规模、Poll CPU、ChangeMask 与最终逐连接网络结果。

第十一部分回答了“当前状态中的谁值得检查、谁确实变化”。下一部分将进入对象引用与依赖：当一个复制成员指向另一个 UObject 时，Iris 如何用 NetObjectReference 表达身份，怎样处理未解析引用、必须先存在的依赖，以及 owner、subobject 与 dependent object 如何共同影响发送顺序。
