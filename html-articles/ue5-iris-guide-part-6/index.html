🎯 第六部分：优先级系统

过滤系统决定“哪些包裹可以进入这条连接的仓库”，优先级系统决定“仓库里哪些包裹更值得先装车”；真正能否装进本帧数据包，最后仍由 Replication Writer 按状态成本和位预算裁决。

在一局百人战术竞技游戏里，即使空间过滤、队伍规则和 Owner Filtering 已把全服数万个对象压缩到某条连接可见的 800 个对象，这 800 个对象仍不可能每个网络 Tick 都完整发送。近处正在开火的敌人、自己的载具、远处的空投、比分板和地面物资，重要性并不相同；而且“这帧不重要”也不能等于“永远饿死”。

Iris 用一个很克制的模型解决这个问题：优先级器为候选对象计算一个 本 Tick 的优先级贡献值，Writer 把贡献累加到连接自己的调度优先级；达到阈值的脏对象进入调度候选，再按累计值排序，并在真实位预算下逐个写入。对象完整发送并变干净后，累计值才归零。

先记住三层边界

Filtering 决定对象是否在连接作用域；Prioritization 计算本 Tick 应贡献多少优先级；Replication Writer 才负责阈值、累计、排序、依赖对象和位预算。把三者混成一个“限流器”，是配置 Iris 时最常见的认知错误。

本文以 Unreal Engine 5.5 官方文档和本机安装版引擎源码为基准。Iris 位于 Experimental 模块，不同小版本的默认值、配置定义与内部批处理细节可能变化；项目升级时请重新核对目标版本。

🧭 6.1 优先级系统概述

优先级位于哪一段流水线

服务器网络对象
    ↓
过滤：得到该连接的 ObjectsInScope
    ↓
轮询与脏状态：找出需要发送或重发的对象
    ↓
优先级器：计算本 Tick 的 priority contribution
    ↓
Writer：累加 → 阈值判断 → 降序排序
    ↓
Writer：按状态成本与位预算写包
    ↓
完整发送并变干净：累计调度优先级归零

优先级不会让一个已被过滤掉的对象重新进入作用域，也不会直接承诺“本帧一定发送”。它只影响已经在作用域内、并且有更新或需要重发的对象之间的竞争顺序。

静态优先级与动态优先级

类型计算方式适合对象典型 API
静态优先级每个 Tick 使用固定贡献值比赛阶段、规则对象、低频全局状态SetStaticPriority(Handle, Priority)
动态优先级由 Prioritizer 根据连接、视图、位置或拥有关系批量计算玩家、载具、投射物、空间物品SetPrioritizer(Handle, PrioritizerHandle)

没有显式设置静态优先级或 Prioritizer 的对象，默认静态优先级是 1.0。而带有效世界位置标签的对象，可以由默认空间 Prioritizer 自动接管，前提是默认定义存在且对象能提供位置。

数值不是“概率”，而是调度贡献

官方语义以 1.0 为关键门槛：累计调度优先级低于 1 的对象，本 Tick 不进入普通调度候选；达到或超过 1，且对象有脏状态，才会被考虑。以一个持续脏的远距离对象为例：

Tick本 Tick 贡献累计调度优先级结果
10.250.25未过阈值
20.250.50未过阈值
30.250.75未过阈值
40.251.00进入 Writer 候选
50.251.25若上帧没装下，竞争力继续上升

因此 0.25 更接近“理想情况下约每 4 个网络 Tick 获得一次发送资格”，而不是 25% 的随机概率。它能自然缓解饥饿：只要对象持续需要发送，没被写入的时间越久，累计值越高。

何时计算，何时重置

FReplicationPrioritization::Prioritize() 会在发送更新流程中处理新建/删除对象、通知脏对象、调用 Prioritizer 的前置阶段，再逐连接计算动态优先级。随后 FReplicationWriter::UpdatePriorities() 将结果加到该连接的 SchedulingPriorities。

重置发生在 Writer 侧：对象状态完整写出并已干净时，调度优先级清零。若受位预算、附件、子对象或仍有脏状态影响而未完整完成，它不会简单地因为“被考虑过”就归零。丢失状态和新建对象也有各自的优先级增益路径，所以调试时必须观察 Writer 的累计值，而不能只截取 Prioritizer 单帧输出。

🧱 6.2 UNetObjectPrioritizer

基类职责与生命周期

UNetObjectPrioritizer 是动态优先级器的基类。它围绕 Iris 内部对象索引和批量参数工作，而不是每条连接遍历所有 UObject。主要回调可按生命周期理解：

回调发生时机实现关注点
Init / Deinit优先级器创建与销毁读取配置、缓存系统指针、分配数组
AddConnection / RemoveConnection连接进入或离开维护每连接状态
AddObject / RemoveObject对象绑定或解绑该优先级器申请/释放对象私有信息；可拒绝不支持的对象
UpdateObjects相关对象变脏后批量更新把 UObject/复制状态转换成紧凑缓存
PrePrioritize本轮所有连接计算前恰好一次构建跨连接共享的临时数据
Prioritize按连接、按批次调用只修改输出 Priority，并保留更高已有值
PostPrioritize所有连接计算完成后一次清理本轮临时状态

8 字节的 FNetObjectPrioritizationInfo

每个动态优先级器可以为对象保存一份紧凑私有信息。UE 5.5 中 FNetObjectPrioritizationInfo 按 8 字节对齐，内部是四个 uint16，总计正好 8 字节。内置实现会在此基础上解释字段，例如位置型优先级器保存状态偏移、状态索引和已分配的位置索引。

// UE 5.5 结构摘录（为聚焦语义省略宏和命名空间）
struct alignas(8) FNetObjectPrioritizationInfo
{
    uint16 Data[4];
};

这个设计提醒自定义实现：对象级热数据应该紧凑、可批处理。复杂业务对象、字符串或稀疏映射不应直接塞进逐对象热路径；更大的数据可以放在 Prioritizer 自己管理的 chunked array 中，再让这 8 字节保存索引。

Prioritize 参数里有什么

字段含义
ObjectIndices / ObjectCount分配给当前 Prioritizer，且本轮需要优先级更新的对象
Priorities按 Iris 内部索引访问的浮点优先级数组；这是回调允许修改的输出
PrioritizationInfos每对象 8 字节私有信息
ConnectionId当前正在计算的连接
View该父连接及其子连接的一个或多个复制视图

接口契约要求实现不能无条件覆盖已有值，而应保留两者中的较大值。可把核心规则理解为：

// 解释性伪代码，不是引擎逐字源码
for (ObjectIndex : Params.ObjectIndices)
{
    const float Calculated = CalculateForConnection(ObjectIndex, Params.View);
    Params.Priorities[ObjectIndex] =
        Max(Params.Priorities[ObjectIndex], Calculated);
}

注意：UE 5.5 的 FReplicationPrioritization 为每个对象保存一个动态 Prioritizer 索引，公共设置路径并不是让一个对象同时挂多个独立 Prioritizer。这里的“保留较大值”是回调契约，可保护默认值、先前贡献或实现内部组合结果，不能据此推导出任意多 Prioritizer 自动叠加。

与过滤系统如何协作

Prioritizer 看到的不是全服对象集合，而是 Writer 判定需要优先级更新的候选。过滤系统先决定对象能否进入某连接作用域；拥有关系也可由 FReplicationFiltering 提供给 Owner Boost 和 Count Limiter 更新缓存。两者会共享连接和对象事实，但职责仍然不同。

Owner Boost ≠ Owner Filtering

Owner Filtering 是可见性规则：非拥有连接根本不接收对象。Owner Boost 是排序偏好：其他通过过滤的连接仍可接收，只是拥有者连接的优先级更高。隐私、权限和反作弊规则必须由过滤或业务授权保证，不能依赖 Boost。

🧰 6.3 内置优先级器

LocationBased：所有空间实现的共同底座

ULocationBasedNetObjectPrioritizer 负责把网络对象位置转换为适合批量计算的缓存。它优先从 Iris 的 FWorldLocations 取得位置；若没有，则尝试从带 RepTag_WorldLocation 的复制状态成员读取。两者都不存在时，AddObject 会失败，调用方最终可能回落到静态优先级。

位置缓存在 TChunkedArray<VectorRegister> 中，以位数组跟踪已使用槽位。对象移除时释放位置索引；对象更新时从 WorldLocations 或外部状态 Fragment 刷新缓存。这样 Sphere 和 FOV 的逐连接热循环可以直接读取连续、向量化友好的位置数据。

Sphere：距离越近，贡献越高

USphereNetObjectPrioritizer 定义内球、外球和球外三段：

  • 内球以内使用 InnerPriority；
  • 内外半径之间按距离从 InnerPriority 线性下降到 OuterPriority；
  • 外球以外使用 OutsidePriority。
// 概念模型
distance <= InnerRadius:
    priority = InnerPriority

InnerRadius < distance < OuterRadius:
    priority = Lerp(InnerPriority, OuterPriority, normalizedDistance)

distance >= OuterRadius:
    priority = OutsidePriority

UE 5.5 本机头文件的构造默认值是：内半径 2000、外半径 10000、内优先级 1.0、外优先级 0.2、球外优先级 0.1。Epic 的 UE 5.5 优先级文档页面示例曾展示 1000/5000 的半径组合，因此生产项目应在自己的配置中显式写出参数，不要依赖记忆中的“默认值”。

实现以最多 1024 个对象为一批，按四个对象做 VectorRegister/SIMD 计算，并分别优化单视图、双视图和多视图。双/多视图取对象到所有 View 中的最近距离；多于 8 个 View 时，内部视图位置临时数组会触发额外分配警告。

SphereWithOwnerBoost：空间结果上再加拥有者奖励

USphereWithOwnerBoostNetObjectPrioritizer 先执行 Sphere，再对当前连接拥有的对象加上 OwnerPriorityBoost。UE 5.5 默认 Boost 为 2.0，采用加法而不是乘法：

// 解释性模型
Priority = SpherePriority;
if (OwningConnection == Params.ConnectionId)
{
    Priority += OwnerPriorityBoost;
}

它通过过滤系统获取对象的 Owning Connection，并把连接 ID 缓存在分块数组中。适合自己的 Pawn、载具或关键交互对象需要比同距离其他对象更及时的场景，但依然不改变其他连接的可见性。

FieldOfView：视线、锥体和近身保障取最大值

UFieldOfViewNetObjectPrioritizer 同时计算四类几何贡献，最后取最大值：

区域作用UE 5.5 默认要点
内球近身对象不因玩家背对而降级半径 3000，优先级 1.0
外球给较远的四周对象一个保底半径 10000，优先级 0.2
视野锥视线方向内对象按距离衰减45°，内锥长 3000，总长 20000，1.0 → 0.2
视线胶囊瞄准方向上的窄通道保持高优先级宽 200，优先级 1.0
区域外最低保底贡献0.1

一个容易误读的细节是：FReplicationView::FView 确实携带 FoVRadians，但 UE 5.5 这个内置 FOV Prioritizer 构造锥体时使用配置项 ConeFieldOfViewDegrees，并非按每个 View 的 FoVRadians 动态改变锥角。View 的方向和位置仍参与计算。

FOV 同样按四对象向量化和最多 1024 个对象分批，但方向相关几何使它对多 View 逐个循环，而不像 Sphere 那样有独立双视图最近距离快路径。

NetObjectCountLimiter：控制“本轮给谁贡献”，不是连接带宽上限

UNetObjectCountLimiter 适合大量同类、低频但需要轮流更新的对象，例如 PlayerState、远端排行榜条目或低重要度全局信息。它有两种模式：

模式选择方式优势代价/陷阱
RoundRobin每次预选全局下一批 N 个对象成本稳定、天然轮转若被选对象不脏，即使其他对象脏，本轮也可能一个都不贡献
Fill逐连接从脏对象中选择最久未被考虑的 N 个尽量填满配额，更关注当前脏集合排序成本更高；若一个连接拆成多批，需增大批次或改 RoundRobin

MaxObjectCount 表示这个 Prioritizer 每轮选择多少对象设置 Priority；拥有对象可进入 Fast Lane，不计入 MaxObjectCount，并使用 OwningConnectionPriority。它不保证恰好发送 N 个对象，更不限制其他 Prioritizer 或静态对象。被选对象仍要跨过 Writer 阈值、参与排序并消耗真实位预算。

UE 5.5 中 Mode 不是 UPROPERTY(Config)，不能指望仅靠普通 ini 覆盖切换；内置 UPlayerStateCountLimiterConfig 在 C++ 构造函数里把它设为 Fill，同时配置 MaxObjectCount=2、Priority=1、OwningConnectionPriority=2 并启用 Fast Lane。自定义模式时，最稳妥的做法是派生配置类并在构造函数中设置。

👁️ 6.4 ReplicationView

FView 的五个事实

// UE 5.5 结构含义摘录
struct FView
{
    FNetHandle Controller;   // 通常是 PlayerController
    FNetHandle ViewTarget;   // 通常是正在观察的 Pawn/Actor
    FVector Pos;             // 观察位置
    FVector Dir;             // 观察方向
    float FoVRadians;        // 默认 UE_HALF_PI
};

FReplicationView 内部是 View 数组，默认内联容量由 UE_IRIS_INLINE_VIEWS_PER_CONNECTION 控制，UE 5.5 默认为 4。空间过滤和空间优先级都可消费它，因此错误的 View 不只影响“先发谁”，也可能影响“谁在作用域”。

NetDriver 如何填充 View

在 Iris 发送更新前，UNetDriver::UpdateIrisReplicationViews() 会更新连接视图，然后调用 UReplicationSystem::SetReplicationView()。服务端对每个主连接组合主连接和所有 Child Connection：

  1. 若在服务端，先通过 GetConnectionViewTarget() 刷新连接 ViewTarget；
  2. 若有 ViewTarget，先用它的 Actor 位置、旋转和 NetHandle；
  3. 若有 PlayerController，再以 GetPlayerViewPoint() 覆盖 Pos/Dir，并写 Controller Handle；
  4. 若存在 PlayerCameraManager，用当前 FOV Angle 写入 FoVRadians；
  5. 主连接和所有子连接的 FView 一起存到父 ConnectionId。

这解释了为什么第三人称相机、观战、切换 ViewTarget 或分屏问题，常常表现为“空间优先级不对”。服务器最终使用的是连接维护的观察信息，而不是简单读取 Pawn 根组件。

多视图语义：对任一观察者重要就不能忽略

分屏时一条父连接可能代表多个玩家。Sphere 会取对象到所有 View 的最小距离，所以对象只要靠近其中任一视图，就得到较高空间优先级。FOV 则逐 View 计算锥体、视线胶囊与球体，再保留最大贡献。

性能边界

默认内联 View 容量是 4；Sphere 的多视图临时快路径内联 8 个位置。普通游戏通常远低于此数，但自定义多摄像机、服务器观察器或特殊分屏方案应测量 View 数量与热路径分配。

⚙️ 6.5 配置与绑定

先注册定义，再按类或运行时绑定

Prioritizer 定义把“名字、实现类、配置类”关联起来。下面是一个命名一致的项目示例；它展示结构，不要求照搬数值：

; DefaultEngine.ini — 配置示例
[/Script/IrisCore.NetObjectPrioritizerDefinitions]
+NetObjectPrioritizerDefinitions=(PrioritizerName=BattleRoyaleSpatial,ClassName=/Script/IrisCore.SphereWithOwnerBoostNetObjectPrioritizer,ConfigClassName=/Script/IrisCore.SphereWithOwnerBoostNetObjectPrioritizerConfig)
+NetObjectPrioritizerDefinitions=(PrioritizerName=AimAwareSpatial,ClassName=/Script/IrisCore.FieldOfViewNetObjectPrioritizer,ConfigClassName=/Script/IrisCore.FieldOfViewNetObjectPrioritizerConfig)
+NetObjectPrioritizerDefinitions=(PrioritizerName=FarInfoLimiter,ClassName=/Script/IrisCore.NetObjectCountLimiter,ConfigClassName=/Script/MyGame.FarInfoLimiterConfig)

[/Script/IrisCore.SphereWithOwnerBoostNetObjectPrioritizerConfig]
InnerRadius=2500.0
OuterRadius=18000.0
InnerPriority=1.0
OuterPriority=0.25
OutsidePriority=0.1
OwnerPriorityBoost=2.0

[/Script/IrisCore.ObjectReplicationBridgeConfig]
+PrioritizerConfigs=(ClassName=/Script/MyGame.BRCharacter,PrioritizerName=BattleRoyaleSpatial,bForceEnableOnAllInstances=true)
+PrioritizerConfigs=(ClassName=/Script/MyGame.Projectile,PrioritizerName=AimAwareSpatial,bForceEnableOnAllInstances=true)
+PrioritizerConfigs=(ClassName=/Script/MyGame.FarInfo,PrioritizerName=FarInfoLimiter,bForceEnableOnAllInstances=true)

Count Limiter 的自定义 FarInfoLimiterConfig 应在 C++ 构造函数中明确设置 RoundRobin 或 Fill，因为 UE 5.5 的 Mode 不是 Config 属性。配置类可以继续通过 ini 调整 MaxObjectCount、Priority、OwningConnectionPriority 和 Fast Lane。

运行时 API：静态值和动态实现互斥切换

// C++ 用法示意；Handle 获取方式取决于你的 Bridge/对象注册路径
UReplicationSystem* ReplicationSystem = ...;
FNetRefHandle ObjectHandle = ...;

// 低频全局对象：每 5 个网络 Tick 左右获得一次普通资格
ReplicationSystem->SetStaticPriority(ObjectHandle, 0.2f);

// 切换为空间动态优先级
const FNetObjectPrioritizerHandle SpatialHandle =
    ReplicationSystem->GetPrioritizerHandle(TEXT("BattleRoyaleSpatial"));

if (SpatialHandle != InvalidNetObjectPrioritizerHandle)
{
    const bool bApplied =
        ReplicationSystem->SetPrioritizer(ObjectHandle, SpatialHandle);
}

SetStaticPriority 会移除对象原来的动态 Prioritizer；SetPrioritizer 会把默认静态值设为 0、注销旧实现，再调用新实现的 AddObject。若新实现拒绝对象，例如空间 Prioritizer 找不到位置，源码会记录日志并回退到可用的静态策略。调用方仍应检查返回值。

默认定义与类配置的版本陷阱

UE 5.5 安装版 BaseEngine.ini 中，Prioritizer 定义使用过 PlayerStatePrioritizer，而 ObjectReplicationBridge 的类映射和配置类注释却引用 PlayerState。因为 GetPrioritizerHandle() 是按名字线性精确匹配，项目不应盲目复制这组不一致片段。

稳妥做法是：在项目最终合并配置中用同一个名字注册和引用，启动时检查无效 Handle/Verbose 日志，并在引擎升级后对比目标版本 BaseEngine.ini。这里讨论的是 UE 5.5 安装版配置现象，不代表所有 5.5 分支或后续版本都相同。

DefaultPrioritizer 的特殊位置

第一个有效定义是默认空间 Prioritizer，Handle 0 具有特殊含义。源码也识别名字 DefaultPrioritizer。对象带有效 RepTag_WorldLocation 时可以自动应用默认实现；若项目重排定义、删除默认项或让默认实现无法读取对象位置，行为会随之变化。

🔬 6.6 内部实现

FReplicationPrioritization 持有什么

UE 5.5 的核心管理器维护：

  • 每个对象一个 uint8 Prioritizer 索引，最多容纳 256 个定义；
  • 每对象默认/静态优先级数组；
  • 每连接的当前动态优先级数组；
  • 新静态优先级、创建/删除对象和活跃 Prioritizer 的位集合；
  • 已实例化的 Prioritizer 名称、对象和配置。

对象位置和拥有者等扩展数据则由各 Prioritizer 以 TChunkedArray 管理。分块数组能按需增长，避免按“最大对象数 × 大结构”一次性分配，同时保持索引访问和批处理友好。

Prioritize 的五段主流程

// 根据 UE 5.5 源码整理的流程，不是逐字复制
1. UpdatePrioritiesForNewAndDeletedObjects()
2. NotifyPrioritizersOfDirtyObjects()
3. 对活跃 Prioritizer 调用 PrePrioritize()
4. 逐连接：
   a. 取得 Writer.GetObjectsRequiringPriorityUpdate()
   b. PrioritizeForConnection()
   c. Writer.UpdatePriorities()
5. 对活跃 Prioritizer 调用 PostPrioritize()

若当前没有待发送连接，流程会在通知脏对象后提前结束。若连接没有任何 View，逐连接计算也会跳过。该事实可用于排查“优先级器完全没被调用”的问题。

脏对象更新与优先级计算是两种批处理

FUpdateDirtyObjectsBatchHelper 按 Prioritizer 分组脏对象和 Instance Protocol，每批最多 512 个，调用 UpdateObjects；FPrioritizerBatchHelper 则按 Prioritizer 收集当前连接候选，每批最多 1024 个，再调用 Prioritize。

这种“先按实现分桶，再批量调用”的结构减少虚函数切换和缓存抖动。自定义实现应顺着这个模型写连续循环，不要在每个对象上反查 UObject、做昂贵射线或临时分配。

PrioritizeForConnection 做了什么

管理器为当前连接组装 FNetObjectPrioritizationParams，把候选按绑定的 Prioritizer 放入 batch。对于动态对象，进入计算前会以对象默认值重置该连接本 Tick 的输出，然后由实现写出更高结果。最后可选地把 Controller 和 ViewTarget 提升到 1.0e7 的高优先级。

这个 View Target 提升由控制变量路径启用，目的是让连接自己的观察者与目标极难在排序中落后。它是内部保护机制，不应拿来替代正常的 Owner 配置或业务优先级。

真正的累加、门槛和排序在 Writer

FReplicationWriter::UpdatePriorities() 对脏对象执行累加，而非覆盖：

// 解释性伪代码
SchedulingPriority[Object] +=
    PrioritizerOutputForThisTick[Object];

if (ObjectIsDirty && SchedulingPriority[Object] >= 1.0f)
{
    AddToScheduleCandidates(Object);
}

SortCandidatesByDescendingSchedulingPriority();
WriteUntilBitBudgetOrOtherConstraintsStopUs();

UE 5.5 Writer 会对候选做降序排序，并用一个 128 对象的局部排序优化开头最重要的部分；最终写入仍考虑状态大小、依赖、子对象、附件、协议和当前位预算。因此，两个优先级同为 2.0 的对象，若状态成本相差十倍，实际“能装下几个”也会不同。

Count Limiter 与 Writer 调度必须分开看

Count Limiter 的 MaxObjectCount
    = 本轮由这个 Prioritizer 选中并赋予贡献的对象数

Writer 的发送数量
    = 所有来源候选经过累计、排序、依赖和位预算后的结果

如果 FarInfoLimiter 的 MaxObjectCount=4，它只表示每轮最多四个普通 FarInfo 获得该 Prioritizer 的 Priority；拥有者 Fast Lane 还可额外加入。它既不会拦住 100 个空间对象，也不保证这四个 FarInfo 最终都写入。

饥饿避免并不等于无条件公平

持续脏、持续获得正贡献的对象会累加并逐渐超过新来的高频对象，这是基础的抗饥饿机制。但若配置为 0、对象始终不进入 Prioritizer 候选、被过滤、没有 View，或某类关键对象每 Tick 贡献远高于所有其他对象且长期耗尽预算，低优先级对象仍可能延迟很久。公平性需要靠合理的贡献下限、分层过滤、Count Limiter 和真实压测共同保证。

🎮 6.7 完整实战：800 个对象如何争夺一帧预算

场景与教学假设

设一条战术竞技连接通过第五部分的过滤系统后，当前作用域中有 800 个对象。为让数字可手算，我们假设网络拥塞时本 Tick 的位预算约等于 120 个“平均成本对象”。这不是 Iris 的固定对象上限：真实 Writer 使用 bit budget 和每个复制状态的实际成本，大对象可能一个顶多个，小对象也可能多装几个。

分组数量策略本 Tick 典型贡献业务意图
A：自己/交战核心24SphereWithOwnerBoost 或 FOV 高区2.0开火、受击、自己的 Pawn/载具优先
B：近距离角色与载具96Sphere 内外圈插值0.55约两 Tick 获得资格
C：中距离活动对象180Sphere 外圈附近0.25约四 Tick 获得资格
D：远距离世界对象420OutsidePriority0.10持续脏时约十 Tick 获得资格
E：低频信息对象80Count Limiter Fill每轮选 4 个并赋 1.0轮流更新，避免 80 个一起竞争

为了聚焦优先级，先假设这 800 个对象都在作用域、都有需要发送的状态，而且每组对象状态成本近似。真实项目要把脏率、状态大小、新建对象和依赖关系加入回放。

单个对象如何跨帧累加

代表对象F1F2F3F4F5资格与重置
A 核心对象（2.0）2.02.02.02.02.0若每帧都脏且完整发出，每帧重置后再次达到 2.0
B 近距对象（0.55）0.551.10发出后 0.551.10发出后 0.55理想情况下约每两 Tick
C 中距对象（0.25）0.250.500.751.00若没装下则 1.25落选后继续升高，不会回到 0
D 远距对象（0.10）0.100.200.300.400.50第十 Tick 才首次达到 1.0

表里的“发出后”假设对象已完整变干净；若状态只写了一部分或继续变脏，Writer 的实际累计轨迹会不同。关键结论是：Prioritizer 每 Tick 给出贡献，Writer 保存跨 Tick 历史。

前五帧的候选爆发与预算竞争

帧达到 1.0 的主要群体候选规模（近似）教学预算结果
F1A 24 + E 428预算充足，通常全部写入
F2A 24 + B 96 + E 4124约 120 个写入，少量 B/E 留待后续并继续累加
F3A、F2 遗留对象、E 新一轮 4 个约 32遗留 B 已达约 1.65，通常先于刚过阈值对象
F4A + 大部分 B + C 180 + E约 300A 以 2.0 居前；B 约 1.1；大量 C 首次达 1.0，只有一部分装下
F5A + F4 未发送的 C + E超过预算未发送 C 升到 1.25，开始压过新近达到 1.0 的对象

这就是累加抗饥饿的直观效果：F4 落选的中距离对象不会每帧重新以 1.0 排在末尾，而会在 F5、F6 继续增长，逐渐获得靠前位置。与此同时，A 组核心对象用 2.0 保证即时性。两者不是绝对公平，而是“即时性”和“等待时间”共同参与排序。

第十帧的远距离同步波峰

如果 420 个远距离对象同时持续脏、并且贡献完全相同，它们会在 F10 一起达到 1.0，形成同步波峰。Writer 能挡住超预算写入，却不能消除这一大批候选的排序成本与延迟尾部。

可采用的工程手段包括：

  • 先确认它们真的需要持续脏；能事件驱动或降低轮询频率的，不要让状态每 Tick 变化；
  • 按对象类型设置不同 OutsidePriority，让首次跨阈值分散；
  • 对同质低频集合使用 Count Limiter，让每轮只有少量对象获得 1.0 贡献；
  • 在自定义 Prioritizer 中引入稳定、可复现的小幅分桶偏移，而不是每帧随机抖动；
  • 若对象对该连接没有价值，应回到过滤层移出作用域，而不是把优先级压到接近零长期占用集合。

四类游戏如何映射同一机制

类型高优先级信号低频集合推荐组合
战术竞技自己的 Pawn/载具、视线内敌人、近处枪火远处物资、非交战玩家信息FOV/SphereWithOwnerBoost + Count Limiter
FPS准星方向、近身侧后方、投射物远端环境交互FOV 的锥体 + LoS 胶囊 + 近身球
RPG/MMO自己的角色、队友、当前目标、Boss 技能远处 NPC 与装饰交互Owner Boost + Sphere；队伍可见性仍交给 Group Filter
竞速自己的车、前后相邻车辆、碰撞相关对象赛道远端车辆与计时条目沿赛道自定义距离 Prioritizer + Count Limiter

竞速游戏尤其能说明“欧氏距离不等于业务距离”：隔着立交桥很近的车辆可能毫不相关，而赛道前方几百米的车辆更重要。此时自定义 Prioritizer 应缓存赛道进度或拓扑距离，在批量回调里计算；过滤仍负责彻底排除不可能交互的赛段。

一套可落地的调参流程

  1. 记录输入。按连接统计 ObjectsInScope、脏对象数、各 Prioritizer 候选数和 View 数量。
  2. 记录贡献。采样每类对象本 Tick 输出和 Writer 累计值分布，而不是只看配置值。
  3. 记录成本。按对象类型统计实际写入 bits、完整/部分写入和等待 Tick 数。
  4. 先保护核心。让交战对象在一 Tick 内达到阈值，但避免无意义地把大批对象都设成 10 或 100。
  5. 再控制尾部。检查 P95/P99 等待 Tick、同步波峰和持续未发送对象，调整保底贡献或 Count Limiter。
  6. 用真实回放验证。模拟百人密度、载具高速移动、观战、分屏和丢包，而非只在空地图测平均带宽。

✅ 6.8 本章小结

  • 优先级系统不决定可见性，也不直接限定发送对象数；它为 Writer 的连接级调度提供贡献。
  • 静态优先级适合稳定低频对象；动态 Prioritizer 适合基于位置、方向、拥有关系或集合轮转的对象。
  • FNetObjectPrioritizationInfo 只有 8 字节，内置实现通过索引连接更大的分块缓存。
  • Sphere 使用距离分段和线性衰减；Owner Boost 是加法奖励；FOV 组合锥体、视线胶囊和双球体并取最大值。
  • Count Limiter 的 MaxObjectCount 只约束自身本轮选中的对象，不是 Writer 的带宽上限。
  • ReplicationView 由 NetDriver 在发送前更新，主/子连接共享父 ConnectionId 下的多 View。
  • Writer 将本 Tick 贡献累加到 SchedulingPriority，达到 1.0 的脏对象才进入候选；未写入对象继续累加。
  • 真实调度由优先级、对象状态成本、依赖与位预算共同决定，“800 → 120”只能作为等成本教学模型。

推荐固定排查链：对象是否在作用域 → 是否脏/需重发 → 绑定了哪个 Prioritizer → View 与位置输入 → 本 Tick 输出 → Writer 累计值 → 是否达到 1.0 → 排序位置 → 实际状态 bits 与预算。只看抓包结果，无法区分“未获得资格”和“获得资格但装不下”。

⚡ 性能建议

  1. 先过滤，再排序。永久无关对象应离开连接作用域，不能靠 0.01 的优先级长期拖着。
  2. 让热路径只读紧凑缓存。位置、拥有者、赛道进度等在 UpdateObjects 中更新，Prioritize 中批量读取。
  3. 避免每对象射线检测。内置 FOV 的“LineOfSight”是几何胶囊，不是物理遮挡查询；真射线需要更低频、更粗粒度的业务方案。
  4. 减少同相位波峰。大量对象使用完全相同的低贡献会同时跨过阈值；用类型分层或稳定分桶分散。
  5. 同时看对象数与 bits。120 个 20-bit 状态和 120 个 2-KB 状态不是同一预算。
  6. 保护拥有者但不要无限放大。Boost 过高会让长期高成本拥有对象压住其他必要状态。
  7. 测多 View。分屏会扩大高优先级空间，FOV 也会按 View 增加几何计算。
  8. 关注等待分布。平均等待 2 Tick 可能掩盖远端对象 P99 等待数十 Tick。
