std::mutex ABI 不兼容崩溃排查记录

一句话结论： 用 MSVC 14.38+ 编译的 std::mutex 采用”首次加锁时懒初始化”新约定，而玩家机器上的老版 msvcp140.dll（VS2015）仍是”构造时初始化”旧约定，二者对锁对象创建时机的假设相反，老 DLL 解引用了新约定下故意留空的指针，触发 EXCEPTION_ACCESS_VIOLATION 崩溃。

1. 问题现象

游戏客户端在部分玩家机器上，登录后进入网络轮询阶段时稳定崩溃于 PollIO::Poll。

  • 崩溃类型： EXCEPTION_ACCESS_VIOLATION
  • 崩溃位置： mtx_do_lock 内部

2. std::mutex 内部结构

std::mutex 不直接持有操作系统锁，而是内部存了一个指向真正锁对象的指针（指向 ConcRT / SRWLOCK 的内部实现结构 _Mtx_internal_imp_t）。

加锁（lock()）时，运行时函数 mtx_do_lock 的操作：

  1. 读取 [rsi+8] 处的指针
  2. 解引用该指针去操作底层锁对象

若指针为 0，第 2 步即为空指针访问，直接崩溃。

3. 新旧版本行为对比

老版本 — VS2015（msvcp140.dll 14.0）

构造函数不是 constexpr，在运行时构造时立即调用 _Mtx_init_in_situ，完成锁对象初始化。

约定：构造时就建好锁对象，加锁时指针必然有效。

新版本 — VS2022 17.8+ / MSVC 14.38+

为支持 std::mutex 的常量初始化（constexpr / constinit），构造函数改为 constexpr，只把内存清零，不再创建锁对象，改由首次 lock() 时懒初始化。

约定：首次加锁时才建锁对象。懒初始化逻辑在新版 msvcp140.dll 里。

新旧约定对照表

维度 老版本（14.0） 新版本（14.38+）
构造函数类型 运行时构造 constexpr
锁对象创建时机 构造时 首次 lock() 时（懒初始化）
构造后 [+8] 指针 有效地址 0（故意）
懒初始化逻辑 无需要 在新版 DLL 中

4. 崩溃触发路径

实际情况：**用新工具链（14.44）编译，却在玩家机器上加载了老版 msvcp140.dll**。

环节 来源 行为
std::mutex 构造 新编译器内联的 constexpr 构造 [+8] 清零，不创建锁对象，等待懒初始化
首次 lock() 玩家机器上老 msvcp140.dll 无懒初始化逻辑，直接解引用 [+8]=0 → 崩溃

本质：新编译器故意把指针留 0，等运行时来懒初始化；老 DLL 根本不知道有”懒初始化”这回事，拿到 0 指针直接解引用，炸了。

5. 为什么偏偏在 PollIO::Poll 触发

LockerPoll() 返回的是一个**全局 / 静态 std::mutex**。

进程首次执行网络轮询时，调用 std::lock_guard l(LockerPoll())，这是该 mutex 首次 lock() 的时刻。老 DLL 在此踩中空指针，触发崩溃。

这也解释了为何崩溃点稳定出现在登录后建立网络连接、刚进入 Poll 的时间点。

6. 解决方案

✅ 方案一（最终采用）：随包下发新版运行时

使用 -applocaldirectory 随安装包下发 **≥ 14.38 的新版 msvcp140.dll**，让”新代码 + 新 DLL”重新自洽。新 DLL 内含懒初始化逻辑，能正确处理 [+8]=0 的新约定。

优点： 从根本上对齐新约定，彻底修复。

⚠️ 方案二（兜底）：禁用 constexpr 构造宏

编译时加入宏 _DISABLE_CONSTEXPR_MUTEX_CONSTRUCTOR，让新编译器退回老约定——构造时就创建锁对象、填好指针，不再依赖懒初始化。即便配老 DLL 也能兼容。

缺点： 只解决 std::mutex 这一处 ABI 差异，无法覆盖其他潜在 ABI 不兼容问题，不推荐作为长期方案。

7. 总结

问题 说明
崩溃类型 EXCEPTION_ACCESS_VIOLATION，空指针解引用
根因 新编译器（14.38+）与老运行时（14.0）对 std::mutex 初始化时机的 ABI 约定不一致
触发条件 新工具链编译 + 玩家机器加载老版 msvcp140.dll
是否业务代码 bug ❌ 否，纯 ABI 兼容性问题
最终修复 随包下发新版运行时（-applocaldirectory）
