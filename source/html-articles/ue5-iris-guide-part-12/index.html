Chapter 01 · Problem Space

第十二部分开场指针出不了进程：复制的第一步是重建“身份”

服务器上的 UObject* 只是服务器地址空间里的一串位。把它原样写进网络包，客户端既不能解引用，也无法判断它对应场景中的哪个实例。网络复制真正需要传递的是一种双方可以独立解释的身份。

对象引用解决“它是谁”；依赖关系解决“谁应先被调度或创建”；NetToken 解决“重复身份数据如何只导出一次”；NetObjectFactory 解决“远端怎样得到实例”。

这四个问题处于同一条接收链路，却属于不同层次。Iris 把它们拆开，换来更明确的缓存、排序和生命周期边界。代价是：看到一个引用字段时，不能再用“序列化一个地址”来解释，而要追踪 Handle、Path、Token、Creation Header 与队列状态。

IDENTITY

网络可寻址

FNetObjectReference 用 Handle 和可选 PathToken 表达远端能够理解的对象身份。

ORDER

调度与创建顺序

FNetDependencyData 保存松散调度边和更强的创建前置边。

INSTANTIATION

远端落地

UNetObjectFactory 把 Creation Header 解释为查找、绑定、Spawn 或 NewObject。

本篇阅读坐标

我们以“动态角色引用一个静态资产和一个动态装备”为贯穿案例：资产可能需要加载，装备需要在角色存在后创建，重复路径通过 Token 导出，而角色与装备分别由 Actor/SubObject Factory 落地。

Chapter 02 · End-to-end

先看全景一条对象引用从发送端走到接收端

以下流程不是说每个引用都必然触发全部步骤，而是展示最完整的路径。已知引用会命中缓存，已导出的 Token 不必重复携带原始数据，已存在的动态实例也不需要再次创建。

01 / GAMEUObject* 进入量化

NetSerializer/ReferenceCollector 发现属性中的对象引用。

02 / CACHE构造 Reference

查找或创建 RefHandle，必要时生成路径 Token 与 Outer 链。

03 / EXPORT收集待导出身份

先记录 Token export，再记录 object reference export。

04 / WIRE写入包

导出区先写，随后正文状态只引用较紧凑的标识。

05 / REMOTE导入远端映射

Remote Token State 与 ObjectReferenceCache 得到新映射。

06 / RESOLVE解析或排队

已存在则返回 UObject；依赖未就绪则把对象批次挂起。

07 / FACTORY创建并应用状态

Factory 绑定/创建实例，Bridge 应用初始复制状态。

为什么导出必须在正文之前？

正文里的量化值可能只保存一个 Token 或 Handle。如果接收端先读正文，它还没有“索引 → 原始数据/对象身份”的映射。UE 5.7.4 的写端在 WritePendingExports 中先处理 Token exports，再处理对象引用 exports；读端也先建立这些映射，才让状态反序列化使用它们。

“收到身份”和“拿到对象”之间还有距离

静态对象可能尚未加载，动态对象可能尚未收到创建批次，Streaming Level 可能已加载却不可见，Creation Dependency 的 Parent 也可能尚不存在。因此解析结果不只有“成功/失败”，还需要可重试的 Pending和不应继续等待的 Broken。

Chapter 03 · Mental Model

先划边界对象引用、调度依赖、创建依赖、SubObject 所属

工程中最常见的误用，是看到两个对象“有关联”就把四种机制互换。下面这张矩阵应当先于任何 API 记忆。

关系回答的问题发送端作用接收端作用不能替代
对象引用属性里的对象是谁？收集并导出 Handle/Path解析为本地 UObject不保证调度或创建顺序
调度依赖Parent 发送时，Dependent 怎么排？联动优先级、过滤和发送候选按收到的批次正常应用不保证同包，不等于硬前置
创建依赖Child 创建前谁必须存在？写入 Parent Handle缺 Parent 时挂起 Child 批次不表达属性指针值
SubObject 所属谁是 Root/Outer，生命周期归谁？组织 Root 与 SubObject 批次绑定或创建到正确 Root/Outer不等于任意对象依赖图
Property EdgeWeaponAsset 属性指向静态资产FNetObjectReference
Schedule Edge角色更新带动背包更新AddDependentObject
Create Edge武器实例要等角色存在AddCreationDependencyLink
Ownership Edge武器组件属于角色 RootSubObject / Outer
两个重要的“不保证”

AddDependentObject 是 loose dependency，官方文档明确说明不保证数据位于同一个 packet；对象属性持有另一个对象引用，也不会自动建立创建依赖。需要硬创建前置时，应显式使用 AddCreationDependencyLink。

Chapter 04 · 12.1

12.1 ObjectReferenceCache双向缓存、引用形态与 Authority

FNetObjectReference 是“网络可寻址对象”的值类型。UE 5.7.4 中，它可以同时携带：

IDENTITYFNetRefHandle

Authority 分配的对象身份；可区分 Static 与 Dynamic。

PATHFNetToken PathToken

稳定对象的相对或完整路径，需结合相应 Token Store State 解释。

CAPABILITYENetObjectReferenceTraits

例如引用是否可导出。引用只要 Handle 或 PathToken 有效即可被视为有效。

缓存为何需要双向索引

发送端从属性拿到 UObject*，需要快速得到网络引用；接收端拿到 Handle，需要反向找到缓存对象、路径、Outer 与解析状态。因此 FObjectReferenceCache 同时维护 Object→Handle 和 Handle→CachedReference。

ObjectToNetReferenceHandleTMap<const UObject*, FNetRefHandle>

量化和引用收集时从本地实例出发。

⇄
ReferenceHandleToCachedReferenceTMap<FNetRefHandle, FCachedNetObjectReference>

解析时从线上身份回到弱对象、路径、Outer 和状态。

缓存项不是简单的 Handle → UObject*。它还保存弱对象、ObjectKey、RelativePath Token、Outer Handle，以及 bNoLoad、bIsPackage、bIsPending、bIsBroken、bIgnoreWhenMissing、bIsPreRegistered 等状态。

静态对象与动态对象

类型典型对象身份来源缺失时
StaticPackage、稳定命名 Actor、资产、默认 SubObjectStatic Handle + PathToken + Outer 链可查找、加载、Pending 或 Broken
Dynamic运行时 Spawn Actor、动态 SubObjectAuthority 分配的 Dynamic Handle等待复制创建；不能靠路径加载出同一实例

Authority 负责分配权威 Handle。客户端可以使用服务端已经分配并告知的预注册引用，但不能自行伪造一个权威 Handle。对客户端发送给服务端的稳定对象引用，Iris 还可以使用相对 Outer 的路径或完整静态路径表达。

Chapter 05 · Export & Resolve

引用解析先导出身份，再解析；解析不了就让批次等待

发送端：Compact Reference 与 Full Reference

已被对端认识的引用可以只写紧凑 Handle。首次出现、静态路径或尚未导出的引用需要 Full Reference：写入 Handle、是否必须导出、NoLoad、是否带 Path、Path Token 数据，以及递归的 Outer Reference。写端通过 ExportContext 记录已导出的 Token/Handle，避免同一导出区重复写。

// 逻辑顺序示意，不是 UE 源码逐行拷贝
CollectReferences(QuantizedState);

for (FNetToken Token : PendingTokenExports)
{
    WriteTokenAndAssociatedData(Token);
}

for (FNetObjectReference Ref : PendingObjectExports)
{
    WriteFullReference(Ref);
}

WriteMustBeMappedList();
WriteReplicatedState();

接收端：从快速命中到递归解析

Lookup先按 Handle 查缓存的弱对象。
Resolve Outer静态路径先递归确保 Outer 已解析。
Pending异步加载或前置对象未完成，可稍后重试。
Resolved找到对象并回填缓存。
Broken确认不可解析，不再无休止等待。

解析相对路径前，接收端必须从当前连接的 Remote NetTokenStoreState还原字符串。随后尝试在 Outer 中查找对象；Package 或可加载静态对象可能进入同步/异步加载。对象所在 Level 还必须满足 Bridge 的可见性判断。

MustBeMapped：这是对象批次的应用门槛

当一个可加载静态引用必须在状态应用前存在，写端把它列入 MustBeMapped。读端若暂时解析不了，会把包含该引用的整个对象批次放入 Pending 队列，而不是先把属性写成一个错误指针。

Batch Arrives

状态中引用 /Game/Weapons/Rifle，但本地尚未加载。

PendingMustBeMappedReferences

批次连同待解析 Handle 和异步优先级一起排队；后续帧重试。

Apply or Break

解析成功后重新处理；确认 Broken 后停止等待并报告问题。

MustBeMapped 不是 ACK

ACK 回答“数据包是否送达”；MustBeMapped 回答“接收端应用这批状态前，指定对象身份是否已经能映射到 UObject”。两者处于不同层次。

Chapter 06 · Cycles

循环引用处理A 指向 B、B 指向 A，为什么不会无限递归？

复制对象图不是把 UObject 内容递归内联。A 的属性引用 B 时，量化结果保存 B 的网络身份；B 引回 A 时保存 A 的身份。对象自身状态仍由各自 NetObject/Protocol 复制。于是环变成两个可去重的身份边：

Object A state: Target = RefHandle(B)
Object B state: Owner  = RefHandle(A)

ExportContext:
  exported A? yes → do not write A identity again
  exported B? yes → do not write B identity again

真正存在递归的地方：Outer 引用链

Full Reference 可以递归携带 Outer Reference。UE 5.7.4 的读取路径设置了 16 层递归上限，超过后把序列化上下文标为内部错误。这是对畸形或异常深引用链的防御，不是允许业务构造 16 层循环。

Pending 与 Broken 必须区分

状态含义处理方式常见原因
Unresolved当前没有 UObject判断是否可加载/可等待首次遇到引用
Pending存在进行中的可恢复工作保留批次，后续重试异步加载、Parent 未创建
Resolved已得到有效 UObject缓存并应用状态缓存命中或加载完成
Broken确认无法再正确解析停止永久等待，报告/断开策略按上下文执行非法路径、加载失败、前置对象已 Broken

ReplicationSystemTestPlugin 中的 TestPartialResolveOfArrayReferences、TestOutOfOrderResolveInFastArray 等测试说明，引用可以乱序到达并在后续修正；这正是“量化引用身份”和“最终解析 UObject”分阶段的价值。

不要把所有 Null 都当成数据丢失

调试时先区分：引用本身无效、远端映射尚未导入、静态对象正在加载、动态对象尚未创建、Level 不可见，还是缓存已经标记 Broken。只有最后一类通常不应继续重试。

Chapter 07 · 12.2

12.2 NetDependencyData用稀疏关系图记录“谁带动谁”

FNetDependencyData 以 InternalIndex 为键保存稀疏关系。它不仅记录 Parent→Dependent，还保留 Dependent→Parent 的反向索引，以便对象移除、过滤更新和图遍历。

Parent: CharacterDependent list: Inventory, AbilityState
Creation ParentWeapon 必须等待 Character
Parent → Child⇄Child → ParentCreation⇒Hard gate
Dependent: InventoryParent list: Character + SchedulingHint
Child: WeaponCreation parent indices: Character

添加调度依赖时做了什么检查？

UObjectReplicationBridge::AddDependentObject 会检查：

  • Parent 与 Dependent Handle 必须有效，且不能相同；
  • 当前实现只接受 Root Object，不能把 SubObject 当作这些关系的端点；
  • 重复关系的 SchedulingHint 必须一致；
  • 沿 Dependent 图遍历，拒绝会形成环的边；
  • 成功后同时更新 Parent→Dependent 和 Dependent→Parent，并刷新相关位图/缓存标志。
UObjectReplicationBridge* Bridge = /* 已初始化的 Bridge */;

Bridge->AddDependentObject(
    CharacterHandle,
    InventoryHandle,
    UE::Net::EDependentObjectSchedulingHint::ScheduleBeforeParentIfInitialState);

// 生命周期结束或关系变化时成对移除
Bridge->RemoveDependentObject(CharacterHandle, InventoryHandle);
调度边不是强一致事务

Epic 的 API 说明称其为 very loose form of dependency：Dependent 可以独立复制，也可以因 Parent 被复制而进入调度，但最终数据不保证位于同一 packet。

创建依赖：没有 Parent，就不创建 Child

AddCreationDependencyLink 比调度依赖强：它要求 Child 在客户端只于 Parent 已存在后创建，并使 Child 在 Parent 不相关时也不能相关。

Bridge->AddCreationDependencyLink(CharacterHandle, WeaponHandle);

// 关系不再需要时移除
Bridge->RemoveCreationDependencyLink(CharacterHandle, WeaponHandle);

Writer 把 Parent Handle 写入 Child 创建批次。Reader 若找不到 Parent，就把 Child 批次与 CreationDependentParents 一起排队；Parent 到达后重试。若 Parent 已 Broken，Child 也不能继续创建。这里的等待发生在实例化之前，不同于普通属性引用稍后重新映射。

选择规则

只是希望两个对象一起被考虑发送，用调度依赖；Child 的构造、Outer 或初始化确实要求 Parent 先存在，用创建依赖；只是属性要指向对方，保留对象引用即可；属于同一个 Root 的复制子对象，则使用 SubObject 关系。

Chapter 09 · 12.3

12.3 NetTokenStore首次导出完整数据，后续只传类型化标识

FNetToken 用来代表稳定、可重复的数据。它不是一个只为对象路径服务的类：名称、字符串以及自定义结构体都可以有自己的 Token DataStore。

Index20 bits · 0 invalid
TypeId3 bits
Authority1 bit
Padding8 bits

20 位 Index 给出最多 2^20 - 1 个有效索引；3 位 TypeId 最多表达 8 种 Token 类型；Authority 位区分该 Token 是否由 Authority 分配。改变这些位宽会影响网络兼容性。

StoreKey 与 Token Index 不是同一个命名空间

FNetTokenDataStore 保存具体类型的数据，并以内部 StoreKey 访问；FNetTokenStoreState 保存 Token Index→StoreKey 的映射。FNetTokenStore 拥有一个 Local State，并为每条连接维护 Remote State。

Local Store State

  • 本节点创建的 Token
  • Index → 本地 StoreKey
  • Authority 与节点角色一致

Connection A Remote State

  • A 发来的 Token 映射
  • A 已导出哪些数据
  • 解析必须传入此 State

Connection B Remote State

  • 与 A 独立的远端命名空间
  • 相同 Index 可映射不同 StoreKey
  • 不能当全局字符串池

第一次为什么不一定省？

首次使用 Token 时，发送端既要写 Token，又要导出其关联数据；只有对端确认掌握映射后，后续出现才主要支付 Token 成本。因此重复次数、原始数据大小、唯一值数量和连接生命周期共同决定收益。

NetToken 是同步字典，不是魔法压缩

一次性 UUID、每帧变化的长字符串、玩家输入拼出的无界文本，可能让 Store 持续增长且几乎没有复用收益。Token 最适合小集合、稳定、重复率高的数据。

Chapter 10 · Token Types

名称、字符串、结构体 Token三个 DataStore 的行为与成本

DataStore本地去重键导出数据主要边界
FNameTokenStoreFName名称字符串接收端重建 FName；首次仍写文本

Name Token

FNameTokenStore 维护 FNameToKey 和 StoredFNames。创建 Token 时先复用已有 StoreKey；导出时把名称转成字符串写出，接收端再构造 FName 并建立 Remote 映射。

String Token

FStringTokenStore 把字符串复制到持久内存，并以 CityHash64→StoreKey 查找。工程上应控制输入集合，并把散列碰撞视为需要规避的正确性风险，而不是依赖“概率足够低”。

Struct Token：为小而稳定的值集合准备

TStructNetTokenDataStore<T> 可以让一个 USTRUCT 自动以 Token 形式序列化。每个相同 GetUniqueKey() 的值共享 Token；Store 会保存结构体完整副本，而且没有 age-out 机制。

USTRUCT()
struct FMyStableData : public FNetTokenStructBase
{
    GENERATED_BODY()
    UE_NET_NETTOKEN_GENERATED_BODY(MyStableData)

    UPROPERTY()
    uint16 ItemType = 0;

    UPROPERTY()
    uint8 Variant = 0;

    uint64 GetUniqueKey() const
    {
        return (uint64(ItemType) << 8) | uint64(Variant);
    }
};

Struct Token 的硬边界

结构体应被视为不可变值，不要保存指向可变子数据的指针；不要嵌套其他 NetToken、ObjectReference、FName、GameplayTag 或含 NetGUID 的类型；GetUniqueKey() 必须在语义上唯一。大结构、无界排列、复杂散列都会把带宽收益换成内存和 CPU 成本。

Chapter 11 · 12.4

12.4 NetObjectFactoryCreation Header 是远端实例化协议

UNetObjectFactory 的职责不是复制属性，而是回答：远端收到一个新 NetObject 时，如何查找、绑定或创建对应 UObject？为此，发送端先生成 FNetObjectCreationHeader，Header 带有 ProtocolId 和 FactoryId，并由具体 Factory 序列化自定义内容。

SENDER 01CreateHeader

基类验证 Handle，派生类填充具体 Creation Header。

SENDER 02WriteHeader

具体 Factory 把类型、引用和 Spawn/Template 数据写入位流。

WIRE 03FactoryId + ProtocolId

远端知道用哪个 Factory，以及用哪个 Replication Protocol。

RECEIVER 04ReadHeader

对应 Factory 分配 Header 类型并反序列化。

RECEIVER 05Instantiate

Resolve、Spawn、NewObject 或绑定预注册实例。

RECEIVER 06Apply + Callbacks

Bridge 应用初始状态，再进入 PostInit/生命周期回调。

基类接口的职责切分

接口执行端职责
CreateAndFillHeader发送端从本地实例提取远端创建所需数据
SerializeHeader双向配对定义 Header 的线上格式
CreateAndDeserializeHeader接收端分配正确 Header 类型并读取
InstantiateReplicatedObjectFromHeader接收端返回 Instance、可选 Template 和 Bridge Flags
PostInstantiation接收端实例已存在、初始属性尚未应用
PostInit接收端初始复制状态已经应用
DetachedFromReplication接收端销毁动态实例、重置稳定对象或回收到池

Factory 返回的 Flags 决定远端是否允许销毁实例、是否在状态应用后触发 SubObject 创建回调等。Factory 创建成功也只是第一步：Bridge 仍需用 Header 中的 ProtocolId 找到一致的 Replication Protocol，分配状态并应用初始数据。

Chapter 12 · Factories

内置与扩展NetActorFactory、NetSubObjectFactory 与自定义注册

UNetActorFactory：查找、预注册或 Spawn

UNetActorFactory 根据 ActorReference 选择三种 Header：

STATIC

Static Actor

Header 携带 ObjectReference；接收端按稳定路径查找已有 Actor，不新建。

PRE-REGISTERED

预注册 Actor

Handle 已关联本地实例，Header 当前不需额外创建数据。

DYNAMIC

Dynamic Actor

Header 携带 Archetype、Level、Location、Rotation、Scale、Velocity 与自定义创建位。

动态 Actor 在接收端先解析 Archetype 和可选 Level，再用 SpawnActorAbsolute 创建远端拥有的实例。Streaming Level 若当前不可用，创建不能凭空继续。

UNetSubObjectFactory：稳定对象绑定，动态对象 NewObject

UNetSubObjectFactory 只负责 SubObject：

  • 稳定命名或 Static Handle：Header 携带 ObjectReference，远端找到既有 SubObject；
  • 动态 SubObject：Header 携带 TemplateReference，以及 Transient、Root 或 OuterReference 的选择；
  • 远端解析 Template 和 Outer 后，用 NewObject 创建；
  • 若自定义 Outer 不可解析，当前实现可以回退到 RootObject，并可通过 CVar 打印警告；
  • 创建结果请求 Bridge 在所属 Actor 状态应用后再触发 SubObject 回调。

自定义 Factory：注册必须早于 Iris ReplicationSystem

FNetObjectFactoryRegistry 保存 Factory 模板。UE 5.7.4 的默认上限是 16，注册名称必须唯一，类必须派生自 UNetObjectFactory。

namespace UE::Net
{
    void InitMyNetFactories()
    {
        FNetObjectFactoryRegistry::RegisterFactory(
            UMyNetObjectFactory::StaticClass(),
            UMyNetObjectFactory::GetFactoryName());
    }

    void ShutdownMyNetFactories()
    {
        FNetObjectFactoryRegistry::UnregisterFactory(
            UMyNetObjectFactory::GetFactoryName());
    }
}
注册时机是协议的一部分

所有 Factory 必须在任意 Iris ReplicationSystem 创建前注册；运行中注册/注销会被拒绝。注销后 Registry 保留槽位位置，后续 Factory 可能复用旧 FactoryId，因此缓存 ID 的代码必须同时维护有效期，不能把 ID 当永久类型号。

ReplicationSystemTestPlugin 的 UReplicatedTestObjectFactory 是很好的最小参考：自定义 Header 保存 Archetype 和组件数量，实现填充、序列化、反序列化、实例化、PostInit、Detach 与 SubObject 回调；NetFactoryRegistration 测试还展示了注销后 ID 槽位复用。

Chapter 13 · Practice

案例、排错与源码导航把四套机制串成一条可验证的链

案例：角色、武器实例与静态配置资产

1角色进入复制

NetActorFactory 为动态角色写 Archetype、Level 与 SpawnInfo。

2配置资产被引用

ObjectReferenceCache 为稳定资产生成 PathToken；首次向该连接导出字符串。

3武器等待角色

Creation Dependency 让 Weapon Child 批次在 Character 存在前保持 Pending。

4SubObject 落地

NetSubObjectFactory 解析 Template/Outer，用 NewObject 创建并在初始状态后回调 Actor。

如果配置资产还在异步加载，角色批次可能因 MustBeMapped 暂停；如果角色已创建但背包对象只要求更新顺序，可以另加 ScheduleBeforeParentIfInitialState。注意这两个等待分别来自引用解析与依赖调度，日志和修复方向不同。

排错清单：从身份到实例逐层排

  1. Reference 是否有效：Handle 或 PathToken 是否存在，Traits 是否允许导出？
  2. Export 是否发生：Token 与 Full Reference 是否先于使用它们的状态写出？
  3. Remote State 是否正确：解析远端 Token 时是否使用当前连接对应的 State？
  4. Resolve 状态是什么：缓存命中、Pending、Level 不可见，还是 Broken？
  5. 批次在等谁：MustBeMapped Reference，还是 CreationDependentParents？
  6. 调度边是否合理：是否意外形成环，或把 SubObject 当作 Root 关系端点？
  7. Factory 是否一致：两端是否以相同顺序和名称在 Iris 启动前注册？
  8. Header 是否足够：动态实例的 Archetype、Level、Template、Outer 是否都能解析？
  9. Token 集合是否失控：是否不断创建高基数 Token，Struct Store 是否无界增长？

本篇结论

REFERENCE

身份先于指针

Handle/Path/Token 让远端重建身份；解析可能 Pending，也可能 Broken。

DEPENDENCY

关系各司其职

调度依赖是 loose ordering，创建依赖是实例化门槛，SubObject 负责所属。

FACTORY

Header 驱动创建

Factory 把线上 Creation Header 变成本地实例，属性状态随后才真正落地。

Iris 不复制“地址”，它复制可解析的身份；不把所有关联都叫依赖，而是分别约束调度、创建和所属；也不假设远端已经有实例，而是用 Factory 明确定义创建协议。

下一部分将进入条件复制（Conditionals），继续讨论对象已经可寻址、可调度之后，哪些成员在什么连接、什么阶段、什么条件下才应该进入 ChangeMask 与最终输出。
