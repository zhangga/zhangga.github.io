🎯 第八部分：数据流与传输

第七部分把游戏值变成了比特；第八部分回答三个更靠近网络的问题：这些比特怎样和其他数据共用一个包、丢包后谁知道该补什么、接收端何时才真正修改游戏对象。

继续沿用前文的百人大逃杀角色：服务器上某玩家受到伤害并开了一枪，Health 从 100.0 变为 73.5，Ammo 从 30 变为 29。轮询或 Push Model 已经发现变化，过滤和优先级也已为某条连接做完决定。现在接棒的是逐连接 Writer、Replication DataStream、Manager 与 Reader。

这条链路最容易被“可靠/不可靠”四个字误导。Iris 对象状态的目标通常是收敛到最新值，可靠 Attachment 的目标是不能漏掉事件，不可靠 Attachment 的目标是有机会就发，丢了不补旧事件。它们可以出现在同一个 Replication DataStream 中，却不能用同一种心智模型解释。

版本与证据边界

本文以本地 Unreal Engine 5.7.4 源码为准，Build.version 的 CompatibleChangelist 为 47537391。公开接口参考 Epic Games UE 5.7 文档；Writer、Reader、Manager 包格式和 Chunk 常量属于实现细节，升级引擎时必须重新核对。

🗺️ 端到端传输总览

三条互相咬合的路径

Send Path选择并写入

Pre-Send 阶段准备量化状态、Scope、Priority 与 ChangeMask；逐连接 Writer 把可发送对象组织成 batch。

Receive Path读取再派发

Reader 先把整包读入接收状态，处理创建和依赖，再通过 Fragment/Bridge 应用到游戏对象。

Feedback Path确认或恢复

包的 Delivered、Lost、Discard 结果沿 Record 返回各个 DataStream，驱动确认、重排、重发或清理。

官方的 Iris 流程也把这两阶段分开：Pre-Send Update 才访问游戏状态并完成脏数据复制、量化、过滤和优先级；Send Update 面向每条连接调度对象并序列化。换言之，Writer 不是回到 Actor 上逐属性检查一遍，它消费的是 ReplicationSystem 已整理好的内部状态。

层与层之间交付什么

交接点上游交付下游职责不负责什么
ReplicationSystem → Writer逐连接 Scope、Priority、Dirty ChangeMask、生命周期和 Attachment调度、组 batch、按位预算写入重新轮询 Gameplay 属性
Writer → ReplicationDataStreamReplication 协议载荷与 in-flight record适配统一 DataStream 生命周期改变对象复制语义
子流 → DataStreamManager位流片段、WriteResult、可选 Record多流 framing、状态握手、反馈分派通用权重公平调度
Manager → 连接本包 Iris DataStream 载荷包发送和投递结果理解对象或属性含义
Reader → Fragment/Bridge接收状态、ChangeMask、创建/销毁意图实例化、Dequantize、Apply、通知在读到单个字段时立即调用 RepNotify

每条连接都有自己的“账”

过滤、优先级、对象创建确认、已发送 ChangeMask、可靠 Attachment 队列和 Baseline 都具有连接语义。同一个服务器对象可能对连接 A 已创建并确认，对连接 B 还未进入 Scope，对连接 C 的上一包刚刚丢失。因此 Writer/Reader 是连接级状态机，不能用“对象已经发过”这种全局判断代替。

同一个 NetObject Connection A: Created + no loss
Connection B: PendingCreate
Connection C: Created + lost state pending resend

🔌 8.1 DataStream 系统

UDataStream 是协议插槽

UE 5.7 官方 API 将 UDataStream 定义为复制自定义数据的接口，适合 bulky data 或需要特殊投递保证的数据。它不是一个现成的“可靠字节流”；它规定何时准备、如何写、如何读、何时知道包结果，具体可靠性由子类利用这些回调实现。

// UE 5.7.4 接口骨架（为阅读省略导出宏和注释）
class UDataStream : public UObject
{
public:
    virtual void Init(const FInitParameters& Params);
    virtual void Update(const FUpdateParameters& Params);
    virtual EWriteResult BeginWrite(const FBeginWriteParameters& Params);
    virtual EWriteResult WriteData(FNetSerializationContext& Context,
                                   FDataStreamRecord const*& OutRecord);
    virtual void EndWrite();
    virtual void ReadData(FNetSerializationContext& Context);
    virtual void ProcessPacketDeliveryStatus(
        EPacketDeliveryStatus Status,
        FDataStreamRecord const* Record);
    virtual bool HasAcknowledgedAllReliableData() const;
};

生命周期不是一对 Read/Write 那么简单

阶段调用核心用途关键约束
建立Init接收 ReplicationSystemId、ConnectionId、Name、PacketWindowSize、NetExports必须先初始化，Manager 才能创建流
帧更新Update驱动 PreSendUpdate 或 PostTickFlush 对应工作不等同于真正写包
批次准备BeginWrite建立可跨多次 WriteData 使用的临时上下文返回 NoData 后本批次不会继续写
逐包写入WriteData向 BitStream 写数据并返回 Record只有真正写入才应产生 Record
批次收尾EndWrite释放批次临时状态、统计未写完工作不是 ACK 回调
接收ReadData按与 WriteData 对称的格式读取Context 错误必须立即传播
包反馈ProcessPacketDeliveryStatus处理 Delivered、Lost 或 DiscardRecord 顺序与 WriteData 顺序对应
拆除Deinit释放流级资源outstanding record 会先得到清理机会

WriteMode 与 WriteResult

Full

默认写入模式，允许流写常规数据。ReplicationWriter 在这里处理对象状态、创建/销毁和各类 Attachment。

PostTickDispatch

用于主网络 Tick 尾部的特定数据。UE 5.7.4 Writer 当前只在该模式准备未发送的非可靠 OOB Attachment。

MaxPackets

限制一个 Begin/EndWrite 批次中的常规包数；0 表示不限。它是调用上限输入，不是带宽保证。

EWriteResult含义Manager 可以如何理解
NoData本次没有提交数据丢弃子流，不为接收端设置该流 presence bit
Ok本帧写完，或即使还有数据也不想继续写本次有效，但无需仅为该流继续开包
HasMoreData还有数据，若预算允许希望再次写可能继续调用；绝不代表一定获得下一包

FDataStreamRecord：寄给未来的回执存根

WriteData 写进包的是比特，包的结果却稍后才知道。Record 把这两个时间点接起来：子流在 Record 中保存“这包写了哪些状态、哪些可靠队列条目或哪些 chunk sequence”；连接得到投递结果后，Manager 把原 Record 原样还给该流。

WriteData(packet 412)→Record: state/chunk refs→packet in flight→Delivered / Lost / Discard
  • Record 的具体字段不由基类规定，子类可以派生自己的结构。
  • Record 由 DataStream 分配和拥有，但 Manager/连接在回调前保存指针。
  • 返回 NoData、发生 Context 错误或一个比特都没写时，不应伪造 Record。
  • 连接关闭时 outstanding 记录会以 Discard 清理，避免把资源永久留在 in-flight 状态。

动态流也有握手状态

Invalid→PendingCreate→WaitOnCreateConfirmation→Open
Open→PendingClose→WaitOnCloseConfirmation→Invalid

创建与关闭信息也要经过网络。如果包含状态迁移的包丢失，Manager 会把等待确认状态退回 PendingCreate 或 PendingClose 并重新标脏。关闭前还会询问 HasAcknowledgedAllReliableData，防止可靠队列尚未确认就直接撤销流。

结论

DataStream 提供的是“可实现特殊投递语义的回调契约”。真正的可靠、重传、合并或丢弃策略存在于 ReplicationWriter、NetTokenDataStream、ChunkedDataStream 或你的自定义流中。

🧭 8.2 DataStreamManager

Manager 自己也是一个 DataStream

UDataStreamManager : UDataStream 形成组合器：连接只需要驱动 Manager，Manager 再把生命周期调用转发给活动子流。每条连接拥有自己的 Manager 和流实例，因此暂停某个连接的 Replication 流不会全局暂停其他连接。

UE 5.7.4 的引擎基础配置自动创建两个核心流：

名称类默认发送状态角色
NetTokenUNetTokenDataStreamSend同步 Token 导出，使名称等压缩标识可在远端解析
ReplicationUReplicationDataStreamSend把连接的 ReplicationWriter/Reader 接入统一流接口

FDataStreamDefinition 还包含 bAutoCreate 和 bDynamicCreate。Fixup 按配置数组顺序分配 StreamIndex，所以两端定义的名称、类和顺序必须一致。UE 5.7.4 Manager 上限是 32 条流；这也是内部 framing 的实现约束，不应当作永远不变的产品 API。

5.7.4 的 Manager 包头

下面是当前源码可直接复核的概念布局。N 是本包声明的流数量；只有状态掩码命中的流才带 4-bit 状态值，只有数据掩码命中的流才跟随 payload。

StreamCount - 1
5 bits DataStreamMask
N bits HasStateChanges
1 bit StateMask
N bits（可选） State Values
4 bits / dirty stream
Stream 0 payload（若存在） Stream 1 payload（若存在） … 按 StreamIndex 顺序

Manager 先创建子 BitStream，并预留流数量和 DataStreamMask；各子流写完后再回填真实 mask。如果连 Manager 头都放不下，它丢弃整个子流并返回 NoData。子流返回 NoData 或报错时，其子流同样被丢弃，不会把“写了一半的自定义协议”暴露给接收端。

多流协调的真实算法

  1. 遍历活动流，跳过 Pause。
  2. 只允许 Open 或 PendingClose 的流写业务数据。
  3. 为当前流从 ManagerStream 创建 SubStream。
  4. 子流写入成功则提交 SubStream、设置 presence bit 并保存 Record。
  5. 子流 NoData 或出错则丢弃 SubStream；错误继续传播给上层 Context。
  6. 所有流处理后回填 mask，提交 ManagerStream，并把各子流 Record 汇总为一个包级 Record。
大纲中的“带宽分配”要校准

UE 5.7.4 Manager 没有通用的 per-stream 权重、公平份额或令牌桶。活动流按 StreamIndex 顺序共享当前 BitStream 剩余容量；前面的流可以消耗更多空间，后面的流只能看到剩余预算。HasMoreData、MaxPackets 和连接包容量共同决定是否还有后续写入机会。

因此，真正需要流间 QoS 时，不能只期待 Manager 自动公平；你需要审视定义顺序、子流自限策略、连接允许的包数，以及是否应该把业务拆成独立 DataStream。

接收和反馈保持同一索引顺序

接收端读取 StreamCount、DataStreamMask 和状态变化；验证数量与 mask 后，只为 DataStreamMask 命中的流调用 ReadData。任何流设置 Context error 或发生 overflow，Manager 都停止继续解析，避免下一个流从错误位偏移开始读。

投递反馈端先处理 NetExports，再依据包级 Record 中的两个 mask：

  • 状态迁移包 Lost：把 Create/Close 等等待确认状态退回待发送状态并重新标脏。
  • 数据流命中：把相同 Status 和对应子 Record 转发给子流。
  • 处理结束：按写入顺序弹出 Manager Record。

自定义动态流配置

下面是解释性配置，类路径必须换成项目模块实际导出的 UDataStream 子类。动态创建要求两端共享同一 Definition，并通过 CreateStream / CloseStream 走握手。

[/Script/IrisCore.DataStreamDefinitions]
+DataStreamDefinitions=(
    DataStreamName=MatchBootstrap,
    ClassName=/Script/MyGame.MatchBootstrapDataStream,
    DefaultSendStatus=EDataStreamSendStatus::Send,
    bAutoCreate=false,
    bDynamicCreate=true)

SetSendStatus(Pause) 只是阻止 Manager 调用该流的写入路径，不清空流的可靠队列，也不等于关闭流。恢复 Send 后，子流仍要根据自己的状态决定返回 NoData、Ok 或 HasMoreData。

📤 8.3 ReplicationWriter（发送端）

UReplicationDataStream 是薄适配层

DataStream 回调UE 5.7.4 转发目标作用
UpdateFReplicationWriter::Update驱动对应更新阶段
BeginWriteFReplicationWriter::BeginWrite检查待发送工作并建立批次上下文
WriteDataFReplicationWriter::Write写销毁、OOB Attachment 和对象 batch
ReadDataFReplicationReader::Read解析远端 Replication 载荷
ProcessPacketDeliveryStatusProcessDeliveryNotification处理 Delivered/Lost/Discard
HasAcknowledgedAllReliableDataAreAllReliableAttachmentsSentAndAcked判断可靠 Attachment 是否全部完成

这层不复制状态，也不另做可靠队列；它让复杂的 Writer/Reader 能作为 Manager 的一个标准子流存在。

Writer 接到的是整理后的输入

Scope / Priority

哪些对象属于本连接，以及调度优先级；来自前面的过滤、优先级和累积流程。

Dirty ChangeMask

ReplicationSystem 已完成 Poll/Push、Copy 与 Quantize；Writer 同步连接级 mask，而不是查询 Gameplay 属性。

Lifecycle / Attachments

创建、销毁、TearOff、SubObject、依赖、可靠与非可靠队列共同参与写入。

BeginWrite 首先检查复制是否启用、ReplicationRecord 是否还有容量，以及普通模式下是否存在 dirty objects、pending destroy、OOB 或 Huge Object。若有工作，它调用 ScheduleObjects 建立本批次对象数组；PostTickDispatch 模式当前只面向未发送的非可靠 OOB Attachment。

对象生命周期状态决定写什么

状态含义典型写入/反馈动作
PendingCreate远端尚未创建携带创建信息与初始状态
WaitOnCreateConfirmation创建包已在途Delivered 后进入 Created；Lost 后恢复并提升调度紧迫度
Created远端已确认对象发送 ChangeMask 命中的状态与 Attachment
WaitOnFlush销毁/TearOff 前等待状态或可靠数据排空持续检查 in-flight state 和可靠队列
PendingTearOff需要发送 TearOff确认后停止正常复制
PendingDestroy准备发送销毁进入 WaitOnDestroyConfirmation
WaitOnDestroyConfirmation销毁包已在途Delivered 后安全停止跟踪；Lost 后重排销毁
HugeObject内部特殊记录通过 Huge Object Attachment 路径处理超大复制 batch

BeginWrite → Write → EndWrite

1. BeginWrite：建立调度快照

收集本批次待写对象，记录 WriteMode、MaxPackets，并初始化索引、统计和依赖队列。

2. Write：先写 16-bit batch count 占位

随后写 pending destroy、OOB Attachment 和普通对象；完成后回填真实 batch 数。

3. 对象 batch：root + subobjects

WriteObjectBatch 与 WriteObjectAndSubObjects 处理对象头、创建信息、状态、附件和依赖。

4. 位预算失败：回滚

FNetBitStreamRollbackScope 确保没有完整对象可提交或发生 overflow 时，整个本次 Replication 写入回退。

5. 成功：Push ReplicationRecord

记录本包涉及的对象状态与可靠队列条目，等待后续投递通知。

6. EndWrite：释放批次内存

统计未发送对象/依赖和 Huge Object 队列，释放 ScheduledObjectInfos；不是确认网络到达。

Writer 不是简单地遇到大对象就停止。当前实现会在最后一个常规包中继续尝试若干更小对象，减少一个大 batch 阻塞后续小状态的机会；具体阈值由 Writer 参数控制。真正超大的对象 batch 还可能走 Huge Object 路径。

可靠 vs 不可靠：必须拆成三类

数据Lost 后保证的核心语义适合
对象状态丢失 ChangeMask 与当前 dirty mask 合并，必要时提高调度优先级；后续可发送更新后的最新值最终收敛到最新状态，不重放每次中间变化Health、Ammo、位置、姿态
可靠 Attachment可靠记录保留，重新排队直到确认不能静默漏掉的有序业务数据可靠 RPC、必须执行的 Attachment
非可靠 Attachment不按可靠队列补发旧事件一次发送机会；业务允许丢失高频、可被下一次覆盖的瞬时事件
状态丢包不等于“原包原样重发”

假设包 A 带 Health=80，A 丢失期间服务器又变成 Health=73.5。Writer 恢复的是“Health 仍需要到达”的 ChangeMask 语义，下一次可以直接发送 73.5；客户端不需要先看见 80 再看见 73.5。若 80 和 73.5 分别代表两个必须执行的事件，就不该建模为普通状态。

Delivered、Lost、Discard 三条分支

Delivered

确认创建/销毁和已发状态，处理 Attachment ACK；新 Baseline 也只有在确认路径上才能安全参与后续决策。

Lost

恢复创建/销毁状态，合并丢失 ChangeMask，给丢失状态增加调度紧迫度，并通知可靠 Attachment 队列。

Discard

连接关闭或记录被放弃时清理 tracking；不把它当成仍应继续在该连接上重传的普通 Lost。

ProcessDeliveryNotification 从 ReplicationRecord 按包弹出记录，再明确分派到 HandleDeliveredRecord、HandleDroppedRecord 或 HandleDiscardedRecord。这正是 DataStream Record 契约在实际复制流中的落点。

Delta 的边界

Writer 可以根据初始状态、有效 Baseline 和当前 ChangeMask 选择 initial/full/delta 序列化路径。若包含新 Baseline 的包丢失，当前源码会通知 BaselineManager 丢失并使 pending baseline 失效。本文只解释这个投递交接点；Baseline 生成、确认窗口、Delta 位成本和回退策略将在第十部分展开。

📥 8.4 ReplicationReader（接收端）

Read 先把整包变成“待派发状态”

FReplicationReader::Read 设置内部序列化上下文、PackageMap、对象 ResolveContext 与 NetBlobReceiver，然后执行以下顶层步骤：

1. 读取流级调试特性和 16-bit batch count

UE 5.7.4 还用 8192 作为接收 batch 数的 sanity guard；这是当前实现防线，不是业务可用配额。

2. 读取 pending destroy

先处理应停止复制的对象信息，并建立临时 ObjectsToDispatch 数组。

3. ReadObjects

解析 root/subobject batch、Handle、创建信息、状态和 Attachment。

4. ProcessHugeObject

如果存在完整 Huge Object，组装后再按内部复制格式处理。

5. DispatchStateData

解析依赖、Dequantize 并 Apply；状态不是读到一个字段就立即写入 Gameplay 对象。

6. Resolve + EndReplication

继续尝试之前未解析的引用，最后派发应该结束复制的对象。

对象创建不是 Reader 直接随手 NewObject

收到未知 Handle 的 initial state 时，Reader 读取创建信息并借助 ReplicationBridge/Factory 路径建立游戏实例和 Replication Protocol。SubObject 还依赖 root/owner；创建依赖或对象引用未准备好时，batch 可以被延迟，而不是在缺少类型与父对象时强行反序列化。

接收情形Reader 需要确认未就绪时
已有对象更新Handle 映射、Protocol 与 receive buffer非法 Handle 进入 Context error

网络数据首先修改 Iris 的 ReceiveStateBuffer。这样 Reader 能先读完整包、收集待派发对象、处理 unresolved references，再以正确的 root/subobject 和 Attachment 顺序触碰 Gameplay 实例。

引用和依赖为什么会推迟 Apply

一个状态成员可能包含 NetObjectReference；initial state 也可能依赖尚未加载的对象。Reader 使用 ObjectReferenceCache 收集已解析与未解析引用，并维护 unresolved ChangeMask。之后引用变得可解析时，相应成员重新加入 Apply 范围，而不要求业务对象接受无效指针。

Deserialize→Collect references→MustBeMapped?→Dispatch now / defer

这也解释了一个常见现象：抓包看见数据到达，Reader 也没有 BitStream 错误，但属性尚未变化。问题可能不在传输，而在创建依赖、MustBeMapped 引用或 deferred batch。

Apply 与 RepNotify 的真实调用边界

Reader
DispatchStateData DequantizeAndApply
Initialize PreNetReceive + Fragment::ApplyReplicatedState PostNetReceive + Fragment::CallRepNotifies PostRepNotifies

UE 5.7.4 的 FDequantizeAndApplyHelper 先按 owner 调用 PreNetReceive，再让各 Fragment 执行 ApplyReplicatedState。批次后处理阶段调用 PostNetReceive；具备 HasRepNotifies trait 的 Fragment 才执行 CallRepNotifies，最后调用 PostRepNotifies。

因此更准确的表述是：Reader 决定何时派发，Fragment/Descriptor 决定如何应用和调用哪些 RepNotify。是否 OnChanged、是否 Always、旧值参数、Fast Array 专用通知和 unresolved reference 都会影响最终行为。

可靠 RPC 与状态 Apply 的顺序

Reader 还要协调 Attachment。UE 5.7.4 有 net.Iris.ExecuteReliableRPCsBeforeApplyState 控制可靠 RPC 是否在状态 Apply 前派发；非可靠和剩余 Attachment 通常在后续阶段处理。本文只强调顺序可配置且由 Reader 派发阶段协调，NetBlob、RPC Handler 和 Attachment 载荷结构留到第九部分。

位流错误必须立即止损

  • batch count 超出 sanity guard。
  • 对象 batch 声明长度与实际读取位数不一致。
  • Reader overflow/underflow。
  • Handle、Owner Handle 或创建依赖非法。
  • 在不期待状态数据时收到状态。
  • Huge Object 不完整或内部载荷损坏。

这些路径会向 FNetSerializationContext 设置具体错误并停止继续解析。网络协议最危险的做法不是“报错”，而是吞掉错误后让下一对象从错误位偏移继续读。

🧩 8.5 ChunkedDataStream（分块数据流）

它是实验性的自定义大载荷流

UE 5.7.4 头文件把 UChunkedDataStream 标成 Experimental：用于拆分并承载可能带 exports 的大型 payload。它不是默认自动创建的 Replication 或 NetToken 流。项目要使用它，需要定义匹配的 DataStream、让两端获得同一 StreamIndex，并处理创建、派发和错误后的关闭策略。

192 B单个 Chunk 的最大 payload
11 bits线上显式 sequence 位数
2048最大未确认 chunk 窗口
10 MiB默认发送队列与未派发接收缓冲上限

此外，当前发送端为 export payload 预留的临时上限是 512 KiB。以上都是本地 5.7.4 快照的实现值；公开 Setter 可以调整两侧 10 MiB 上限，后续版本也可能改变常量。

Chunk 的线上结构

HasChunk
1 bit InSequence
1 bit Sequence
11 bits（可选） IsFirst
1 bit IsExport
1 bit（首块） PartCount
Packed（首块） IsFull
1 bit PartBytes
Packed（可选） Payload ≤ 192 B

连续 sequence 可以省掉显式 11-bit 值；首块携带 PartCount，并把最后一块长度编码在自己的 PartByteCount 字段中，由接收端恢复总大小。每次 WriteData 最后还写一个 false continuation marker，告诉 Reader 当前包没有更多 chunk。

发送端：共享载荷、分块、等待 ACK

  1. EnqueuePayload 检查空载荷和发送缓冲上限；有 PackageMap exports 时先生成 export payload。
  2. SplitPayload 按 192 字节创建 FDataChunk，分配本地递增 sequence，并让每个 chunk 引用同一个 FSendQueueEntry。
  3. WriteData 在当前包剩余位中尽量写多个未发送 chunk，把 sequence 加入 DataChunksPendingAck。
  4. Record 指针位置被当前实现用来编码本包 chunk 数；回调按数量从 PendingAck 队列处理。
  5. Lost 清除对应 sequence 的 Sent 标记，使它重新具备发送资格；Delivered 设置 Acked。
  6. PopDeliveredChunks 只从队头连续弹出已确认 chunk，保证 exports 在前序数据确认后再正式 acknowledge。

HasAcknowledgedAllReliableData 在发送队列为空时才返回 true。可靠性不是“把整份 64 KiB 每次重发”，而是对 chunk sequence 做确认和补缺。

接收端：允许乱序到达，但按序组装

Reader 用 ExpectedSeq 计算收到 sequence 的环形差值，并在 DataChunksPendingAssembly 中为缺口保留位置。只有从 ExpectedSeq 起连续可用的 chunk 才进入载荷组装。遇到新首块时，接收端根据 PartCount 和最后一块大小计算总长度。

Seq 40 arrived→Seq 42 buffered→Seq 41 arrives→assemble 40, 41, 42

完整 export payload 会先被处理；真正业务 payload 的 DispatchReceivedPayload 还会等待 MustBeMapped 引用可解析。若业务长时间不 Dispatch，累计未派发字节超过上限，Reader 设置错误状态。当前实现公开 HasError，使用者应检测并按流生命周期策略关闭，而不要继续向错误流投递。

64 KiB 载荷怎么算

解释性案例：不含任何协议头和 exports ChunkCount = ceil(65,536 / 192) = 342
341 × 192 = 65,472 bytes
LastChunk = 65,536 - 65,472 = 64 bytes

假设序列 120–125 在一个包里，其中 123 所在包丢失：已确认 chunk 不需要回到 SendQueue 起点；123 的 Sent 标志清除后重新获得写入机会。真实网络成本还包括 continuation、sequence、首块、长度、DataStream mask、NetExports 和底层包头，不能用 65,536 字节除以 MTU 直接得出最终包数。

🎮 一包角色状态的完整旅程

场景输入

解释性场景只跟踪两个成员：Health 使用第七部分的 0–1000、0.1 精度模型，字段载荷为 14 bits；Ammo 假设 0–63，字段载荷为 6 bits。实际对象 batch 还包含 Handle、对象头、ChangeMask、条件、创建/销毁信息、对齐和流/包级开销。

正常到达

T0 · Pre-Send Update

Poll/Push 发现变化，复制状态被 Copy + Quantize，过滤与优先级完成，ChangeMask 进入逐连接 Writer。

T1 · Writer BeginWrite

对象在 Scope 内且有 dirty state；ScheduleObjects 把它排入本连接本批次。

T2 · Writer Write

对象 batch 写入 Handle、状态标记、ChangeMask、Health 与 Ammo 字段；ReplicationRecord 记住本包状态。

T3 · Manager

Replication 子流成功，Manager 设置对应 DataStreamMask 位并保存子 Record；若 NetToken 同包也有数据，它按自身 StreamIndex 写入。

T4 · Reader Read

接收 Manager 选择 Replication 流；Reader 解析对象 batch，把量化值写入 ReceiveStateBuffer。

T5 · Dispatch / Apply

引用和依赖就绪，Fragment 应用 Health=73.5 与 Ammo=29；随后按 RepNotify 条件调用通知。

T6 · Delivered

包回执沿 Manager Record 返回 Writer，确认本次 in-flight state；相关 ChangeMask 不再因这次发送保持待确认。

同一包丢失

客户端没有应用 73.5/29。发送端收到 Lost 后，Writer 恢复需要补达的 mask 并提高丢失状态的调度紧迫度。若此时服务器 Health 又降到 61.0、Ammo 变为 28，下一包可以直接发送 61.0/28；它不是录像回放，不必先还原 73.5/29。

Packet A
73.5 / 29 Lost Server latest
61.0 / 28 Dirty mask merged + priority bump Packet B
61.0 / 28

如果同一包还带可靠 Attachment，它的可靠记录另行重排；非可靠 Attachment 则不会因为状态要补发而自动复活。这正是“一个包、三种语义”的关键。

性能原则

  • 先减少候选：过滤和优先级比在 Writer 末端反复尝试塞不下的对象更有效。
  • 控制状态尺寸：大对象更容易触发回滚、TrySmallObject 或 Huge Object 路径。
  • 可靠事件要克制：可靠队列需要保存记录、等待 ACK 并可能阻塞 Flush/Close。
  • 及时消费 Chunked payload：接收队列是背压边界，不是永久文件缓存。
  • 监控最坏情况：平均包很小不能掩盖创建状态、动态数组或对象引用同时爆发。
  • 用 Insights/NetTrace 看真实位数：字段模型不能替代流头、对象头、exports 与包级统计。

版本迁移检查

从旧文章或 UE 5.5 迁移到 5.7 时，至少重新检查：Iris 源码根目录、DataStream 虚函数签名、WriteMode、Manager framing、Writer 对 PostTickDispatch 的处理、ChunkSize/sequence 常量、Huge Object 实现和 RepNotify Apply 路径。内部类多在 Private 目录，不能把本文中的私有符号当作项目稳定扩展 API。

本章结论

  • DataStream 的核心不是“写字节”，而是写入、读取与未来投递反馈三者闭环。
  • Manager 负责 framing 与协调，不提供通用公平带宽调度。
  • Writer 消费已准备好的逐连接状态，通过回滚和 Record 管理包预算与 in-flight 状态。
  • Reader 将 Read 与 Apply 分离，引用、依赖和 Attachment 顺序都可能推迟 Gameplay 更新。
  • 对象状态、可靠 Attachment 和非可靠 Attachment 具有不同丢包语义。
  • ChunkedDataStream 是独立的实验性大载荷流，不是 ReplicationWriter Huge Object 的别名。
