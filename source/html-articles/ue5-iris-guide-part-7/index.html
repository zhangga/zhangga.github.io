🎯 第七部分：序列化系统

过滤决定“发给谁”，优先级决定“谁先发”，序列化决定“一个游戏值最终变成哪些比特”。这三件事依次收缩问题，却不能互相替代。

设想服务器里一名角色从 100.0 点生命降到 87.3。游戏层看到的是一个 float；Descriptor 知道它属于哪个复制成员；Serializer 把它映射成离散索引 873；ChangeMask 表示这个成员变了；Writer 把 14 个有效比特写入包；客户端读取后得到同一个量化索引，再还原为 87.3 并应用到对象。本文就沿着这一个值走完全程。

本文不是把所有 Serializer API 罗列一遍，而是回答四个工程问题：表达是否足够准确、平均与最坏需要多少位、量化状态由谁拥有、错误在哪一层被发现。最后会把结论放回位置、旋转、生命、弹药、背包和玩家名称组成的大逃杀角色状态中。

版本边界

分析基线是 UE 5.7.4。当前源码根目录为 Engine/Source/Runtime/Net/Iris/；旧文章常见的 Engine/Source/Runtime/Experimental/Iris/Core/ 是较早版本布局。复制旧路径、宏或注册时机前，先确认目标引擎版本。

🧭 7.1 序列化系统概述

它和存档序列化有什么不同

普通存档经常追求“完整、稳定、方便演进”；网络状态序列化首先受实时预算约束：数据要按连接调度，允许在业务容忍范围内损失精度，要能跳过未变化成员，还要处理基线、对象引用、包边界和恶意输入。FArchive 的通用性很有价值，但 Iris 的热路径需要更明确的类型、量化状态和比特级读写契约。

可以把它类比成搬运乐高城堡。游戏对象是已经搭好的城堡；SourceType 是城堡当前形态；QuantizedType 是按颜色和格点整理后的零件清单；BitStream 是把清单紧密装进运输箱。接收端按同一规则拆箱，再搭成业务可用的近似城堡。规则不一致，得到的不是“精度差一点”，而是从错位的比特开始全部误读。

四层职责不能混淆

层次回答的问题典型机制常见误区
变化检测什么变了？Poll、Push Model、IsEqual、ChangeMask以为量化会自动标脏
量化用什么离散形式表示？Quantize / Dequantize只看位数，不定义范围和误差
序列化按什么顺序写入/读出？Serialize / Deserialize、BitStream发送端与接收端条件分支不对称
调度本帧是否有机会发送？过滤、优先级、Writer 位预算以为省 18 位就一定降低整包 18 位

为什么 Iris 保存量化状态副本

Iris 在复制系统内部保留完整的量化网络状态。这样，昂贵的 Gameplay 读取与量化可以在 Pre-Send 阶段做一次，结果供多个连接共享；连接级 Writer 在填包时主要访问复制系统内部的紧凑状态，而不反复触碰 UObject。官方 Components 文档也把“维护量化状态副本、共享一次性工作、便于并行”列为核心设计。

代价同样真实：量化状态占内存；字符串、数组、软引用等还可能持有动态分配；基线需要生命周期管理。优化不能只盯网络位数，还要把量化 CPU、缓存局部性和动态状态分配算进去。

🧱 7.2 FNetSerializer 结构详解

它是一张函数表，也是一份数据契约

UE 5.7.4 的 FNetSerializer 不是一个靠虚函数继承的 UObject，而是由 TNetSerializerBuilder 根据实现结构在编译期拼出的函数表。它保存版本、Traits、所有入口、默认 Config，以及量化类型与 Config 的大小和对齐。Descriptor 最终持有的正是这张表和对应 Config。

入口按数据方向理解

入口输入状态输出/作用关键约束
Serialize量化 Source写 BitStream只写约定的位；不应重新读取 Gameplay
DeserializeBitStream量化 Target验证长度、枚举或无效编码；与 Serialize 完全对称
SerializeDelta当前量化 Source + 已确认 Prev写相对基线的数据Prev 语义由系统保证，不能自行猜测客户端值
DeserializeDeltaBitStream + Prev量化 Target即使等于 Prev，也必须形成有效 Target
Quantize非量化 Source量化 TargetTarget 初始内容未知；结果要可位确定比较
Dequantize量化 Source非量化 Target定义误差与非法状态行为

Validate非量化 Source布尔合法性它不替代接收端对不可信比特流的检查

SourceType、QuantizedType、Config、Context

SourceType

Gameplay 看到的类型。可能是 float、FString 或 USTRUCT；Validate 与非量化 IsEqual 面向它。

QuantizedType

Iris 内部与线上编码面向的状态。应为 POD、位确定，零初始化必须代表有效状态；动态数据通过显式生命周期管理。

ConfigType

Serializer 的配置契约，继承 FNetSerializerConfig。Descriptor 将 Serializer 与具体 Config 配对。

Context

携带 Reader/Writer、ChangeMask、连接、Token Store、内部内存与错误上下文。连接相关序列化会降低共享能力。

量化状态最容易被低估的规则

UE 5.7.4 源码明确要求 QuantizedType 是 POD、具有位确定性，零初始化应有效，并避免未初始化 padding。只要状态含指针或动态分配，就必须提供深拷贝与释放；否则基线、接收状态和缓存之间会出现浅拷贝、重复释放或悬空指针。

Traits 是系统承诺，不是装饰标签

ENetSerializerTraits含义工程影响
IsForwardingSerializer转发到其他 SerializerBuilder 会更严格要求可能入口齐全
HasDynamicState量化状态持有动态资源必须实现 Clone/Free
HasConnectionSpecificSerialization输出依赖连接难以跨连接共享序列化结果，增加 CPU/内存
HasCustomNetReference含需收集的网络引用必须实现 CollectNetReferences
UseSerializerIsEqual脏检测使用网络语义相等可让落在同一量化格点的值不产生无效更新
HasApply提供自定义 Apply适合只覆盖结构体部分成员的 Serializer

bUseDefaultDelta 会影响 Builder 选择默认 Delta 路径，但它不是上述公开 Trait 枚举的一项。默认实现通常先写一位“是否与 Prev 相同”；如果关闭默认 same-value 优化，Builder 会让 Delta 入口直接转发普通 Serialize/Deserialize。

三条核心调用路径

比较路径：Source A/B → IsEqual(non-quantized) → 是否标记成员变化
发送路径：Source → Validate → Quantize → Serialize[Delta] → Writer
接收路径：Reader → Deserialize[Delta] → Quantized → Dequantize → Apply

这些入口不保证只调用一次，也不能依赖 UObject 可随时访问。特别是 Serialize 阶段，应把计算尽量前移到 Quantize：后者通常每对象每帧至多执行一次，而 Serialize 可能为多个连接执行。

📐 7.3 量化深入解析

连续值如何变成有限状态

StateCount = floor((Max - Min) / Step) + 1 RequiredBits = ceil(log2(StateCount))

对范围 0–1000、步长 0.1 的生命值，状态索引是 0–10000，共 10001 种，需要 14 位，因为 213=8192 不够，而 214=16384。多出来的 6383 个码位是无效状态，接收端必须拒绝，不能把它们静默夹成 1000。

Quantized = round((clamp(Value, Min, Max) - Min) / Step)
Value'    = Min + Quantized * Step
MaxError  ≈ Step / 2

32 位精度不等于 32 位价值

IEEE 754 float 能表达大量业务永远不会使用的中间值，也包含 NaN、Infinity、正负零等特殊表示。生命值 UI 只显示一位小数时，传完整 32 位尾数没有直接玩家价值；但物理积分、确定性回滚或极大坐标附近的精细差值可能确实需要更多信息。量化标准应来自可感知误差和后续算法，不是“网络字段一律压缩”。

模型每分量状态步长最大舍入误差三分量最坏负载
Rotator Byte2561.40625°约 0.703125°3 标志 + 24 数据 = 27 bits
Rotator Short65536约 0.005493°约 0.002747°3 标志 + 48 数据 = 51 bits
原始 3×float（概念对照）巨大随指数变化远小于多数视觉需求至少 96 bits，未计协议

UE 5.7.4 的 FRotatorNetSerializer、FRotatorAsShortNetSerializer、FRotator3fNetSerializer 和 FRotator3dNetSerializer 都走 Short 量化基类；只有 FRotatorAsByteNetSerializer 走 Byte。3f/3d 描述 Source 精度，不代表线上仍发送 32/64 位每分量。

舍入、夹取与非法值

  • 舍入：发送端和接收端共享离散公式。最大理论误差约为半个步长，但多次业务运算可能累积。
  • 夹取：是防御措施，不应掩盖业务越界。先由 Validate 报错，再让 Quantize 保证即便异常值漏入也不写出非法索引。
  • NaN/Infinity：比较、Clamp 和哈希都可能产生反直觉结果，应在 Validate 明确拒绝，防御性量化映射到安全默认值。
  • 无效码位：14 位能表示到 16383，Health 只允许到 10000；Deserialize 必须设置 Context 错误。
  • 版本变更：一旦修改 Min、Max、Step 或位数，旧客户端会用另一套规则解释同一串比特，这是协议兼容性变化。
什么时候量化可能是负优化

低频字段本就很少发送；为了省 2 位却增加分支、查表、动态分配和跨版本维护，CPU 与复杂度可能超过带宽收益。先测字段分布和发送频率，再决定是否自定义。

🧰 7.4 内置序列化器详解

整数：已知值域优先固定宽度

UE 5.7.4 有 Int/Uint 8、16、32、64，Range 变体和各尺寸 Enum 变体。固定整数 Config 提供 BitCount；序列化只写低 BitCount 位，带符号读取后进行符号扩展。Validate 会检查 BitCount 合法并确认值所需位数不超配置。

整数基类在 BitCount >= 16 时启用零值优化：先写一位“是否为零”。零只花 1 位，非零花 BitCount + 1。所以“16 位整数固定只用 16 位”并不总成立；实际成本取决于值与配置。

场景建议原因
弹匣 0–606-bit Range/定宽模型值域明确，成本稳定
枚举 0–7对应 Enum Serializer保留类型语义并验证值宽
稀疏大 ID先判断是否应使用 Token/引用整数压缩不能解决跨进程标识语义
数值分布偏小、上界大PackedInt用少量前缀换平均小值成本

PackedInt：按字节数自适应，不是 ZigZag

32 位 PackedInt 先写 2 位字节数前缀，再写 1–4 个字节，因此成本是 10、18、26 或 34 位。64 位版本用 3 位前缀，成本是 11、19、27、35、43、51、59 或 67 位。UE 5.7.4 的有符号实现按所需有符号位数选择字节数，读取后符号扩展；它不是 ZigZag。

这意味着 127 和 -1 都可能很便宜，但一个接近 32 位极值的值会比直接固定 32 位多 2 位。是否划算取决于真实分布，不取决于类型最大值。

Float：保留比特表示，只优化零

FFloatNetSerializer 把 SourceType 视为 uint32 位模式，以区分 +0.0 和 -0.0。普通 Serialize 先写 1 位非零标志：正零位模式花 1 位，其他值花 33 位。Double 对应 1 或 65 位。它们有自己的 Delta 位数表，但普通路径并没有替你做“保留两位小数”的业务量化。

UE 5.7.4 源码摘录FloatNetSerializers.cpp
if (Writer->WriteBool(Value != 0))
{
    Writer->WriteBits(Value, 32U);
}

因此“用内置 Float 就是 32 位”不精确，“内置 Float 已经有损压缩”也不正确。正零是 1 位，非零是 33 位，Delta 另算；数值精度仍保持原始位模式。

Rotator：先写存在标志，再写非零分量

Byte 与 Short 基类都先写 3 位 XYZIsNotZero，只为量化后非零的 Pitch/Yaw/Roll 写 8 或 16 位。纯 Yaw 角色朝向因此分别是 11 位或 19 位；三分量都非零才到 27 或 51 位。Delta 路径同样先写三分量变化掩码。

用途推荐起点理由
第三人称角色身体朝向Byte，实测观感约 0.7° 最大误差通常可接受
炮塔精确瞄准Short小角度误差会放大到远距离落点
高速车辆姿态Short + 插值三轴变化频繁，视觉稳定性更重要
不影响玩法的装饰物Byte 或降低更新频率先减少更新，再压位数

    static constexpr uint32 MaxQuantizedValue = 10000U;
    static constexpr uint32 BitCount = 14U;

    static void Serialize(FNetSerializationContext&, const FNetSerializeArgs&);
    static void Deserialize(FNetSerializationContext&, const FNetDeserializeArgs&);
    static void Quantize(FNetSerializationContext&, const FNetQuantizeArgs&);

};

FHealthNetSerializer::QuantizedType
FHealthNetSerializer::QuantizeValue(float Value)
{
    // Validate 会拒绝非法值；这里仍保证发布版本得到安全、有效的量化状态。
    const float FiniteValue = FMath::IsFinite(Value) ? Value : MinValue;
    const float ClampedValue = FMath::Clamp(FiniteValue, MinValue, MaxValue);
    return static_cast<QuantizedType>(
        FMath::RoundToInt((ClampedValue - MinValue) / Step));
}

void FHealthNetSerializer::Serialize(
    FNetSerializationContext& Context,
    const FNetSerializeArgs& Args)
{
    const QuantizedType Value =
        *reinterpret_cast<const QuantizedType*>(Args.Source);
    Context.GetBitStreamWriter()->WriteBits(Value, BitCount);
}

void FHealthNetSerializer::Deserialize(
    FNetSerializationContext& Context,
    const FNetDeserializeArgs& Args)
{
    QuantizedType& Target =
        *reinterpret_cast<QuantizedType*>(Args.Target);
    Target = 0U; // 错误退出时仍保持有效零状态。

    const uint32 Value = Context.GetBitStreamReader()->ReadBits(BitCount);
    if (Value > MaxQuantizedValue)
    {
        Context.SetError(GNetError_InvalidValue);
        return;
    }

    Target = static_cast<QuantizedType>(Value);
}

bool FHealthNetSerializer::IsEqual(
    FNetSerializationContext&,
    const FNetIsEqualArgs& Args)
{
    if (Args.bStateIsQuantized)
    {
        return *reinterpret_cast<const QuantizedType*>(Args.Source0)
            == *reinterpret_cast<const QuantizedType*>(Args.Source1);
    }

    const SourceType& A =
        *reinterpret_cast<const SourceType*>(Args.Source0);
    const SourceType& B =

}

{

✍️ 7.6 BitStream 读写

为什么按位而不是按字节

14 位 Health 如果强制按字节存放，至少需要 16 位；按比特连续写入，可以让后一个字段紧跟在第 15 位开始。Iris 的 FNetBitStreamWriter 写 Value 的最低 BitCount 位，Reader 也把读取结果放在返回值低位。属性边界不要求字节对齐。

Writer/Reader 行为UE 5.7.4 契约工程含义
InitBytesBuffer 至少 4 字节对齐，ByteCount 是 4 的倍数这是底层缓冲约束，不等于字段按 4 字节写
WriteBits写最低 N 位；空间不足时一位不写并置 overflow不能假设发生部分写入

Health 的对称读写

源码核对的最小片段14-bit 对称契约
// Sender: QuantizedValue 必须在 [0, 10000]
Writer->WriteBits(QuantizedValue, 14U);

// Receiver
const uint32 QuantizedValue = Reader->ReadBits(14U);
if (QuantizedValue > 10000U)
{
    Context.SetError(GNetError_InvalidValue);
    return;
}

WriteBits 自己会记录 overflow；业务非法值应调用 Context.SetError，默认还会让当前 BitStream overflow。统一判断可用 Context.HasErrorOrOverflow()。不要捕获错误后继续按正常路径读下一字段，否则一次错位会把余下包解释成随机数据。

读写对称的真正含义

不仅位数相同，条件分支也必须相同。例如发送端只在 bHasValue 为真时写 14 位，接收端必须先读同一标志，再决定是否读 14 位。任何一侧多读或少读一位，错误会沿整个流传播。

位置、容量与溢出

Writer/Reader 都提供 GetPosBits、GetAbsolutePosBits、GetBitsLeft、Seek 和 IsOverflown。调试自定义 Serializer 时，最有效的最小证据是记录入口前后位置：发送写了多少位、接收读了多少位、在哪个字段第一次不相等。Seek 能在回到有效位置后清除 overflow，但生产 Serializer 不应靠反复回退掩盖错误设计。

Δ 7.7 Delta Compression 基础接口

当前值、Prev 与 Delta

Delta Serializer 接收的是当前量化状态与一个已确认的 Prev 量化状态。它可以写“与 Prev 相同”、写差值，或按类型定制压缩。接收端必须用同一个 Prev 重建有效 Target。

Current Quantized State  ─┐
                           ├─ SerializeDelta → Delta Bits
Acked Prev/Baseline State ─┘

Delta Bits + 同一 Prev → DeserializeDelta → Target Quantized State

基线不是“服务器上一帧值”，而是协议路径提供、双方对齐的已确认状态。若基线不可用、未确认或该类型不适合 Delta，系统需要走非 Delta 路径或使用默认状态；自定义 Serializer 不应在函数内部从 Gameplay 对象猜一个 Prev。

Builder 提供的默认行为

若实现没有自定义 Delta 且 bUseDefaultDelta=true，Builder 的默认 SerializeDelta 会先写 1 位网络相等标志：相同则结束，不同则调用普通 Serialize。DeserializeDelta 读到相同标志时会从 Prev 构造 Target；动态状态会先释放 Target，再深拷 Prev。

Health 路径未变已变适用条件
默认 same-value Delta1 bit1 + 14 = 15 bits调用中经常出现与 Prev 相同
本文实现：关闭默认 Delta14 bits14 bits上游 ChangeMask 已保证通常只调用变化成员
自定义小差值编码取决于方案可能低于 14 bits变化幅度分布稳定，值得增加复杂度

本文的 Health 实现设置 bUseDefaultDelta=false，所以 Builder 生成的 Delta 入口直接转发普通 14 位读写。这是一个明确而简单的起点，不声称对所有负载最优。真正的差值位数表、基线维护、确认与系统级启用策略留到本系列第十部分。

什么时候不要急着写 Delta

  • 固定负载本来只有几位，前缀可能抵消收益。
  • 值变化接近随机，差值分布与原值分布没有明显差异。
  • ChangeMask 已把未变成员跳过，而 Serializer 只在变化时调用。
  • 动态状态深拷与基线内存成本高于节省的网络位。
  • 还没有真实分布数据，只凭几个示例决定编码。

🎭 7.8 ChangeMask 系统

它回答“一个 State 里的哪些成员变了”

对象进入脏集合，只说明复制系统需要关注它；一个对象可能包含多个 Replication State，每个 State 又有多个成员。ChangeMask 把变化细化到成员，使 Serializer 只处理需要发送的部分。

层次含义例子
对象脏这个 Net Object 有待处理状态角色任一复制属性变化
State 脏某个 Replication State 有变化位常规属性 State 发生变化
成员脏Descriptor 指定的成员位区间有任意位为 1Health、Ammo 两个成员变化
数组元素变化数组拥有位之外，额外位映射到元素背包第 7 格数量变化

Descriptor 如何把成员映射到位

FReplicationStateMemberChangeMaskDescriptor 为每个成员记录 BitOffset 和 BitCount。普通成员通常占 1 位；数组等特殊类型可以占多位。FReplicationStateOperations::SerializeWithMask 检查成员区间是否有任意位被置位，只有命中时才构造 Args 并调用该成员 Serializer。

Serializer 接收到的 ChangeMaskInfo 是自己在整个 Protocol ChangeMask 中的位区间。普通 Health 不需要读取它；数组 Serializer 会用额外位筛选元素。

内存与线上表示不是同一件事

UE 5.7.4 的 Replication Writer/Reader 记录使用 FChangeMaskStorageOrPointer：不超过 64 位时把掩码直接内联在一个 uint64 中，超过 64 位才通过提供的 Allocator 分配存储。这是连接记录等内部结构的存储优化，不能泛化成“所有 ChangeMask 永远只有 64 位”；State Buffer 也会按 Descriptor 位数保存自己的掩码区域。

线上并不是机械写出全部原始位。Protocol 路径调用 WriteSparseBitArray / ReadSparseBitArray 编码完整 ChangeMask，初始状态还可使用“多数为 1”的 Hint。稀疏掩码的真实成本取决于位数和分布，所以预算表不能简单把 100 个成员写成固定 100 位。

数组不只是“长度 + N 个元素”

动态数组同时包含长度、元素布局、增删改、动态分配、元素引用和元素自身 Serializer。UE 5.7.4 的内部 FArrayPropertyNetSerializer 以 FScriptArray 为 SourceType，量化状态只保存容量、有效元素数和一块元素量化存储：

全量路径与元素复用

普通 Serialize 先写 1 位空数组标志；非空时再写 ElementCountBitCount 位长度，然后根据元素 State Descriptor 逐个调用元素 Serializer。接收端若长度超过 MaxElementCount，会设置 GNetError_ArraySizeTooLarge。

Empty array: 1 bit
Non-empty:   1-bit not-empty

因此数组并没有一套“万能元素格式”；TArray<FReplicatedHealth> 可以复用 Health 的 14 位量化，TArray<UObject*> 则需要对象引用收集和解析。

元素级 ChangeMask 与碰撞

当数组成员分到多于 1 个 ChangeMask 位时，第一个位属于数组属性，剩余位映射元素。UE 5.7.4 当前使用简单的模映射：

    MemberBitOffset + 1 + (ElementIndex % (MemberBitCount - 1))

🔤 7.10 字符串与名称序列化器

FString：ANSI 快路与 UE 自定义宽字符编码

FStringNetSerializer 的量化状态包含 bIsEncoded、容量、元素数和动态字节存储，因此同样具有 Dynamic State。纯 ANSI 字符串按单字节保存；非纯 ANSI 字符串使用 TStringCodec。

UE 5.7.4 这里不是 UTF-8

源码注释明确称该 Codec “比 UTF-8 更紧凑”，每个 Codepoint 最多 24 位；它按 7 位组和高位继续标志编码，并把 UTF-16 surrogate pair 合成一个 Codepoint。旧文章若直接写“FString 转 UTF-8”，在本版本并不准确。

线上格式是 1 位 bIsEncoded、PackedUint32 长度，再加不含终止符的字节流。接收端重新补零终止符，验证编码，并把编码长度限制在 65535 字节范围内。空字符串也有标志和长度成本，并不是零位。

Writer->WriteBits(Value.bIsEncoded, 1U);

FName：可能是硬编码 EName，也可能仍是字符串

FNameNetSerializer 先写 1 位 bIsString。可安全按硬编码名称复制的 EName 直接写一个固定宽整数；其他 FName 要写 Number 的编码方向、Packed Number、字符串编码标志、Packed 长度和字节内容。量化状态也包含动态字符串存储。

所以 FName 不是“天然只传一个很小的整数”。进程内 Name Index 不能直接当跨机器稳定 ID；只有双方共享可解释的硬编码 EName，或通过网络 Token Store 建立映射，才能安全压成稳定标识。

FNameAsNetToken 与使用边界

FNameAsNetTokenNetSerializer 对非硬编码名称从 FNameTokenStore 获取/创建 Token，序列化 Token 并安排 Export；接收端在远端 Token State 中解析。重复稳定名称可避免每次发送完整字符串，但首次导出、Token Store 状态和解析仍有成本。

“Source 是 double”不意味着网络必须保持 double 精度。反过来，Byte 能省 24 位最坏负载，也不意味着炮塔瞄准就应接受它。先把角度误差换算成实际玩法空间误差，再做选择。

Delta 的分量掩码

Byte Delta 先比较当前与 Prev 的三个量化分量，写 3 位变化掩码，只为变化分量写新 8 位值；Short Delta 则写量化差值的 16 位。两者都不会为未变分量写完整值。这一层是 Serializer 内部的分量级 Delta，与外层成员 ChangeMask 是两级筛选。

🔢 7.12 PackedInt 实现细节

32 位：2 位前缀 + 1–4 字节

值范围（uint32）字节数前缀数据位总成本

Unsigned 路径用 GetBitsNeeded(Value | 1U) 保证 0 也至少写一个字节。Signed 路径根据包含符号位所需的位数选择字节数，读取后手动符号扩展，因此 -128–127 是 10 位，128 或 -129 进入 18 位。它没有 ZigZag 把负数映射成偶数/奇数。

64 位：3 位前缀 + 1–8 字节

64 位版本总成本从 11 位开始，每多一个字节增加 8 位，最坏 67 位。对绝大多数值很小但偶尔暴涨的经验值或金币，它可能降低平均成本；对随机哈希、加密 ID 或均匀分布的大整数，前缀只会增加最坏成本。

业务分布候选解释
Ammo：绝大多数 0–255PackedUint32 = 10 bits，或更紧的已知范围定宽若确定 0–60，6 位定宽更好
金币：常见 < 65,536，少量百万PackedUint32常见 18 位，百万级 26 位
经验值：跨阶段逐渐变大先测分位数，再选 Packed/Range游戏后期分布可能让早期收益消失
随机 32-bit 标识固定 32 位或 TokenPacked 最坏 34 位，且通常命中最坏

Delta 接口存在，但不是普通 Packed 格式

PackedInt 四个实现都有专门的 SerializeDelta/DeserializeDelta。32 位使用 {0, 4, 14}，64 位使用 {0, 14, 32} 的差值位数候选表，再由 BitPacking 选择路径。这里先确认接口和设计目的：小变化可用更少位；具体索引、回退与基线生命周期在第十部分展开。

🏭 7.16 自定义序列化器生产化指南

什么时候值得自定义

  • 字段高频、对象数量大，且内置表示与业务可接受误差明显不匹配。
  • 值域、精度和非法值策略稳定，能写成清晰协议契约。
  • 真实分布证明平均位数节省可覆盖实现、测试和升级成本。
  • 内置 Range、Packed、Vector、Rotator 或 Fast Array 不能直接满足。
  • 团队愿意维护跨版本兼容、自动化测试和 Descriptor 注册验证。

协议与版本演进

Version 存在不代表可以随意改变线上格式。UE 5.7.4 的 Args 注释仍提醒 Version 当前没有在所有位置正确传播；项目不能只把 Version 从 0 改成 1 就假定旧客户端会自动分支。修改 BitCount、字段顺序、范围或 Codec 时，要把它当成网络协议变更：协调客户端/服务器版本、回滚策略、灰度环境和录制回放兼容。

Config 默认值也属于协议。若同一 Serializer 允许多套范围，必须确认 Descriptor 两端构造出相同 Config；否则 14 位可能被解释成另一套步长。

动态状态所有权

  • 零初始化的 QuantizedType 必须是有效、可释放的状态。
  • Clone 必须深拷所有分配，使用 Context 提供的内部内存接口。
  • Free 必须可重复调用，并把指针、容量和数量复位。
  • Deserialize 覆盖旧 Target 前，要遵守目标状态已有资源的生命周期。
  • 结构体 padding 必须初始化，否则位比较、哈希和基线可能不确定。

引用、错误与不可信输入

QuantizedType 若包含对象或远端引用，要声明 bHasCustomNetReference 并实现 CollectNetReferences。仅仅把指针或路径写进比特流并不能保证接收端对象已创建。Deserialize 要检查长度、枚举、无效码位和 Token 解析；业务错误用 Context 设置具体 Net Error，BitStream 容量错误由 Reader/Writer overflow 反映。

⚡ 7.18 性能优化建议

三层优化金字塔

基础：少复制

减少不必要对象、状态和变化；正确过滤、休眠、Push Model 与更新频率。一次不发送通常胜过省几个比特。

中级：选对表示

按值域、误差和分布选择 Range、Packed、Vector、Rotator、Fast Array、Token。

高级：定制热字段

只为高频高数量字段实现自定义量化/Delta，并用端到端数据证明收益。

同时衡量六类指标

指标为什么重要建议统计
平均 bits决定常态带宽按 Serializer/字段 P50
最坏 bits影响包预算和尖峰P95/P99/Max，特别是 Packed、String、Array
Quantize CPU对象共享阶段可能处理大量脏状态每对象/每状态耗时
Serialize CPU可能按连接执行每连接、每 Serializer 调用与耗时
动态分配影响抖动、内存和基线 CloneAlloc/Free 次数、峰值字节
缓存局部性热路径常比算术位操作更贵Insights CPU、Cache Miss、状态尺寸

大逃杀预算怎么排优先级

  1. 先确认背包和名称是否在普通战斗帧被跳过；避免大动态状态无效更新。
  2. 再确认位置、旋转的精度和更新频率，它们通常比单个 Health 字段更高频。
  3. 把 Health 32/33 位压到 14 位，适合大量角色且变化频繁的场景。
  4. 检查 Ammo 是 6 位定宽更优，还是因总弹药分布使用 10–18 位 Packed 更合适。
  5. 最后才考虑自定义 Delta 位数表；它会带来基线和兼容复杂度。

FPS 更看重位置/朝向/武器状态的低延迟和误差；MMO 更看重对象数量、低频小整数、重复 Token 和动态状态分配。不存在一张跨类型游戏通用的“最省位 Serializer 表”。

一个典型负优化

为了把 14 位 Health 再省到平均 12 位，引入堆分配的可变 Delta 对象，并让每连接无法共享序列化结果。网络省 2 位，CPU、内存、缓存和维护全部变差。优化必须看系统总成本。

五个带走的结论

  1. 先定义语义，再数位。范围、步长、误差、非法值和版本是一个完整协议。
  2. 量化状态有真实生命周期。数组、字符串和引用的动态内存不能靠 memcpy 糊过去。
  3. ChangeMask 与 Delta 不同。前者选择成员，后者相对 Prev 编码当前值。
  4. 内置 Serializer 优先。Range、Packed、Rotator、Vector、Fast Array 和 Token 已覆盖大量场景。
  5. 字段节省不等于整包节省。对象记录、掩码、引用、创建、协议和调度共同决定真实带宽。
