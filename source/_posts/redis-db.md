## 一、Redis 概览

Redis 是一个广为人知的全内存数据库，作为「存储」（而非仅仅是缓存）来看，它拥有完整的数据库功能。在字节跳动内部也广泛使用，其稳定性、性能、成本控制和故障恢复能力都经过验证。

正如「没有银弹」所言，不同场景需要不同的存储选型，这也是各种存储引擎百花齐放的原因。

### 为什么选择 Redis？

Redis 是全内存数据库，Key 和 Value 都存储在内存中，性能在业界处于领先地位。以至于 2022 年的反思文章《13 Years Later – Does Redis Need a New Architecture?》的结论是：​**NO**​。

本文将结合存储主题的核心模块，深入回顾 Redis 的技术细节。

## 二、IO 流程与线程模型

### 2.1 线程模型

Redis 常被描述为「单线程」，实际上是指在​**默认情况下，Redis 的数据面由主线程单独完成**​。后台仍有其他线程执行文件写入操作（如 AOF 持久化、Lazy Free 大对象），甚至会 fork 出子进程创建快照。

另外，可以通过配置开启**网络 ​IO**​​**​ ​多线程**​，让主线程专注于执行逻辑。但经典且高性能的模式仍是​**单线程 Run-to-Complete**​。

### 2.2 IO 流程

Redis 协议层的核心是 ​**IO​ 多路复用**​，封装了各平台最高效的多路复用工具：

* Linux → epoll
* Solaris → evport
* BSD/macOS → kqueue

**数据流向：**

1. 从网络 IO poll read 读取数据，记录到 client 对象的读 buffer
2. 进入命令执行阶段
3. 执行结果写入 client 对象的 buffer
4. 下次 poll write 时写入 socket

### 2.3 性能考量

Redis 命令都是处理内存数据结构，极为讲究效率，通常为 **O(1)** 时间完成。

但对于 ​**O(N) 操作**​（如删除一个包含 10 万个元素的 zset），需要谨慎处理：

* 直接调用 free 10 万次会阻塞主线程
* 因此引入 **Lazy Free** 特性

同理，当 hash 表保存百万级数据时，遇到负载因子过高需要扩容，采用​**渐进式 Rehash**​，避免大规模数据迁移导致卡顿。

> 💡 ​**作为数据库的设计原则**​：必须考虑大数据量场景，不能因为单个用户的极端操作导致系统不可用。

## 三、存储引擎核心机制

### 3.1 WAL 文件：AOF

Redis 的 WAL 称为 ​**AOF（Append Only File）**​，保存具体执行的命令。

虽然 AOF 语义上类似 WAL，但实际上 Redis 的 AOF ​**保存着全量数据**​，WAL 即数据库本身。

**工作流程：**

1. 执行完命令后，向全局 `aof_buf` 放入执行命令
2. 发起异步任务（bio: Background I/O service）在 IO 线程写入文件

**AOF 保存模式：**

| 模式             | 说明                      | 数据安全性              |
| ------------------ | --------------------------- | ------------------------- |
| always           | 每个命令都 fsync          | 最高，但性能低          |
| everysec（默认） | 每秒 fsync 一次           | 平衡，可能丢失 1 秒数据 |
| no               | 由操作系统决定 fsync 时机 | 最低，性能最高          |

默认配置下（everysec），只要不出现系统崩溃、电源故障、硬件故障，数据是安全的。

### 3.2 Data 文件：RDB

**RDB（Redis**​**​ DataBase File）** 是 Redis 的数据快照文件：

* 数据经过紧密序列化，文件更小
* 加载速度更快
* 生成成本高，一般周期性生成（如 900 秒）

**生成方式：**使用 **fork** 机制产生快照。由于 Redis 的全内存特性，fork 后内存数据立即「静止」，原进程采用 ​**COW（Copy On Write）**​。

⚠️ ​**注意事项**​：Redis 写 QPS 可能很高，造成大量 COW，导致内存暴涨和 CPU 紧张。

```undefined
for (j = 0; j < server.dbnum; j++) {
    redisDb *db = server.db+j;
    for (auto item in db->dict) {
       rdbSaveObject(item)
    }
}
```

### 3.3 垃圾回收——AOF 重写

AOF 重写并非改写每条命令，而是当 AOF 文件过大时，根据当前数据库状态重写一遍，精简重复命令。

​**本质**​：用命令记录内存中的所有数据，相当于整个数据库的逻辑表达。

**执行流程：**

1. 基于 fork 创建子进程（需要快照）
2. 父进程将快照后的增量 AOF 写入临时缓冲区
3. 子进程结束后，父进程将增量 AOF 写入重写文件
4. 删除旧 AOF 文件，完成垃圾回收

​**Redis​ 7.0+ 优化**​：引入 **AOF Manifest** 机制，重写后的 AOF 文件 + 增量 AOF 文件组成全量数据，效率更高。

### 3.4 垃圾回收——内存碎片整理

Redis 全内存保存 KV，重的内存操作容易产生碎片，导致实际物理内存开销大于统计值。

| 版本         | 处理方式         |
| -------------- | ------------------ |
| Redis 4 以下 | 只能重启实例     |
| Redis 4+     | 引入 defrag 模块 |

​**实现依赖**​：必须使用 **Jemalloc** 才能启用内存碎片整理。

```C
#if defined(USE_JEMALLOC) && defined(JEMALLOC_FRAG_HINT)
#define HAVE_DEFRAG
#endif
```

​**工作原理**​：

1. 遍历 db 中的 key（每次只遍历 N 个，避免阻塞）
2. 询问 Jemalloc 目标指针移动能否减少碎片
3. 如果可以，在新地址空间申请内存并完成替换

## 四、索引结构

由于 Redis 是全内存数据库，对索引的依赖不如磁盘数据库强。

**主索引**​​**结构**​：哈希表（dict）

```C
typedef struct dictEntry {
    void *key;                // 键
    union {
        void *val;            // 值
        uint64_t u64;
        int64_t s64;
        double d;
    } v;
    struct dictEntry *next;   // 冲突链表
} dictEntry;

typedef struct dictht {
    dictEntry **table;        // 两个数组，实现渐进式 rehash
    unsigned long size;
    unsigned long sizemask;
    unsigned long used;
} dictht;
```

### 渐进式 Rehash

解决扩容 **O(N)** 操作阻塞服务器的问题：

* 一次性申请扩容内存
* 逐步将原 dict 成员迁移到新 dict
* 迁移时机：每次对该 dict 的增删改查操作中

**操作规则：**

* 查询：依次查 table[0]，再查 table[1]
* 添加：只在 table[1]
* 删除：依次 table[0]，再 table[1]

## 五、主从复制

### 5.1 快照

通过 **fork** 即可完成，是所有存储中实现快照最方便的方式，直接使用操作系统命令，无需设计复杂存储结构。

### 5.2 全量复制

新从节点需要同步全量数据：

1. 从节点发送 `FULLRESYNC` 命令
2. 主节点 fork 产生快照
3. 子进程中生成 RDB
4. 直接写入目标 socket（而非文件）

### 5.3 增量复制

已有数据的从节点只需同步增量：

1. 发送 `PSYNC` 命令
2. 同步 `aof_buf` 中的 WAL

### 5.4 主从级联

为减轻主节点生成 RDB 的压力，Redis 支持从节点挂载从节点。但代价是数据一致性延迟更高。

## 六、事务与 Pipeline

### 6.1 Pipeline

保证若干命令通过**一次 TCP 连接**传递到服务端，并顺序执行。但中间可能插入其他客户端的命令。

### 6.2 事务

保障「若干命令顺序执行，中间不会执行其他客户端命令」。

**ACID 特性分析：**

| 特性             | 说明                                                                                     |
| ------------------ | ------------------------------------------------------------------------------------------ |
| **原子性** | ⚠️ 不支持回滚。事务中某命令出错不影响其他命令执行。编写无错误的事务可拥有原子性        |
| **一致性** | ✅ Redis 在各种出错情况下都能满足一致性要求（约束较少）                                  |
| **隔离性** | ✅ 事务总是串行执行，满足隔离性要求（串行化）                                            |
| **持久性** | ⚠️ 仅在开启 AOF/RDB 且`appendfsync=always`时才能保证。主从异步复制切主时可能丢数据 |

## 七、小结

Redis 的「存储引擎」并非传统意义上的标准实现，它更适合：

* 数据量较小的场景
* 可以经常进行全库 Dump（AOF 或 RDB）
* 快照通过 fork 方便完成

其设计哲学体现了「简单即高效」的理念，通过操作系统原语（fork、epoll）和精巧的内存管理，在单线程模型下实现了极高的性能。
