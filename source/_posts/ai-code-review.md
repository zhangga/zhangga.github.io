# 一、背景与动机

> *本节回答：为什么我们需要自行建设 AI 代码审查能力？*

## 现状与痛点

许多项目采用快速迭代的发布节奏，代码集成模式多样（如直接 commit 到主干、cherry-pick 到发布分支等）。当项目**缺少强制的 Merge Request 流程**时，往往**缺少审查拦截点**，代码质量难以保障。

代码审查中常见的主要痛点如下：

| 痛点 | 表现 |
|------|------|
| **缺少前置审查** | 代码直接入主干，缺少强制的审查拦截点 |
| **人工审查成本高** | 开发节奏快，逐个 commit 仔细 review 不现实 |
| **问题发现滞后** | 性能隐患、数据一致性问题往往在线上才暴露 |
| **通用工具不适配** | 部分业务技术架构独立，难以直接接入标准化的集团级审查方案 |

因此，需要结合可用资源，使用合规工具自行建设 AI 代码审查能力，提高代码质量。

## 定位

> 🌟
> AI 代码审查**不是替代人工审查**，而是作为人工审查的**前置环节和辅助工具**。
> 通过 AI 审查拦截大部分基础错误，让人工聚焦架构设计与业务逻辑。

相比直接使用通用 AI 平台进行代码审查，本 SOP 的主要优势如下：

| 特点 | 实现 | 效果 |
|------|------|------|
| **推送及时** | 1. webhook 触发及时；2. Agent 审查速度更快；3. 发现问题时，机器人主动私聊推送卡片 | 问题推送延迟约 **3-5 分钟**（基于离线计算链路的方案延迟通常 > 1h）；实时性更高，方便代码提交后及时发现问题 |
| **有效率高** | 1. 支持按问题类型、严重程度进行过滤，只推送过滤后的问题；2. 支持指定 SOTA 模型 | 垃圾信息少，问题拦截率高；接入负担小，更容易在项目内落地 |

# 二、探索与选型

> *本节回答：我们试过哪些方案，最终为什么选择 CLI 型 Agent？*

## 失败尝试

| 方案 | 做法 | 失败原因 |
|------|------|----------|
| **Webhook 创建 MR** | 每次提交自动创建 MR，邀请 AI 评审 | 1. AI 无法理解高度耦合的复杂业务逻辑，只能检出规范问题；2. 审查速度太慢 |
| **平台定时任务** | 定时拉取更新，平台自动 review | 1. 审查效果有限；2. 授权单一，只能通知任务创建者 |
| **平台 OpenAPI** | 调用审查平台 OpenAPI 创建审查任务 | 1. sandbox 模式每次需要重新生成中间代码；2. copilot 模式无法执行中间代码生成；3. 无状态看板 |

共同教训：复杂项目代码高度耦合，简单的规范检查或 API 调用无法满足需求，必须使用能深入理解大型代码库、支持本地部署和自定义流程的 Agent 工具。

## 为什么选择 CLI 型 Agent

| 特点 | 描述 |
|------|------|
| **代码能力最强** | 同模型下，性能接近主流顶尖编码 Agent；TUI（CLI）形式的 Agent 相比 IDE / Web 形式，在代码任务上性能更优 |
| **二次开发友好** | 可本地部署（命令行直接调用）；支持 OpenAPI、ACP，可无缝集成到各种流水线 |
| **安全合规** | Agent -> 模型网关 -> 模型供应商，应确保具备数据零留存协议 |
| **生态完善** | 支持 MCP、Skills、Subagent、AGENTS.md 等特性；可内置 IM 工具和 Codebase 检索等能力；支持客户端部署、IDE 插件、Web 端直接使用 |

# 三、系统架构与工作流

> *本节回答：整套系统是怎么跑起来的？*

## 系统概览

### 在线审查工作流

核心链路：Git Push → Webhook 触发审查引擎 → 提取提交上下文 → Agent 分析 → 结果落数据表 → IM 机器人推送问题 → 用户反馈回写。

### 离线分析工作流

两条定时任务：每日统计审查指标并推送仪表盘；每月基于反馈数据和代码证据，AI 生成改进建议。

## 任务触发

| 要求 | 实现 |
|------|------|
| 自动触发 | codebase webhook 监听 git push -> 独立 CI/CD job（如 Jenkins） |
| 服务状态可视化 | CI/CD 服务器自带 dashboard 和任务状态留存 |

## 执行审查

| 背景 | 要求 | 实现 |
|------|------|------|
| 公司代码不允许泄漏 | 安全合规，没有泄密风险 | 选用与模型供应商签有数据零留存协议的统一模型网关 |
| webhook 偶发丢包不重试；模型偶发抽风；模型服务不保证 SLA；单点故障（如打版机掉线、Agent crash） | 具备基础的容灾能力 | **滑动窗口**：每次触发时回溯最近的 n 次提交，找出未完成审查的 commit 发起任务；**原子任务**：只有完成了代码审查、表格记录、IM 通知等所有步骤的任务才会落盘；**有限重试**：在回溯窗口内，未完成的任务会不断重试，且不会无限重试 |
| - | 审查效果好（上下文提取） | 查询同一工单的最近提交，一并交给 AI；如有需要，可以按文件类型、文件名过滤 |
| - | 审查效果好（模型更新） | 持续跟进所支持的 SOTA 模型，及时更新模型 |
| - | 审查效果好（提示词优化） | 项目级规范/定义 -> AGENTS.md；代码审查任务要求 -> 代码审查任务提示词 |
| - | 审查效果好（新特性的使用） | Skills -> 固化获取上下文/调用工具的 **SOP**，或开发渐进式披露的知识库/项目规范；Subagent/多轮调用 -> 解耦代码审查和报告验收两部分，过滤模型的幻觉和误报 |

单次审查流程：扫描未审查提交 → 按 Commit 逐个提取工单关联的历史变更 → 连同增量代码一并提交 Agent → 返回审查建议。

## 记录与反馈

> *本节回答：审查结果怎么追溯？怎么形成闭环？*

| 目的 | 实现 |
|------|------|
| **审查记录可追溯** | 所有结果（是否有审查出问题）和反馈（审查问题是否正确）均使用数据表记录，系统的运行细节不是黑盒 |
| **推送问题可反馈** | 人工核查 AI 审查出的问题是否有效后，点击卡片一键反馈 |
| **不过度打扰用户** | **推送分级**：按问题严重程度分级，前期只推送高级别问题，有效率达标后逐步放宽；使用 IM 机器人**私聊**通知用户 |

反馈闭环流程：审查结果存数据表 → 高风险问题通过 IM 卡片私聊通知作者 → 作者点击"有用/无用"一键反馈 → 服务回调更新表格记录。

## 自动总结与定期同步问题

| 目的 | 实现 |
|------|------|
| 持续优化代码审查的效果 | 自动总结：统计高频问题（脚本统计字典词频 + 排序 + 过滤）；召回相关代码，AI 根据问题类型及相关代码给出改进建议；反馈有用/无用的记录分开统计，分别得到对提高代码质量的建议，以及 AI 代码审查的优化建议 |
| 提高组内同学的代码质量 | 定期同步近期 AI 代码审查中出现的高频问题、典型问题 |

月度总结流程：拉取上月反馈记录 → 分词 + 字典匹配做初步分类 → 提取典型案例源码 → AI 归纳生成改进建议。

# 四、接入 SOP

> *本节回答：怎么把 AI 代码审查接入你的项目？*

> 👍 详细 SOP 文档和脚本模板已准备就绪。如果你的项目也有代码审查需求，欢迎按照操作指南接入。

接入要点：

- 本方案**不限语言、不限代码管理工具**，已在 Java、C++、Lua、TypeScript 等语言以及 Git、Svn 等工具上验证，接入成本通常 1-2 天。
- 对于项目定制化需求，可以让 AI 直接 vibe coding 改，AI 写 Python 脚本的效果非常好。

通用接入步骤：

1. **配置 webhook**：在代码仓库设置 git push 触发，对接 CI/CD job。
2. **部署审查脚本**：拉取增量提交、提取工单关联上下文、调用 Agent。
3. **配置数据表**：用于记录审查结果与反馈。
4. **配置 IM 机器人**：推送问题卡片并接收反馈。
5. **编写 AGENTS.md 与审查提示词**：沉淀项目规范、架构特点与审查关注点。
6. **设置定时任务**：每日指标统计、每月改进建议生成。

# 五、落地效果

> *本节回答：实际跑起来效果怎么样？*

从接入到产出有效结果，一个项目通常只需 **1-2 天**部署时间。

本方案已在多种类型的项目中验证，覆盖不同的代码管理工具与编程语言组合：

| 代码管理 | 语言示例 |
|----------|----------|
| Git + Svn | Lua、C++ |
| Git | Java |
| Svn | TypeScript、C++ |

落地参考表现：

- 成熟接入的服务端项目，问题有效率可超过 **70%**，可累计拦截大量数据一致性、性能、空值安全、逻辑缺陷类问题，达到完全可用状态。
- 新接入项目（如前端 TypeScript、客户端 C++）在早期有效率约 50%-60%，随着 AGENTS.md 与提示词的持续打磨稳步提升。

典型发现的问题类型：数据一致性隐患、性能问题、空值安全、逻辑缺陷。

# 六、经验总结与展望

> *本节回答：踩过哪些坑？接下来往哪走？*

## 核心经验

> 适用于项目内所有 AI 工具的落地

- **上下文工程很重要** — 提供正确的上下文（工单、关联变更、框架知识），效果远胜于单纯换更强的模型。
- **反馈闭环是持续优化的基石** — 没有有用/无用的反馈机制，就无法量化改进。
- **先收紧再放开** — 前期只推高风险问题，有效率达标后再逐步放宽，决定了团队对 AI 审查的第一印象。
- **AGENTS.md/Skills 是项目智慧的沉淀** — 项目规范、架构特点、踩过的坑持续写入，AI 越来越懂业务。
- **在系统层面容错** — 滑动窗口、任务原子性、二次校验，接受 AI 会出错，用机制兜底。

## 后续规划

- 持续跟进最新模型，保持审查效果领先。
- 深化 AGENTS.md，提升 AI 对业务的理解深度。
- 降低新人上手门槛，通过 AI 反馈快速熟悉项目。
- 定期同步高频问题，推动编码规范改进。

## AI 基建即新人文档

> 👉 **AI 是超高智商的实习生，为 AI 准备的一切也是为新人准备的。一套投入，两份回报。**

从本质上看，AI 的行为模式和一个高智商但缺乏项目经验的实习生非常相似 —— 它有强大的通用能力，但不**了解你的项目为什么这样设计、哪些写法踩过坑、哪些约定是血泪教训**。如果什么上下文都不给，它只能做出泛泛的、不痛不痒的审查；而一旦你把项目的规范、架构特点、历史踩坑写清楚，它的表现会质变。

这恰恰意味着：**为 AI 建设的知识体系，天然就是最好的新人文档。**

- **AGENTS.md** 沉淀了项目的架构设计理念、编码规范、常见陷阱和历史决策 —— 新人读完就能快速理解"项目为什么是这样的"，而不是靠口口相传。
- **Skills（SOP）** 将获取上下文、调用工具、执行审查等流程固化为可复用的标准操作 —— 新人照着就能上手，不再依赖老人手把手带。
- **审查提示词（prompt.md）** 明确了代码审查的关注点和评判标准 —— 新人可以从中直接学到"这个项目最在意什么"。
- **反馈闭环持续沉淀的高频问题** 本身就是一份活的 FAQ —— 新人可以提前了解团队最常犯的错误，避免重蹈覆辙。

换句话说，打磨 AI 代码审查的过程，其实是在倒逼团队**把隐性知识显性化**。过去散落在老人脑子里的经验，现在变成了结构化的、可机读也可人读的文档。AI 用这些文档做审查，新人用这些文档学上手 —— 一套投入，两份回报。

与其把它看作"给 AI 写文档"，不如说是在为整个团队建设一个持续进化的知识库。**AI 越强，文档越完善；文档越完善，新人上手越快，AI 审查也越准** —— 这是一个正向飞轮。

# 附：Q&A

**Q1：接入 AI 代码审查需要多少人力？技术门槛高吗？**
参照 SOP，1 人 1 天即可完成全流程部署。主要工作是配置 CI/CD、IM 机器人和部署脚本，无需额外开发。Git 和 Svn 项目均有现成模板。

**Q2：AI 审查一次提交需要多长时间？会阻塞开发吗？**
和选用模型强相关，单次审查约 3-5 分钟，**异步执行**，完全不阻塞开发流程。开发者正常提交代码，有问题时通过 IM 私聊通知。

**Q3：70% 有效率意味着还有 30% 误报，如何看待？**
AI 审查定位为辅助工具而非最终裁判。30% 的无效推送成本远低于漏掉一个线上问题的代价。同时通过 Subagent 过滤、提示词优化、推送分级等手段，有效率仍在持续提升。

**Q4：我的项目用的语言/框架不在已验证列表中，能接入吗？**
核心架构不限语言。只需调整 AGENTS.md 中的项目描述和审查提示词，适配新语言/框架。CLI 型 Agent 本身支持主流编程语言。

**Q5：AGENTS.md 怎么写？需要投入多少精力维护？**
初始版本可由 AI 根据项目代码自动生成，后续根据审查反馈中的"无效"案例逐步沉淀即可。维护成本低，但对代码审查等项目下所有 AI 任务的效果提升显著。
